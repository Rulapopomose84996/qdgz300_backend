基于 2026-03-16 对 `ssh kds` 服务器的只读审计，结论是：**当前服务器硬件与系统基线可以支撑本文提出的接收端优化方案，但当前运行配置和代码实现尚未完全落地这些优化点。**

* 线上 `receiver_app` 的 3 个接收线程已按配置绑到 `CPU 16/17/18`，与三阵面接收线程规划一致。

* 后续处理线程尚未固定到数据面核心，实际仍在 `0-15` 范围漂移，不满足“每阵面后处理线程固定核心”的目标。

* 线上安装配置中 `capture.enabled=true`，与本文“高负载默认旁路 capture hook”的建议不一致。
* 当前代码仍存在以下待改项：
  * `RxStage::QUEUE_CAPACITY = 8192`
  * `PacketPool` 仍按 `batch_size * 4` 分配
  * `recvmmsg` 仍为单轮 drain，尚未实现“一次 wakeup 多轮捞取 socket queue”

### 三、审计结论

因此，本服务器的结论应表述为：

* **可以实施本文优化方案，服务器基础条件没有明显硬件或系统级阻碍。**
* **但若要达到本文预期的单阵面 4.8 Gbps 稳定余量，仍需补齐后处理绑核、NUMA 局部性、capture 默认旁路以及接收链路参数扩容等改动。**

## 一、单阵面至少需要多少资源

这里按“**不含重组逻辑，只保证接收模块不阻塞**”来算。

### 1) 网口资源

单阵面目标 4.8 Gbps，对应单个 10GbE 口，链路带宽是够的。
你的四口万兆网卡里，每个光口接一个阵面，这是合理拓扑。

单阵面链路占用率约：

* 4.8 / 10 = **48% 线速**

这意味着链路本身不是瓶颈，瓶颈会在：

* NIC RX queue 配置
* 内核 UDP 栈
* CPU 亲和性和 NUMA
* 用户态 ring 深度
* 下游消费抖动

### 2) CPU 资源

对单阵面，建议这样配：

**最低配置**

* 1 个独占物理核：收包线程
* 1 个共享/非独占核：后续处理线程

**推荐配置**

* 1 个独占物理核：`ArrayFaceReceiver` + `RxStage`
* 1 个独占或半独占物理核：该阵面的后续消费线程
* sibling HT 线程不要跑重负载

原因很直接：你的接收线程里实际做了绑核、NUMA、`recvmmsg`、DSCP 分类、source filter、capture hook、批量投递等动作 ；而 `RxStage` 还会 parse、validate、填 `RxEnvelope` 并推 SPSC 队列 。
这条热路径虽然轻，但已经不是“纯拷包”线程了，所以我不建议把单阵面的接收和后续处理长期压在一个核上。

### 3) 内存 / 缓冲资源

你现在的 `PacketPool` 容量是：

* `pool_size = effective_batch_size * 4`
* 每块固定 65535 字节

而 `UdpReceiverConfig` 默认 `recv_batch_size = 64`

所以当前默认单阵面 `PacketPool` 大致是：

* 64 × 4 = 256 个 buffer
* 256 × 65535 ≈ **16 MB**

这个量级对“4.8 Gbps 持续流 + 短时抖动”来说，**偏小**。它更像“够跑”，不是“够稳”。

单阵面 4.8 Gbps = **600 MB/s**。
这意味着：

* 1 ms 突发 ≈ 0.6 MB
* 10 ms 突发 ≈ 6 MB
* 50 ms 突发 ≈ 30 MB
* 100 ms 突发 ≈ 60 MB

因此单阵面建议：

**最低可行**

* socket rcvbuf: 256 MB（你当前默认值）
* PacketPool: 32~64 MB
* SPSC 队列承载数据的控制面对象：至少覆盖 10~20 ms 抖动

**推荐**

* socket rcvbuf: 256~512 MB
* PacketPool: **64~128 MB / 阵面**
* 若后续线程可能有调度抖动，SPSC 侧要按 20~50 ms 窗口设计

### 4) NUMA 资源

这项在你场景里不是优化项，而是硬要求。

你现在代码已经支持：

* `numa_run_on_node`
* `numa_set_preferred`

但真正需要做到的是：

**单阵面的网口、收包核、PacketPool 内存池必须在同一 NUMA 节点。**

否则 4.8 Gbps 不一定立刻炸，但会表现成：

* CPU 利用率异常偏高
* 时延抖动
* 突发期更容易掉包
* 三阵面一起跑时互相拖累

## 二、单阵面的推荐数据流模型

你现在的实际模型是：

`NIC -> UDP socket -> recvmmsg -> PacketPool buffer -> RxStage(parse/validate) -> SPSC -> Dispatcher/Reassembler/Reorderer`

这个大方向是对的。
我建议把“单阵面接收模型”明确成下面这个形式：

### 推荐模型

`Port_i / RXQ_i -> Core_i -> ArrayFaceReceiver_i -> RxStage_i -> SPSC_i -> ProcessingCore_i`

更具体地说：

#### 第 1 层：硬绑定层

* 一个物理光口对应一个阵面
* 一个阵面对应一个 IP
* 一个阵面对应一个收包线程
* 一个收包线程对应一个独占物理核
* 一个阵面对应一个独立 PacketPool
* 一个阵面对应一个独立 SPSC ring

这点你已经基本做到了。三阵面拓扑在 `app_init.cpp` 中是按 3 个 IP、3 个 source_id、3 个 cpu_affinity 显式组装的 。

#### 第 2 层：热路径最小化

收包线程只允许做：

* `recvmmsg`
* source 过滤
* 批次级打时间戳
* 轻量 classify
* parse header
* validate
* enqueue

你现在 `RxStage` 的职责定义与这个模型一致，而且明确禁止大对象构造、复杂查找、动态分配和重日志 。这是正确的。

#### 第 3 层：后处理隔离

* 重组、重排、投递全部在另一个消费线程做
* 与接收线程仅通过 SPSC 交接
* 不允许后处理反向阻塞 RX

你当前在 `app_init.cpp` 中已经把这层拆开了：实时接收层和后续处理层通过 SPSC 队列完全解耦 。这也是正确方向。

## 三、每个阵面至少多少资源：给你一个可执行口径

按你现在的架构，不上 AF_XDP/DPDK，只沿用 Linux UDP + `recvmmsg`，我建议单阵面按下面规划。

### 最低资源口径（能达到 4.8 Gbps，但余量不大）

* 1 个 10GbE 光口
* 1 个独占物理核给收包线程
* 1 个后处理线程，可先不独占
* socket rcvbuf 256 MB

### 推荐资源口径（工程上更稳）

* 1 个 10GbE 光口

* SPSC ring 覆盖 20~50 ms 突发
* IRQ affinity / RPS/RFS 关闭或收敛，避免包乱窜
* 该端口对应的 IRQ 和线程绑在同一个 NUMA 节点

## 四、你当前模型的主要改进点

这里直接说代码层面的差距。

### 1) `recvmmsg` batch 上限太紧

你当前 `effective_batch_size()` 在 Linux 下硬限制到 64 。
这对一般流量没问题，但对 4.8 Gbps 的突发吸收能力一般。

问题不是“吞吐平均值”，而是“每次从内核 socket queue 捞数据的效率”。

改进建议：

* 先保留 `recvmmsg`
* batch size 继续从配置驱动
* 把接收循环改成“单次 wakeup 连续 drain 多轮”，不要只调用一轮 `recvmmsg` 就回到大循环

也就是说，不一定要突破 64 这个单次上限，但要做：

* 一次线程被唤醒后，尽量把 socket queue 多捞几轮再退出用户态处理窗口

### 2) `PacketPool = batch * 4` 太保守

当前 pool 大小是批量大小的 4 倍 。
这更像实验参数，不像高负载稳定参数。

单阵面 4.8 Gbps 下，建议不要按 batch 推 pool，而要按“突发窗口”推：

* 目标承受 20 ms：准备约 12 MB 净载荷缓冲
* 算上报文头、控制面对象、碎片、回收滞后，建议乘 3~5 倍

所以更实际的口径是：

* **每阵面 PacketPool 不小于 64 MB**
* 如果后处理线程不绑核，直接拉到 **128 MB**

### 3) `RxStage::QUEUE_CAPACITY = 8192` 偏小

你当前每阵面 SPSC 容量是 8192 。

这个数字是否够，要看包率。
在 4.8 Gbps 下：

* 若平均包长 1500B，包率约 0.4 Mpps，8192 约能扛 20 ms 左右
* 若平均包长 512B，包率约 1.17 Mpps，8192 只能扛约 7 ms
* 若平均包长 256B，包率约 2.34 Mpps，8192 只能扛约 3.5 ms

所以：

* 如果你的 FPGA 出来的包不大，8192 明显偏紧
* 即使是中包，8192 也只是“够跑”，不是“够稳”

建议直接把每阵面接收 SPSC 提到：

* **32768 起步**
* 更稳妥是 **65536**

### 4) 批次级共享时间戳是对的，但批次内优先级分桶可能增加尾部停留

你当前收包后是：

* 先 heartbeat_batch / data_batch 分桶
* 再按优先级批量投递

这个策略的优点是心跳优先。
缺点是：

* 数据包会在用户态多停留一小段时间
* 每轮循环会多一次分桶和二次遍历

如果你的目标是纯吞吐优先、心跳流量很低，建议评估两种模式：

* 心跳优先模式：当前方案
* 直通模式：直接进入同一个 ring，在消费侧做轻量优先处理

不一定要改，但这是你热路径里一个明确可评估的开销点。

### 5) capture hook 在高负载下要默认彻底旁路

你当前每批会加载一次 `capture_hook`，然后每包可能调用它 。
这个设计已经尽量轻了，但只要 hook 开启，本质上就把热路径和外部 IO 风险重新耦合了。

结论很简单：

* 单阵面 4.8 Gbps 稳定接收时，**默认禁止抓包**
* 抓包只能做限流、按类型/按 source 采样，不能全量常开

### 6) 处理线程目前“无指定核心”，这是个风险点

`app_init.cpp` 明确写了：

* 实时接收层跑在 CPU 16/17/18
* 后续处理层无指定核心

这对功能正确没问题，但对吞吐稳定性是风险。

因为这意味着：

* 消费线程可能漂移
* 会和别的后台线程抢核
* 会导致 SPSC 水位抬高
* 最终把压力回传成 RX drop

建议：

* 每阵面至少给一个固定 processing core
* 或者给一个 face-local core set，不要让 OS 随机漂

### 7) 统计是全局共享 atomic，缺少“每阵面局部热点指标”

`UdpReceiverStatistics` 是所有阵面共享 atomic 计数器 。
这对总量监控够了，但做 4.8 Gbps/face 调优还不够。

你真正需要的是每阵面、最好每队列级别的：

* socket drops
* SPSC occupancy high-watermark
* PacketPool alloc miss
* `recvmmsg` 每轮实际收到多少包
* 单阵面 pps / bps
* 每阵面 parse/validate/enqueue drop

否则你看到的只是“系统总吞吐”，很难定位是哪一个阵面先掉。

### 你已经做对的部分

### 单阵面至少需要多少资源

按你当前架构，单阵面独立做到 4.8 Gbps，最低建议：

* 1 个 10GbE 光口

### 你当前模型的改进点在哪

关键不是重写，而是把这几项做实：

* 增大 SPSC 深度
* 扩大 PacketPool
* 给后处理线程绑核
* 优化 socket queue 的 drain 模式
* 关闭或限流 capture hook
* 增加每阵面可观测性
