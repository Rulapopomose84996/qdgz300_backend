## 2. 已确认的重构硬约束

- 顶层 `CMakeLists.txt` 只保留编排职责，不继续堆业务源码清单
- 依赖优先走项目内离线化，避免依赖服务器在线安装开发包
- 构建链分三层：
  - 本地开发构建
  - WSL/Linux 交叉构建
  - 麒麟 V10 ARM64 服务器原生构建
- 正式测试入口固定到测试子目录，不依赖顶层 `ctest` 递归发现
- 运行时数据必须与源码目录隔离，统一收敛到 `data/`
- GPU/CUDA 相关模块必须显式兼容 CoreX 编译链，不默认沿用 GCC 路线

---

## 3. 建议的重构顺序

1. 先固化工作入口、部署基线、项目规划文档
2. 再拆顶层 CMake，为模块化装配做准备
3. 然后重组 `src/` 与 `tests/` 目录
4. 补齐服务器原生构建和交叉构建脚本
5. 最后整理 `deploy/`、配置、systemd、环境变量和运行时目录

---

## 4. 当前建议的下一步

- 第三优先级：评估是否需要把当前 `receiver_app` 安装布局扩展到更多可执行组件

## 6. 关键命令

### 6.1 查看当前分支与工作区

```bash
git status --short --branch
```

### 6.2 正式服务器构建入口

```bash
bash scripts/build_production.sh
```

### 6.3 正式测试入口

```bash
ctest --test-dir build_production/tests/unit --output-on-failure
```

### 6.4 正式安装入口

```bash
sudo bash deploy/install.sh build_production
```

### 6.5 已验证的服务验收命令

```bash
sudo systemctl status qdgz300-sysctl.service --no-pager
sudo systemctl status nic-optimization.service --no-pager
sudo systemctl status cpu-performance.service --no-pager
sudo systemctl status qdgz300-receiver.service --no-pager
sudo journalctl -u qdgz300-receiver.service -n 50 --no-pager
```

## 8. 风险与备注

- M02 信号处理模块涉及 `.cu` 与 CoreX 兼容链，后续恢复 GPU 真正算子时仍需单独验证 `clang++` 编译参数与链接路径
- `M03 / M04 / control` 当前尚未恢复真实实现，只是从构建装配中清掉了空壳
- 参考指南默认使用 GCC 7.3.0，但本项目在 GPU 链路上不能照搬该约束
