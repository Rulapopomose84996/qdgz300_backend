# 系统可观测性与健康检查说明

## 1. 观测目标

可观测性体系用于回答三类问题：

- 当前系统是否还能持续输出可信 `TrackData`
- 哪一层发生了退化或故障
- 是否能把异常追溯到明确的事件、资源或时间质量问题

## 2. 分层观测模型

### 2.1 L1 业务层

- `TrackData` 发布间隔
- 输出帧率
- 输出质量标志
- 关键状态机状态

### 2.2 L2 流水线层

- 队列深度
- 重组超时
- 不完整帧比例
- 模块处理延迟
- GPU 超时或回退计数

### 2.3 L3 系统层

- CPU / NUMA / 内存 / 磁盘 / 网络资源占用
- 中断分布与跨 NUMA 访问
- 观测系统自身丢样或降级情况

## 3. 健康度与状态映射

- `Running`：业务输出连续、关键质量指标稳定
- `Degraded`：仍能输出，但已有持续性退化或风险
- `Fault`：可信性红线被击穿，必须停止对外输出或进入故障态

关键裁决输入：

- `trackdata_gap_ms`
- 时间质量是否降为 `FREERUN`
- 数据完整性是否失去可信性
- 关键模块是否失去心跳或崩溃

## 4. 刚性约束

- 观测系统不得对数据面施加反压。
- 丢样、降采样和旁路导出只能影响观测完整度，不能阻塞主链路。
- 观测指标名称、阈值和状态映射必须与故障裁决一致。

## 5. 诊断与审计

- 关键故障应能生成诊断包。
- 诊断包至少包含运行身份、时间状态、关键计数器、最近事件和日志索引。
- 控制命令、状态转移和危险事件需要具备可追溯的审计记录。
