# QDGZ300 重构分阶段计划

**起始日期**：2026-03-11  
**工作分支**：`refactor/kylin-v10-arm64-cuda-clang18`

---

## 1. 重构目标

本次重构不是单点修补，而是把仓库改造成适合麒麟 V10 ARM64 长期维护的工程化形态，重点覆盖：

- 项目组织方式
- 顶层构建与模块装配
- 测试目录与正式测试入口
- 离线依赖和脚本体系
- 部署资产、运行时目录和环境约束文档

---

## 2. 与参考模板的继承关系

直接继承的部分：

- 顶层 CMake 只做编排
- 离线依赖优先
- 三层构建路径
- `tests/unit` 作为正式测试入口
- `deploy/` 和 `data/` 独立

本项目的特化部分：

- M02 信号处理模块会使用 CUDA 兼容链路
- 服务器环境要求 `clang18 + CoreX 4.3.8 + Iluvatar MR100/MR-V100`
- GPU 编译链必须单独建模，不能机械照搬 GCC 7 模板

---

## 3. 分阶段任务

### Phase 1：文档与骨架固化

- 建立工作入口
- 建立部署基线
- 建立分阶段计划
- 建立 `data/` 运行时目录约束

### Phase 2：顶层构建瘦身

- 拆分顶层 `CMakeLists.txt`
- 新增模块化 `cmake` 组织文件
- 新增 `third_party/CMakeLists.txt`
- 收敛构建选项命名

### Phase 3：测试体系重构

- 将 `tests/` 拆成 `unit` 和 `integration`
- 在 `tests/CMakeLists.txt` 固化正式测试入口
- 调整 README、脚本和 CI 入口

### Phase 4：源码组织重构

- 推进 `src/<module>` 自管 `CMakeLists.txt`
- 清理临时目录和重复目录
- 为 M01/M02/M03/M04/control 建立清晰边界

### Phase 5：脚本与部署落地

- 补齐服务器原生构建脚本
- 补齐 WSL/Linux 交叉构建脚本
- 固化 deploy 安装脚本、样例配置、systemd 资产

### Phase 6：README 与基线同步

- 更新 README
- 更新部署文档
- 更新工作入口，形成可持续维护的进入点

---

## 4. 每阶段完成标准

- 每一阶段结束后都要保证当前分支仍可解释、可继续，不留下无人认领的半成品结构
- 影响构建行为的阶段必须补脚本或文档，不能只改代码不改入口
- 高风险改动优先做小步提交准备，但是否提交由当前工作节奏决定

---

## 5. 当前下一步

当前 `Phase 1` 到 `Phase 6` 的基础重构已经完成：

- 文档与工作入口已固化
- 顶层构建已模块化
- 测试目录与正式测试入口已稳定
- 服务器原生构建与交叉构建脚本已补齐
- 安装脚本与 systemd 部署闭环已在服务器验证通过

下一轮主线转入“真实模块恢复”，优先级如下：

1. 恢复 `M03` 的真实实现和最小可测目标
2. 再恢复 `M04` 或 `control`，避免同时摊开过大改动面
3. 每恢复一个模块，就同步补模块级构建、测试和部署说明
