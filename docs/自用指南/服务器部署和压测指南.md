# QDGZ300 服务器部署和压测指南（保姆级）

> **目标环境**：Linux ARM64 服务器（如：Kylin OS、Ubuntu 20.04+）
> **适用场景**：实际服务器上的编译、部署、运行和性能压测
> **前置要求**：服务器已安装基础编译工具链（GCC 7.3+、CMake 3.16+）
> **项目路径**：~/workspace/qdgz300_backend

## 1. 环境准备与检查

使用 SSH 登录到目标服务器：


```bash

tar xzf qdgz300_backend.tar.gz -C qdgz300_backend
cd qdgz300_backend
```

## 4. 编译项目

### 4.1 创建构建目录并配置 CMake

```bash
cd ~/workspace/qdgz300_backend

# 创建 Release 构建配置（生产环境）
cmake -S . -B build_release \
  -DCMAKE_BUILD_TYPE=Release \
  -DFORCE_OFFLINE_BUILD=ON \
  -DBUILD_TESTING=ON \
  -DENABLE_GPU=OFF \
  -DENABLE_PROTOBUF=OFF
```

**参数说明**：
- `-DCMAKE_BUILD_TYPE=Release`：优化编译，生成高性能二进制文件
- `-DFORCE_OFFLINE_BUILD=ON`：强制使用离线依赖，禁止联网下载
- `-DBUILD_TESTING=ON`：编译测试和压测工具
- `-DENABLE_GPU=OFF`：如果服务器无 GPU 或不使用 GPU 加速，设为 OFF
- `-DENABLE_PROTOBUF=OFF`：如果不需要 Protobuf 网关模块，设为 OFF

**预期输出**：
```
-- Found spdlog: using offline source from deps_offline/extracted/spdlog-1.13.0
-- Found yaml-cpp: using offline source from deps_offline/extracted/yaml-cpp-0.8.0
...
-- Configuring done
-- Generating done
-- Build files have been written to: .../build_release
```

### 4.2 开始编译

```bash
# 使用所有 CPU 核心并行编译
cmake --build build_release -j$(nproc)
```

**编译时间估计**：
- 32 核服务器：约 2-5 分钟
- 16 核服务器：约 5-10 分钟
- 8 核服务器：约 10-20 分钟

**预期输出末尾**：
```
[100%] Built target receiver_app
[100%] Built target fpga_emulator
[100%] Built target bench_m01_rx_limit
...
```

### 4.3 验证编译结果

```bash

# 检查压测工具
ls -lh build_release/fpga_emulator
```

**应该看到几十 MB 的可执行文件**，例如：

## 5. 配置文件准备

### 5.1 检查配置文件

```bash
cat config/receiver.yaml
```

### 5.2 根据实际网络环境修改配置

使用文本编辑器修改 `config/receiver.yaml`：

```bash
vi config/receiver.yaml
# 或使用 nano
nano config/receiver.yaml
```

**关键配置项说明**：

```yaml
network:
  listen_port: 9999           # UDP 监听端口
  bind_ips:                   # 接收数据的网卡 IP（三阵面）
    - "192.168.1.101"         # 阵面 1 - 根据实际网络环境修改
    - "192.168.2.101"         # 阵面 2 - 根据实际网络环境修改
    - "192.168.3.101"         # 阵面 3 - 根据实际网络环境修改
  cpu_affinity_map: [16, 17, 18]  # CPU 绑核，根据服务器 CPU 拓扑调整
  numa_node: 1                # NUMA 节点，单路服务器改为 0
```

**如何确定网卡 IP**：

```bash
# 查看所有网卡 IP
ip addr show
```

找到对应的物理网卡（如 eth0、eth1、eth2），将其 IP 填入 `bind_ips`。

**如何确定 CPU 拓扑**：

```bash
# 查看 NUMA 拓扑
lscpu | grep NUMA

# 查看详细 CPU 绑定
numactl --hardware
```

**对于单路服务器（只有 NUMA Node 0）**，修改配置：
```yaml
network:
  cpu_affinity_map: [4, 5, 6]  # 选择空闲核心
  numa_node: 0                  # 单路服务器只有 Node 0
```

### 5.3 创建运行时目录

```bash

cd ~/workspace/qdgz300_backend

./build_release/receiver_app --config config/receiver.yaml1
```

看到类似日志说明启动成功。按 `Ctrl+C` 退出。

### 6.2 后台运行（生产环境推荐）

#### 方式 A：使用 nohup

```bash
cd ~/workspace/qdgz300_backend

# 后台运行，日志输出到 ~/runtime_logs/receiver.log
nohup ./build_release/receiver_app --config config/receiver.yaml \
  > ~/runtime_logs/receiver.log 2>&1 &

# 记录进程 ID
echo $! > ~/runtime_logs/receiver.pid
```

**查看日志**：
```bash
tail -f ~/runtime_logs/receiver.log
```

**停止程序**：
```bash
kill $(cat ~/runtime_logs/receiver.pid)
```

#### 方式 B：使用 systemd 服务（推荐长期运行）

创建服务文件：

```bash
sudo vi /etc/systemd/system/qdgz300-receiver.service
```

填入以下内容（**根据实际路径修改**）：

```ini
[Unit]
Description=QDGZ300 Radar Backend Receiver
After=network.target

[Service]
Type=simple
User=devuser
WorkingDirectory=/home/devuser/workspace/qdgz300_backend
ExecStart=/home/devuser/workspace/qdgz300_backend/build_release/receiver_app --config /home/devuser/workspace/qdgz300_backend/config/receiver.yaml
Restart=on-failure
RestartSec=5s
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

启动服务：

```bash
# 重载 systemd 配置
sudo systemctl daemon-reload

# 启动服务
sudo systemctl start qdgz300-receiver

# 设置开机自启动
sudo systemctl enable qdgz300-receiver

# 查看服务状态
sudo systemctl status qdgz300-receiver

# 查看日志
sudo journalctl -u qdgz300-receiver -f
```

停止服务：

```bash
sudo systemctl stop qdgz300-receiver
```

## 7. 性能压测

### 7.1 单次快速压测

**场景**：快速验证系统是否正常接收数据。

```bash
cd ~/workspace/qdgz300_backend

# 发送 10 秒压测数据（40 万包/秒，包大小 1024 字节）
./build_release/fpga_emulator \
  --target-ip 192.168.1.101 \
  --target-port 9999 \
  --pps 400000 \
  --duration 10 \
  --payload-bytes 992 \
  --threads 2 \
  --output ~/runtime_logs/quick_test.json
```

**参数说明**：
- `--target-ip`：接收程序监听的 IP（config/receiver.yaml 中的 bind_ips 之一）
- `--target-port`：接收程序监听的端口（默认 9999）
- `--pps`：每秒发送的数据包数量（Packets Per Second）
- `--duration`：压测持续时间（秒）
- `--payload-bytes`：有效载荷字节数（992 + 32字节协议头 = 1024字节）
- `--threads`：发送线程数（建议 2-4）
- `--output`：结果输出 JSON 文件

**预期输出**：
```
Emulator started: target=192.168.1.101:9999, pps=400000, duration=10s
Sending...
[====================================] 100%
Actual sent: 4000000 packets
Actual PPS: 400000
Output: ~/runtime_logs/quick_test.json
```

**查看结果**：
```bash
cat ~/runtime_logs/quick_test.json
```

### 7.2 标准压测（逐步加压找极限）

使用标准压测脚本，自动测试不同负载下的性能：

```bash
cd ~/workspace/qdgz300_backend

# 给脚本添加执行权限
chmod +x tools/run_receiver_standard_bench.sh

# 运行标准压测（约 30-60 分钟）
./tools/run_receiver_standard_bench.sh \
  --build-dir build_release \
  --port 9999 \
  --measure-sec 60 \
  --results-dir ~/runtime_logs/bench_results
```

**参数说明**：
- `--build-dir`：指定构建目录
- `--port`：UDP 端口
- `--measure-sec`：每个测试步长的持续时间（秒）
- `--results-dir`：结果输出目录

**快速模式**（10秒/步，用于快速验证）：
```bash
./tools/run_receiver_standard_bench.sh \
  --build-dir build_release \
  --quick \
  --results-dir ~/runtime_logs/bench_results_quick
```

**压测过程**：脚本会自动进行多轮测试，逐步增加负载：
1. 10 万包/秒
2. 20 万包/秒
3. 40 万包/秒
4. 60 万包/秒
5. 80 万包/秒
6. ...

**当满足以下任一条件时自动停止**：
- CPU 使用率超过 95%
- 丢包率超过 0.1%

### 7.3 指定包率和包大小的压测

测试不同场景（小包、大包）下的性能：

**小包高频场景**（128 字节包，100 万包/秒）：

```bash
./build_release/fpga_emulator \
  --target-ip 192.168.1.101 \
  --target-port 9999 \
  --pps 1000000 \
  --duration 60 \
  --payload-bytes 96 \
  --threads 4 \
  --output ~/runtime_logs/small_packet_test.json
```

**大包低频场景**（1500 字节包，30 万包/秒）：

```bash
./build_release/fpga_emulator \
  --target-ip 192.168.1.101 \
  --target-port 9999 \
  --pps 300000 \
  --duration 60 \
  --payload-bytes 1468 \
  --threads 2 \
  --output ~/runtime_logs/large_packet_test.json
```

### 7.4 使用专用压测工具（M01 接收极限测试）

```bash
cd ~/workspace/qdgz300_backend

# M01 接收模块极限吞吐测试
./build_release/benches/bench_m01_rx_limit \
```

**该工具会**：
1. 绑定指定网卡接收数据
2. 测量原始 UDP 接收能力
3. 输出每秒包率、吞吐量、丢包率

## 8. 查看和分析压测结果

### 8.1 实时监控接收程序（在压测期间）

**方法 1：查看日志输出**

```bash
tail -f ~/runtime_logs/receiver.log
```

关注以下指标：
- `valid_pps`：有效接收包率
- `throughput_mbps`：吞吐量（Mbps）
- `loss_rate`：丢包率

**方法 2：使用系统工具监控**

```bash
# 监控进程 CPU 和内存使用
top -p $(pgrep receiver_app)

# 监控网络流量
watch -n 1 "ifconfig eth0 | grep bytes"

# 或使用 nload（需安装：sudo apt install nload）
nload eth0
```

**方法 3：使用 pidstat 监控 CPU**

```bash
# 安装 sysstat（如未安装）
sudo apt install sysstat

# 每秒监控一次
pidstat -u -p $(pgrep receiver_app) 1
```

### 8.2 分析压测结果 JSON 文件

#### 查看发送端结果

```bash
cat ~/runtime_logs/quick_test.json | python3 -m json.tool
```

**关键字段**：
- `actual_pps`：实际发送包率
- `target_pps`：目标包率
- `total_packets`：总发送包数
- `duration_sec`：实际持续时间

#### 查看接收端结果

如果接收程序输出了 JSON 统计（部分版本支持），查看：

```bash
cat ~/runtime_logs/m01_rx_limit.json | python3 -m json.tool
```

**关键字段**：
- `valid_pps`：有效接收包率
- `throughput_mbps`：吞吐量
- `loss_rate`：丢包率（小于 0.001 为优秀）
- `cpu_usage_percent`：CPU 使用率

### 8.3 标准压测结果分析

标准压测脚本会生成多个文件：

```bash
ls -lh ~/runtime_logs/bench_results/
```

**文件列表**：
- `case_01_rx.json` / `case_01_tx.json`：第 1 轮测试的接收和发送数据
- `case_01_rx.log` / `case_01_tx.log`：对应日志
- `case_01_cpu.log`：CPU 使用率监控
- `case_01_case.json`：测试用例汇总
- ...

**查看汇总报告**（如果脚本生成）：

```bash
cat ~/runtime_logs/bench_results/summary.txt
```

或手动统计：

```bash
# 提取所有测试的丢包率
for f in ~/runtime_logs/bench_results/case_*_case.json; do
  echo -n "$(basename $f): "
  cat "$f" | python3 -c "import sys, json; d=json.load(sys.stdin); print(f\"PPS={d.get('target_pps', 0)}, Loss={d.get('loss_rate', 0):.4f}\")"
done
```

### 8.4 性能指标参考标准

| 指标 | 优秀 | 良好 | 需优化 |
|------|------|------|---------|
| **单阵面包率** | > 120 万包/秒 | > 80 万包/秒 | < 80 万包/秒 |
| **三阵面总包率** | > 300 万包/秒 | > 200 万包/秒 | < 200 万包/秒 |
| **丢包率** | < 0.01% | < 0.1% | > 0.1% |
| **CPU 使用率** | < 80% | < 90% | > 95% |
| **吞吐量（1KB包）** | > 1 Gbps | > 800 Mbps | < 800 Mbps |

## 10. 快速故障排查

### 10.1 程序启动失败

**检查配置文件语法**：
```bash
# 使用 Python 检查 YAML 语法
python3 -c "import yaml; yaml.safe_load(open('config/receiver.yaml'))"
```

**检查端口是否被占用**：
```bash
sudo netstat -tulnp | grep 9999
```

如果被占用，杀掉占用进程或修改端口。

**检查网卡 IP 是否存在**：
```bash
ip addr show | grep -E "192.168.(1|2|3).101"
```

如果不存在，修改 `config/receiver.yaml` 中的 `bind_ips`。

### 10.2 压测丢包严重

**检查 CPU 绑核是否生效**：
```bash
# 查看接收程序的 CPU 亲和性
taskset -cp $(pgrep receiver_app)
```

**增大系统 UDP 接收缓冲区**：
```bash
# 临时修改（立即生效）
sudo sysctl -w net.core.rmem_max=268435456
sudo sysctl -w net.core.rmem_default=268435456

# 永久修改（重启后生效）
echo "net.core.rmem_max=268435456" | sudo tee -a /etc/sysctl.conf
echo "net.core.rmem_default=268435456" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p
```

**检查网卡硬件队列**：
```bash
# 查看网卡统计（查找 dropped、errors）
ifconfig eth0

# 或使用 ethtool
sudo ethtool -S eth0 | grep -E "drop|error"
```

### 10.3 性能达不到预期

**确认 CPU 频率是否锁定在性能模式**：
```bash
# 查看当前 CPU 频率调节器
cat /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor

# 设置为性能模式（所有核心）
echo performance | sudo tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
```

**关闭不必要的功能节约 CPU**：
- 如不需要 GPU 加速：确保 `ENABLE_GPU=OFF`
- 如不需要 Protobuf 网关：确保 `ENABLE_PROTOBUF=OFF`
- 降低日志级别：在配置文件中设置 `logging.level: "WARN"`

**检查 NUMA 亲和**：
```bash
# 查看程序的 NUMA 内存分布
numastat -p $(pgrep receiver_app)
```

确保内存主要分配在配置的 NUMA 节点上。
