# 雷达接收模块压测保姆级教程（小白版）

这份教程带你从 0 到 1 跑通：

1. 启动接收程序（Receiver）
2. 启动实时可视化面板（Dashboard）
3. 启动发送程序（Sender）压测
4. 查看曲线并理解结果

全程只用本地机器，不需要数据库和云服务。

---

## 0. 你需要知道的前提

- 你在 Windows 上开发，命令在 **WSL Ubuntu** 里执行。
- 项目目录是：

```text
/mnt/d/WorkSpace/Company/Tower/qdgz300_backend
```

- 我们用到的 3 个程序：
  - Receiver: `build_wsl_native_perf/receiver_app`
  - Dashboard: `tools/receiver_perf_dashboard.py`
  - Sender: `build_wsl_native_perf/fpga_emulator`

---

## 1. 第一次先编译（只需偶尔做）

打开 WSL 终端，复制执行：

```bash
cd /mnt/d/WorkSpace/Company/Tower/qdgz300_backend

cmake -S . -B build_wsl_native_perf -DBUILD_TESTING=ON -DENABLE_GPU=OFF -DENABLE_PROTOBUF=OFF
cmake --build build_wsl_native_perf --target receiver_app fpga_emulator -j$(nproc)
```

如果这一步成功，后续可以直接运行，不用每次重编。

---

## 2. 启动 Receiver（窗口 1）

新开一个 WSL 终端，执行：

```bash
cd /mnt/d/WorkSpace/Company/Tower/qdgz300_backend
./build_wsl_native_perf/receiver_app --config configs/default.yaml
```

你会看到类似日志：

```text
Receiver started at 0.0.0.0 ports=9999
```

这个窗口不要关，保持运行。

---

## 3. 启动 Dashboard（窗口 2）

再开一个 WSL 终端，执行：

```bash
cd /mnt/d/WorkSpace/Company/Tower/qdgz300_backend
python3 tools/receiver_perf_dashboard.py --metrics-url http://127.0.0.1:8080/metrics --port 8090 --packet-size-bytes 1024
```

然后浏览器打开：

```text
http://127.0.0.1:8090
```

你会看到 4 条图：

- 吞吐量
- 包率
- CPU
- 丢包

---

## 4. 启动 Sender 压测（窗口 3）

再开一个 WSL 终端，执行一个 2 分钟测试：

```bash
cd /mnt/d/WorkSpace/Company/Tower/qdgz300_backend
./build_wsl_native_perf/fpga_emulator \
  --target-ip 127.0.0.1 \
  --target-port 9999 \
  --pps 400000 \
  --duration 120 \
  --payload-bytes 992 \
  --threads 2 \
  --output build_wsl_native_perf/runtime_logs/sender_run.json
```

说明：

- `payload-bytes 992` + 协议头约 32B，接近 1024B 包长测试。
- `pps 400000` 表示每秒 40 万包。

---

## 5. 看图时怎么判断是否正常

### 正常现象

- 包率曲线：明显上升并稳定在某个值附近。
- 吞吐曲线：跟包率同步变化。
- 丢包曲线：大多数时间接近 0。
- CPU 曲线：有波动但不应长时间接近 100%。

### 异常现象

- 4 条图都接近 0：
  - 通常是没启动 Sender，或 Sender 已经结束。
- Dashboard 报错 `Connection refused`：
  - Receiver 没启动，或 metrics 端口不对。

---

## 6. 一条命令检查是否真的在跑

在任意 WSL 终端执行：

```bash
cd /mnt/d/WorkSpace/Company/Tower/qdgz300_backend
pgrep -a receiver_app || true
pgrep -a fpga_emulator || true
curl -s http://127.0.0.1:8080/metrics | egrep "^receiver_packets_received_total|^receiver_bytes_received_total" || true
```

如果 `receiver_packets_received_total` 在增长，说明接收在工作。

## 7. 结束测试（非常重要）

- Receiver 窗口按 `Ctrl + C`
- Dashboard 窗口按 `Ctrl + C`
- Sender 窗口按 `Ctrl + C`

## 8. 常见问题（小白高频）

### Q1：页面打开了，但曲线不动？

按顺序检查：

1. Receiver 是否在跑
2. Sender 是否在跑
3. `--target-port` 是否和 Receiver 监听端口一致（默认 9999）

### Q2：报错 `<urlopen error [Errno 111] Connection refused>`？

这是 Dashboard 连不到 Receiver 的 metrics，解决：

```bash
curl http://127.0.0.1:8080/metrics
```

- 如果 curl 失败，先启动 Receiver。
- 如果 Receiver 用了其他端口，把 `--metrics-url` 改成对应端口。

### Q3：吞吐曲线看起来偏低/不稳定？

- 先确认包长设置和你目标一致（`--packet-size-bytes`）。
- pps 太高可能触发系统瓶颈，先降低到 `200000` 再试。
