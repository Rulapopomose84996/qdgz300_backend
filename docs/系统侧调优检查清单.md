# 系统侧调优检查清单

## 1. 目标与适用范围

本文档用于 QDGZ300 后端在 Linux ARM64 服务器上的系统侧性能调优与验收检查，重点覆盖以下目标：

- 保持三阵面独立接收线程架构不变
- 降低跨 NUMA 访问、IRQ 干扰与调度抖动
- 提高 10GbE 数据面的稳定吞吐能力
- 为后续代码侧优化提供稳定、可复现的运行基线

适用对象：

- 运维部署人员
- 联调测试人员
- 性能调优人员

建议适用平台：

- CPU：飞腾 S5000C 或同级 ARM64 多 NUMA 平台
- OS：银河麒麟 / Ubuntu / 其他 Linux 发行版
- NIC：Intel X710 或同级 10GbE 多队列网卡

## 2. 调优原则

- 数据面 CPU 核与管理面 CPU 核严格分离
- 10G NIC、处理线程、热路径内存优先落在同一 NUMA Node
- 所有系统侧配置应可审计、可复现、可回滚
- 每次只调整一类参数，保留调优前后记录
- 未完成基线检查前，不进入正式压测或联调阶段

## 3. 推荐目标基线

### 3.1 CPU 与 NUMA 规划

- Node 1：数据面专用
- CPU 16-18：三阵面接收线程
- CPU 19-31：数据面保留核心，可供后续阵面内 worker、重组、信号处理使用
- CPU 0-15：管理面、日志、监控、SSH、磁盘、BMC、压测控制脚本

### 3.2 网卡角色划分

- 三路 10G 口：仅用于雷达数据接收
- 1G 管理口：仅用于 SSH、文件传输、监控页面、运维命令
- 禁止将高频管理流量混入 10G 数据面网口

### 3.3 进程运行要求

- 接收进程以固定亲和性运行
- 数据面线程与 NIC IRQ 必须对位到 Node 1
- 管理进程、日志、抓包、额外压测工具不占用 CPU 16-31

## 4. 调优检查项

以下检查项按优先级排序。`P0` 表示进入正式联调前必须完成，`P1` 表示建议完成，`P2` 表示压测与极限探测阶段完成。

### 4.1 P0: CPU 隔离

检查目标：

- 隔离 CPU 16-31，避免普通进程和内核噪声干扰实时数据面

建议配置：

- 内核启动参数加入：
  - `isolcpus=16-31`
  - `nohz_full=16-31`
  - `rcu_nocbs=16-31`

检查命令：

```bash
cat /proc/cmdline
```

通过标准：

- 输出中包含 `isolcpus=16-31`
- 建议同时包含 `nohz_full=16-31` 与 `rcu_nocbs=16-31`

补充检查：

```bash
ps -eLo pid,psr,comm | awk '$2>=16'
```

通过标准：

- 除接收进程及明确允许的数据面线程外，不应有常规进程长期运行在 CPU 16-31

回滚说明：

- 删除 GRUB 启动参数并重启系统

### 4.2 P0: NUMA 对位

检查目标：

- 10G NIC、接收线程与热路径内存优先位于 Node 1

检查命令：

```bash
numactl --hardware
cat /sys/class/net/enP1s25f0/device/numa_node
cat /sys/class/net/enP1s25f1/device/numa_node
cat /sys/class/net/enP1s25f2/device/numa_node
```

通过标准：

- 三个数据面网口的 `numa_node` 均为 `1`

启动建议：

```bash
numactl --cpunodebind=1 --membind=1 ./receiver_app --config config/receiver.yaml
```

运行后检查：

```bash
numastat -p $(pgrep receiver_app)
```

通过标准：

- 内存分配主要落在 Node 1
- 远程节点访问占比应尽可能低

### 4.3 P0: IRQ 绑定

检查目标：

- 10G NIC 中断只打到 Node 1 的目标核心
- 非关键 IRQ 从数据面核心移出

检查命令：

```bash
systemctl status irqbalance
cat /proc/interrupts
```

建议操作：

- 关闭 `irqbalance`，或将数据面 NIC IRQ 排除出自动平衡
- 将 10G NIC IRQ 绑定到 `16-18` 或 `16-31`
- 将管理口、磁盘、BMC 等 IRQ 固定到 `0-15`

示例：

```bash
echo 16-31 > /proc/irq/<IRQ_NUM>/smp_affinity_list
```

通过标准：

- 数据面 NIC 的中断计数主要增长在 CPU 16-31
- CPU 0-15 不承担 10G 数据面 IRQ

风险提示：

- IRQ 重新分配后可能短时影响网络，需在维护窗口执行

### 4.4 P0: 线程亲和与进程部署

检查目标：

- 接收进程和关键线程按设计绑定 CPU 16/17/18

检查命令：

```bash
taskset -pc $(pgrep receiver_app)
ps -eLo pid,tid,psr,comm | grep receiver_app
```

通过标准：

- 阵面 1/2/3 接收线程分别固定在 CPU 16/17/18
- 监控、日志、抓包等辅助线程不长期占用数据面保留核

部署建议：

- 通过 `numactl`、`taskset` 或 systemd `CPUAffinity=` 固化部署参数
- 接收进程与监控面板、发送器、抓包脚本分开绑定

### 4.5 P0: 防火墙与业务 UDP 端口放行

检查目标：

- 确保数据面业务 UDP 端口不会被 `firewalld` / `iptables` 默认策略拦截
- 避免出现“`tcpdump` 可见 UDP 包，但应用 socket 与 `netstat -su` 统计均为 0”的假象

检查命令：

```bash
systemctl status firewalld
sudo firewall-cmd --list-ports
sudo iptables-save | grep -E '9999|REJECT|IN_public'
```

建议操作：

```bash
sudo firewall-cmd --permanent --zone=public --add-port=9999/udp
sudo firewall-cmd --reload
```

若需临时放行并限定到接收口，可使用：

```bash
sudo iptables -I IN_public_allow 1 -i enP1s25f0 -p udp --dport 9999 -j ACCEPT
```

通过标准：

- `9999/udp` 已被防火墙显式放行
- 从外部发往 `192.168.1.101:9999` 的业务流量可被本机 UDP socket 正常接收

联动验收：

```bash
ss -lunp | grep 9999
netstat -su
curl -s http://127.0.0.1:8080/metrics | egrep '^(receiver_socket_packets_received_total|receiver_packets_received_total)'
```

通过标准：

- `ss -lunp` 可见 `receiver_app` 监听 `192.168.1.101:9999`
- `receiver_socket_packets_received_total` 与 `receiver_packets_received_total` 随发包增长
- 不再出现“`tcpdump` 抓到包，但 UDP socket 完全无计数”的现象

### 4.6 P1: NIC Ring Buffer

检查目标：

- 减少网卡接收溢出，增加突发流量吸收能力

检查命令：

```bash
ethtool -g enP1s25f0
ethtool -g enP1s25f1
ethtool -g enP1s25f2
```

建议配置：

```bash
ethtool -G enP1s25f0 rx 4096 tx 4096
ethtool -G enP1s25f1 rx 4096 tx 4096
ethtool -G enP1s25f2 rx 4096 tx 4096
```

通过标准：

- Rx ring 达到网卡支持上限，优先目标为 `4096`

### 4.7 P1: RSS / 多队列配置

检查目标：

- 避免单队列过热，确保多队列网卡资源可用

检查命令：

```bash
ethtool -l enP1s25f0
ethtool -x enP1s25f0
```

检查要点：

- 确认网卡开启多队列
- 确认 RSS 哈希与队列分配正常
- 若每阵面只使用一个物理口，也应确认该口未退化为异常单队列状态

通过标准：

- 网卡支持并启用预期数量的 Rx queue
- 队列与目标 CPU 策略一致

### 4.8 P1: 网卡 Offload 策略

检查目标：

- 降低聚合带来的额外延迟与不确定性

检查命令：

```bash
ethtool -k enP1s25f0
```

建议测试项：

- `gro off`
- `lro off`

示例：

```bash
ethtool -K enP1s25f0 gro off lro off
ethtool -K enP1s25f1 gro off lro off
ethtool -K enP1s25f2 gro off lro off
```

执行原则：

- 保留一份默认配置和一份低延迟配置做 A/B 对照
- 不一次性关闭全部 offload，避免引入新的 CPU 瓶颈

### 4.9 P1: Socket 与内核网络缓冲

检查目标：

- 防止内核网络栈在应用层之前先发生拥塞和丢包

建议 sysctl：

```bash
net.core.rmem_max = 536870912
net.core.wmem_max = 536870912
net.core.netdev_max_backlog = 250000
net.ipv4.udp_rmem_min = 262144
net.ipv4.udp_wmem_min = 262144
```

持久化建议：

- 将参数写入 `deploy/sysctl/90-qdgz300.conf`
- 在麒麟 V10 上额外部署 `deploy/systemd/qdgz300-sysctl.service`
- 原因：`/usr/lib/sysctl.d/kylin.conf` 会在 `sysctl --system` 过程中晚于 `/etc/sysctl.d/*.conf` 再次覆盖网络参数

检查命令：

```bash
sysctl net.core.rmem_max
sysctl net.core.netdev_max_backlog
```

通过标准：

- 系统级接收缓冲上限不低于应用 `SO_RCVBUF` 的目标值

### 4.10 P1: HugePages

检查目标：

- 为后续大块内存池、重组缓冲和信号处理缓冲预留低 TLB 开销的页资源

建议策略：

- 第一阶段先启用 `2MB HugePages`
- 第二阶段根据实际内存规模和业务模型评估 `1GB HugePages`

检查命令：

```bash
grep Huge /proc/meminfo
cat /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages
```

通过标准：

- HugePages 已预留且不会影响系统正常启动与管理面服务

注意事项：

- 当前接收代码未直接使用 hugepage 池，但系统应先具备资源条件

### 4.11 P2: 电源与频率策略

检查目标：

- 降低 CPU 频率波动和深度节能状态带来的时延抖动

检查命令：

```bash
cpupower frequency-info
cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor
```

建议配置：

- governor 设为 `performance`
- 在 BIOS 或平台工具中关闭不必要的深度 C-state
- 建议仅对数据面 `CPU 16-31` 持久化为 `performance`，管理面 `CPU 0-15` 保持系统默认策略

通过标准：

- CPU 运行在性能优先模式

推荐持久化文件：

- `deploy/systemd/cpu-performance.service`
- `tools/cpu_performance.sh`

### 4.12 P2: 链路质量与速率

检查目标：

- 排除物理链路协商异常、降速、CRC 错误与丢包

检查命令：

```bash
ethtool enP1s25f0
ethtool -S enP1s25f0 | grep -E "drop|error|crc|miss|fifo"
ip link show enP1s25f0
```

通过标准：

- 链路协商为 `10Gb/s`
- 无持续增长的 CRC、FIFO、miss、drop 异常计数

### 4.13 P2: 运行期观测

检查目标：

- 在压测和联调阶段及时发现系统侧瓶颈

推荐命令：

```bash
pidstat -u -t -p $(pgrep receiver_app) 1
pidstat -w -p $(pgrep receiver_app) 1
numastat -p $(pgrep receiver_app)
perf top -p $(pgrep receiver_app)
```

重点关注：

- 每线程 CPU 使用率
- 主动/被动上下文切换次数
- NUMA 本地/远程访问分布
- 内核收包路径热点函数占比

经验判断：

- 高性能模式下，数据面线程上下文切换应尽量接近于 0
- 若 `softirq`、`__netif_receive_skb_core` 或 IRQ 栈占比过高，应优先回查 IRQ、ring 与 RSS 配置

## 5. 执行顺序建议

建议按以下顺序推进，避免多变量同时变化导致定位困难：

1. 完成 CPU 隔离与 NUMA 对位
2. 完成 IRQ 绑定
3. 固化进程启动方式与线程亲和
4. 调整 NIC ring、RSS、多队列
5. 调整 offload 与 sysctl
6. 预留 HugePages
7. 进入压测并记录观测结果

## 6. 验收记录模板

每次调优建议记录以下内容：

- 日期与执行人
- 服务器型号与 OS 版本
- CPU / NUMA 拓扑摘要
- 三路 10G 网口名称与对应 NUMA Node
- 内核启动参数
- IRQ 绑定结果
- NIC ring / queue / offload 配置
- sysctl 调整项
- HugePages 预留情况
- 联调或压测命令
- 调优前后吞吐、丢包、CPU、上下文切换、numastat 对比

## 7. 当前项目建议的最小通过标准

在当前阶段，建议至少满足以下条件后再进入正式联调：

- 已配置 `isolcpus=16-31`
- 三个 10G 网口确认位于 `NUMA Node 1`
- 三路 NIC IRQ 已绑定至 `CPU 16-31`
- 接收进程以 `numactl --cpunodebind=1 --membind=1` 启动
- 三个网口 Rx ring 已提升到 `4096` 或硬件允许上限
- `irqbalance` 已关闭或已做明确排除
- 已具备 `pidstat`、`numastat`、`ethtool` 基本观测能力

## 单阵面本地自回环调试指标

适用场景：

- 仅启用单阵面接收
- 本机双口直连或通过 network namespace 做本地真实自发自收
- 目标是定位数据卡在 `socket -> source filter -> parse -> validate -> dispatch` 的哪一层

推荐同时观察以下 4 组指标：

### 1. Socket 层原始收包

```bash
curl -s http://127.0.0.1:8080/metrics | egrep \
'^(receiver_socket_packets_received_total|receiver_socket_bytes_received_total|receiver_socket_receive_batches_total|receiver_socket_source_filtered_total|receiver_socket_receive_errors_total)'
```

判读：

- `receiver_socket_packets_received_total` 增长：`recvmmsg/recvfrom` 已经从 UDP socket 取到包
- `receiver_socket_source_filtered_total` 增长：包被源过滤挡掉，未进入协议处理
- `receiver_socket_receive_errors_total` 增长：socket 层有异常或缓冲分配失败

### 2. Pipeline 分层通过量

```bash
curl -s http://127.0.0.1:8080/metrics | egrep \
'^(receiver_pipeline_packets_entered_total|receiver_pipeline_parse_ok_total|receiver_pipeline_validate_ok_total|receiver_packets_received_total)'
```

判读：

- `receiver_pipeline_packets_entered_total` 增长：包已进入上层回调
- `receiver_pipeline_parse_ok_total` 不增长：卡在协议解析
- `receiver_pipeline_validate_ok_total` 不增长：卡在协议校验
- `receiver_packets_received_total` 不增长：说明未完成分发后的有效接收统计

### 3. 丢包与时延

```bash
curl -s http://127.0.0.1:8080/metrics | egrep \
'^(receiver_packets_dropped_total|receiver_processing_latency_us_p99|receiver_processing_latency_us_p999|numa_local_memory_pct)'
```

判读：

- `receiver_packets_dropped_total{...}`：按具体原因定位协议层丢弃
- `receiver_processing_latency_us_p99/p999`：观察热路径尾延迟
- `numa_local_memory_pct`：验证当前单阵面收包是否仍保持本地 NUMA 分配

### 4. 内核 UDP 视角

```bash
netstat -su
ss -ulm | grep 9999 -A1
```

判读：

- `packet receive errors` / `receive buffer errors`：内核 UDP socket 已丢包
- `ss -ulm` 中的 `rb`：确认 socket 实际接收缓冲大小
- `ss -ulm` 中的 `d`：socket 层累计 drop 数

### 推荐联动观察命令

```bash
watch -n 1 '
echo ===metrics===;
curl -s http://127.0.0.1:8080/metrics | egrep "^(receiver_socket_packets_received_total|receiver_socket_source_filtered_total|receiver_pipeline_packets_entered_total|receiver_pipeline_parse_ok_total|receiver_pipeline_validate_ok_total|receiver_packets_received_total|receiver_packets_dropped_total|receiver_processing_latency_us_p99|receiver_processing_latency_us_p999|numa_local_memory_pct)";
echo;
echo ===udp===;
ss -ulm | grep 9999 -A1;
'
```
