## 数据接收模块（M01）优化空间分析

### 现状基线

| 指标 | 当前值 | 设计天花板 |
|------|--------|-----------|
| 单阵面轻量收包 | 325k PPS | 网卡线速 ~820k PPS@1.5KB |
| 单阵面完整路径 | **85k PPS** | 应趋近 325k |
| 完整路径吞吐 | ~1.02 Gbps | 单阵面峰值 3.66 Gbps |
| **瓶颈倍数** | **3.8×** | 轻量→完整之间 |

当前瓶颈**不在网络收包侧**，而在收包之后的管线处理阶段。以下按影响等级列出重大优化点。

---

### 一、架构层面（分层设计）—— 影响级别：决定性

#### 1. 处理线程调度模型：`sleep_for(10μs)` 轮询 vs. 事件驱动

当前处理线程在 SPSC 队列为空时执行 `std::this_thread::sleep_for(std::chrono::microseconds(10))`（见 app_run.cpp）。这是一个**被动等待 + busy-poll 的混合体**：

- **10μs sleep 在 Linux CFS 调度下实际可能休眠 50-100μs**（调度粒度 + 上下文切换开销）
- 在高速突发场景下（CPI 数据以 burst 形式到达），处理线程可能正在 sleep 而错过数据到达窗口
- **与设计文档 [E3-8~9] "热路径禁止 sleep/condvar/sem_wait 等被动等待"矛盾**

**优化方向**：处理线程应采用 **spin-then-yield** 策略——先自旋 N 次（例如 256 次 `try_pop` + `_mm_pause`/`yield`），自旋耗尽后再做极短 `sched_yield()` 或 `nanosleep(1μs)`。对于实时性要求极高的场景，可考虑 `SCHED_FIFO` 实时调度 + 纯自旋。

#### 2. 主事件循环超时扫描与处理线程的竞争

当前 `check_timeouts()` 和 `check_timeout()`（Reassembler/Reorderer 的超时扫描）在**主线程 100ms 周期**中调用（app_run.cpp），但 Reassembler 和 Reorderer 的状态**同时被处理线程写入**。

- Reassembler 的 `contexts_` map 被处理线程 `process_packet()` 和主线程 `check_timeouts()` 同时访问——**如果没有内部锁，这是数据竞争**
- Reorderer 的 `out_of_order_buffer_` 同理

**优化方向**：超时扫描应移入各阵面的处理线程内部（例如每 N 次 pop 后执行一次），消除跨线程共享和潜在锁竞争。

#### 3. DeliveryInterface 的 `delivery_mutex` 序列化

三个阵面的 Reorderer 输出最终汇聚到共享的 `DeliveryInterface`，通过 `ctx.delivery_mutex` 互斥（app_init.cpp）。在投递目标是 SharedMemory 或 UnixSocket 时，这把锁重新引入了串行化瓶颈。

**优化方向**：
- 每阵面独立 delivery channel（例如独立 shm segment / socket），彻底消除共享
- 或改用 MPSC 无锁队列汇聚，由单独的 egress 线程批量写出

### 二、数据路径优化 —— 影响级别：关键

#### 4. 四层 memcpy 复制链（已在性能报告中标记为 P1）——✅ 已实施

~~这是现有报告已识别但**尚未修复**的最大热路径问题。~~ 已完成零拷贝优化实施。

**优化前复制链**：
```
Dispatcher.enqueue_packet() → vector resize + memcpy      ①  ← 已在 P1-9 优化中消除（直调 handler）
Reassembler.add_fragment()  → make_unique + memcpy        ②  ← 本次零拷贝优化消除
Reassembler.complete()      → make_unique + memcpy        ③  ← 已为 std::move（无拷贝）
Reorderer.insert()          → vector resize + memcpy      ④  ← 已有 insert_owned() 零拷贝路径
```

**实施方案**：

1. **FragmentRef 零拷贝分片引用**（`reassembler.h`）：引入 `FragmentRef` 结构体，持有 `PacketBuffer` 所有权。
   分片到达时将网络缓冲区的所有权转移到 `FragmentRef` 中（零拷贝），仅记录 payload 指针和大小。
   同时支持堆分配回退路径（`heap_buffer`），确保与单元测试的向后兼容。

2. **延迟组装**（`reassembler.cpp`）：`complete_reassembly()` 中执行一次性顺序写入，
   从 `fragment_refs` 数组按索引顺序拷贝到最终帧缓冲区。相比优化前逐片随机写入，
   消除了 `ReassemblyContext::assembled_payload` 的预分配 + memset(0)，
   且最终组装为顺序写入（cache-friendly）。

3. **全链路 PacketBuffer 传递**：
   - `app_run.cpp`：处理线程将 `RxEnvelope::packet_data`（PacketBuffer）通过 `dispatch(pkt, std::move(buffer))` 传递
   - `dispatcher.h/cpp`：新增 `ZeroCopyDataHandler` 回调类型 + `dispatch(ParsedPacket, PacketBuffer&&)` 重载
   - `app_init.cpp`：通过 `set_zero_copy_data_handler()` 注册零拷贝回调，调用 `reassembler->process_packet_zero_copy()`
   - 原有 `dispatch(ParsedPacket)` + `process_packet(ParsedPacket)` 路径完全保留（向后兼容测试）

**优化效果**：
- 消除 per-fragment memcpy（原复制链 #②），令分片到达的热路径为 O(1) 移动操作
- 消除 `assembled_payload` 预分配 + memset(0)（省去大缓冲区初始化开销）
- 最终组装改为单次顺序写入，缺失分片同步零填充
- PacketBuffer 生命周期精确覆盖重组周期，重组完成后自动归还 PacketPool

**优化方向**：PacketBuffer 零拷贝贯穿全链路。Reassembler 输出 scatter-gather 描述符列表（`{buffer_ptr, offset, length}` 数组），而非拷贝合并为连续 buffer。

#### 6. 收包循环的逐帧开销（已在性能报告中标记为 P1.3~P1.6）

| 优化项 | per-packet 开销 | 修复方式 |
|--------|----------------|----------|
| `vector<PendingPacket>` 构造/析构 | ~50ns×20k/s | 提升到循环外 clear+复用 |
| `clock_gettime()` 逐包调用 | 20-50ns | 同批次共享时间戳 |
| `atomic_load<shared_ptr>` 逐包 | ~30ns | 同批次缓存一次 |
| 64×3 memset | ~100ns×batch | 优化为增量清零 |

这四项合计在单阵面可节省约 **5-10% CPU**，三阵面效果叠加。

### 三、算法与数据结构 —— 影响级别：显著

#### 7. `frozen_keys_ring_` O(N) 线性扫描

每个分片到达时都需遍历最多 2048 个 frozen key 做去重检查。在 325k PPS 下约 **6.6 亿次比较/秒**。

**优化方向**：替换为 `std::unordered_set<ReassemblyKey>` + 时间轮/TTL 堆淘汰，O(N) → O(1)。

#### 8. `popcount16` 使用朴素位循环

ARM64 提供硬件 `__builtin_popcount`（CNT 指令），单条指令即可完成。当前逐位循环在每个分片的补零计算中都会调用。

### 四、资源与平台优化 —— 影响级别：中等但成本低

#### 9. 编译器优化未充分利用

| 项目 | 当前 | 推荐 | 预期收益 |
|------|------|------|----------|
| LTO | OFF | ON (`CMAKE_INTERPROCEDURAL_OPTIMIZATION_RELEASE=ON`) | 5-15% |
| `-march` | `armv8-a` | `armv8.2-a+crypto+crc` | LSE 原子操作加速 |
| `-mtune` | `cortex-a72` | `neoverse-n1`（更匹配飞腾 S5000C FTC663） | 流水线调度优化 |

这三项属于**零代码修改**的优化，只需调整 CMake 配置。

#### 10. 处理线程缺少 CPU 亲和性绑定

当前三个处理线程由 OS 自由调度（app_run.cpp），没有绑定到 NUMA Node 1 的特定核心。这意味着：
- 处理线程可能被调度到 Node 0（远端 NUMA 节点），导致跨节点内存访问延迟翻倍
- 处理线程可能与收包线程（CPU 16-18）竞争同一核心

**优化方向**：为处理线程绑定 CPU 19-21（Node 1 剩余核心），与收包线程互不干扰。

#### 11. PacketPool 预分配偏小

当前每阵面仅 `64 × 4 = 256` 个 65535B buffer ≈ 16MB。在突发场景下可能触发 fallback 堆分配。建议扩至 1024 个（~64MB），在 Node 1 的 32GB 内存中占比微不足道。

### 五、设计文档要求 vs. 实现差距 —— 需注意的未实现项

| 设计要求 | 当前状态 | 影响 |
|----------|---------|------|
| 心跳优先级：DSCP 硬件队列分离 | 仅软件 deque 优先 | 数据洪峰下心跳可能饿死 |
| ExecutionSnapshot 闭环验证 | 提取已实现，mismatch 检测未完全闭环 | 控制面一致性验证缺失 |
| `SO_RCVBUF` 回读验证 | 设置后未 `getsockopt` 确认 | 可能静默降级 |
| RxStage 队列深度/drop 导出到 Prometheus | 代码中有 `(void)` 占位 | 运维不可观测 |
| 录制旁路隔离（Recorder 不阻塞 Receiver） | Recorder 未接入 | 未来集成需确保隔离 |

### 结论

**存在显著优化空间**。当前完整路径（85k PPS）与轻量收包（325k PPS）之间 **3.8 倍的性能差距**主要来自管线处理阶段的设计问题，而非网络层能力不足。核心优化点集中在：

1. **数据拷贝消除**（P0）—— ✅ 已实施：四层 memcpy 零拷贝优化
2. **处理线程调度模型**（P0）—— sleep(10μs) 与实时性要求矛盾
3. **跨线程共享消除**（P1）—— 超时扫描的数据竞争 + delivery 互斥锁
4. **编译器优化**（P3）—— 零成本的 LTO + 指令集升级

将 #1 和 #2 解决后，完整路径 PPS 预期可提升至 **200k+**；配合 #3 和 #4 进一步逼近轻量收包的 325k 基线。
