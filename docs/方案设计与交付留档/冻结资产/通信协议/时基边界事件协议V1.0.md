# 时基边界事件协议V1.0

> 适用范围：前端 DACS / FPGA → 服务器 SPS
> 报文定位：**时基边界事件接口**
> 报文名称：**PPS Notify / 软件秒边界通知包**
> 报文类型：`PacketType = 0x05`

---

## 0 设计目标与定位

### 0.1 协议定位

本报文用于在 **服务器侧没有硬件秒边界输入、也不依赖本地硬中断节拍** 的条件下，把前端执行侧已经感知到的物理秒边界，上送为后端可消费的 **软件秒边界事件**。

该报文的作用不是让服务器“获得了真正的硬件 1PPS”，而是为控制面建立一套统一、可追踪、可回放、可用于调度推进的秒边界软件语义。

### 0.2 设计边界

本报文**负责提供边界事实**，不直接承担以下职责：

1. 不直接触发前端表切换
2. 不直接裁定服务器进入哪个运行状态
3. 不直接让数据面推进处理周期
4. 不替代 GNSS / PTP / 本地晶振本身的授时链路
5. 不承诺硬件级高精度同步保证

PPS 接收线程的职责是接入、校验、去重、投递；权威周期状态由时基管理组件维护；任务调度中心消费该状态推进周期；数据面只消费共享周期状态与参数快照。

### 0.3 最小语义冻结

PPS Notify 至少必须表达以下 6 类信息：

1. 边界序号
2. 对应 UTC 秒值
3. 前端本地记录的边界时间
4. 时间来源
5. 时间质量
6. 边界标志位

这是该报文的最小冻结语义。

## 1 平面归属与传输规则

### 1.1 平面定义

系统整体协议采用五平面划分，其中第 5 类正式接口定义为：

| 平面     |              报文类型 | 方向         | 核心语义      | 可靠性策略                  |
| ------ | ----------------: | ---------- | --------- | ---------------------- |
| 时基边界平面 | `0x05` PPS Notify | DACS → SPS | 上送软件秒边界事件 | 尽力传输 + 去重/乱序检测 + 失边界降级 |

说明：

* 该平面独立于控制、数据、遥测、维护平面
* 它属于**控制面的时基输入**，不是数据面输入
* 协议总表纳入 `PacketType = 0x05`

### 1.2 端口分配

采用独立 UDP 端口，避免与数据/心跳复用：

| 报文类型                | 方向         | 源端口  | 目标端口 |
| ------------------- | ---------- | ---- | ---- |
| PPS Notify (`0x05`) | DACS → SPS | 8890 | 8890 |

原因：

1. 与 `0x03/0x04` 的 9999 端口分离，避免数据洪峰干扰边界接入
2. 便于单独绑核、单独统计、单独抓包
3. 便于 PPS 接收线程独立实现“快收、快校验、快投递”

### 1.3 可靠性与重传原则

PPS Notify 冻结为 **不做应用层重传、不做 ACK 闭环**。

原因：

1. 该报文表达的是“边界事实”，不是事务型控制对象
2. 边界报文天然带有时效性，迟到重传价值很低
3. 后端应通过 `PpsSeq` 检测重复、乱序、失边界，而不是依赖 ACK
4. 若发生丢失，应由时基管理组件和调度中心依据失边界规则进入守时、冻结表、降级或安全态，而不是要求前端补发旧边界

这与现有“提前下发、边界生效、秒内按表推进”的总体原则一致。

## 2 通用报文头使用规则

PPS Notify 继续复用 **V1.0 通用报文头 32B**，字段定义、字节序、`SourceID/DestID/ControlEpoch` 规则与现有协议保持一致。

### 2.1 通用头约束

* `PacketType = 0x05`
* 多字节字段均采用 **小端序**
* `ProtocolVersion = 0x10`
* `SourceID` 必须为具体阵面 DACS，如 `0x11 / 0x12 / 0x13`
* `DestID` 固定为 `0x01`（SPS）
* `ControlEpoch` 继续作为链路纪元隔离字段使用

### 2.2 Common Header.Timestamp 语义扩展

对 `PacketType = 0x05`，通用头 `Timestamp` 解释为：

> **本 PPS Notify 报文的发送时间戳（Unix Epoch ms）**

说明：

* 它表示“前端把边界事实发出来”的时刻
* 它不是“物理边界到达时刻”的唯一权威字段
* 真正的边界时刻与 UTC 秒值，应以 PPS Payload 内的字段为准

### 2.3 SequenceNumber 语义

`Common Header.SequenceNumber` 继续按发送端逻辑流递增，作为链路级序号使用。
但 **软件秒边界去重与乱序判断的主键**，应优先使用 Payload 内的 `PpsSeq`，而不是单纯依赖通用头序列号。

理由：

* `SequenceNumber` 是通用逻辑流序号
* `PpsSeq` 是专门面向秒边界对象的时基序号
* 两者职责不同，不应混用

## 3 PPS Notify Payload 结构

### 3.1 包结构

```txt
┌──────────────────────────────────────────────┐
│ UDP Payload                                  │
├──────────────────────────────────────────────┤
│ Common Header (32B)                          │
│   PacketType = 0x05                          │
│   Timestamp  = 报文发送时刻                  │
├──────────────────────────────────────────────┤
│ PPS Notify Payload (固定 40B)                │
└──────────────────────────────────────────────┘
```

### 3.2 PayloadLen

```txt
PayloadLen = 40
UDP Payload Total = 32 + 40 = 72 Bytes
```

### 3.3 设计说明

* 固定长度，便于快速解析
* 4 字节自然对齐
* 无变长载荷
* 不携带大对象，不承载调度信息，不与控制表耦合

## 4 PPS Notify Payload 字段表

> Offset 基准：相对于 PPS Notify Payload 起始位置（即 UDP Payload Offset = 32）

| Offset | 字段名称                                 |         类型 | 字节数 | 说明                                                                                               |
| ------ | ------------------------------------ | ---------: | --: | ------------------------------------------------------------------------------------------------ |
| **0**  | **边界序号（PpsSeq）**                     |   `uint32` |   4 | **含义**：软件秒边界事件序号。<br/>**用途**：去重、乱序判断、丢边界检测。<br/>**规则**：每产生 1 次新的秒边界事件递增 1，允许自然回绕。                |
| **4**  | **UTC 秒值（UtcSeconds）**               |   `uint64` |   8 | **含义**：本次边界对应的 UTC 秒值，单位为秒。<br/>**定义**：Unix Epoch Seconds。<br/>**用途**：将逻辑边界与绝对时间绑定。              |
| **12** | **边界本地时间戳（EdgeLocalTimestampNs）**    |   `uint64` |   8 | **含义**：前端本地记录的边界到达时间戳，单位 ns。<br/>**语义**：记录“前端执行侧感知到物理边界”的本地时间。<br/>**用途**：延迟、抖动、重同步分析。           |
| **20** | **时间来源（TimeSource）**                 |    `uint8` |   1 | **含义**：本次边界所依据的时间来源。<br/>**取值**：`0x00=GNSS`，`0x01=PTP`，`0x02=Holdover/XO`，其余保留。                  |
| **21** | **时间质量（TimeQuality）**                |    `uint8` |   1 | **含义**：本次边界的时间质量档位。<br/>**取值**：`0x00=Locked`，`0x01=Holdover`，`0x02=Freerun/Unknown`。             |
| **22** | **边界标志位（EdgeFlags）**                 |   `uint16` |   2 | **含义**：本次边界的附加状态标志。<br/>**位定义**见 4.4。                                                            |
| **24** | **当前全局周期号建议值（SuggestedGlobalEpoch）** |   `uint64` |   8 | **含义**：前端基于本次边界感知所回显/建议的全局周期号。<br/>**用途**：便于后端诊断“前后端周期认知是否一致”。<br/>**说明**：该字段是诊断辅助字段，不是后端权威周期状态。 |
| **32** | **保留（Reserved）**                     | `uint8[4]` |   4 | **含义**：保留字段。<br/>**发送端**：必须填 0。<br/>**接收端**：主版本一致时允许非 0 并计数告警。                                   |
| **36** | **CRC32C**                           |   `uint32` |   4 | **含义**：PPS Notify Payload 完整性校验。<br/>**覆盖范围**：Offset `0..35`。                                    |

**总长度 = 40 字节**

## 4.4 EdgeFlags 位定义（冻结）

|    Bit | 名称                | 说明                 |
| -----: | ----------------- | ------------------ |
|    `0` | `EDGE_VALID`      | 本次边界有效             |
|    `1` | `DUP_EDGE`        | 检测到本次边界为重复边界       |
|    `2` | `OUT_OF_ORDER`    | 本次边界相对最近边界出现乱序     |
|    `3` | `TIME_JUMP`       | 边界对应 UTC 秒值出现非预期跳变 |
|    `4` | `RESYNC`          | 当前为重同步后的首个有效边界     |
|    `5` | `SRC_SWITCH`      | 时间来源发生切换           |
|    `6` | `QUALITY_DEGRADE` | 时间质量较上一边界下降        |
|    `7` | `QUALITY_RECOVER` | 时间质量较上一边界恢复        |
|    `8` | `LEAP_OR_ADJUST`  | 存在外部时统调整/特殊秒处理     |
| `9-15` | Reserved          | 发送端置 0             |

说明：

* `DUP_EDGE` 与 `OUT_OF_ORDER` 由前端本地检测后可上报；后端也应再做一次独立判断
* `RESYNC / SRC_SWITCH / QUALITY_DEGRADE / QUALITY_RECOVER` 是后端非常需要的诊断语义
* `TIME_JUMP` 用于标识本次秒值与前次秒值不符合预期递增关系

## 5 CRC32C 规则

### 5.1 算法

与现有协议保持一致，采用 **CRC-32C (Castagnoli)**。现有协议中 Control、ScheduleApplyAck、Heartbeat 已采用该算法。

### 5.2 覆盖范围

| 报文                  | 覆盖范围（Payload Offset） | 校验字段位置     |
| ------------------- | -------------------- | ---------- |
| PPS Notify (`0x05`) | `0 .. 35`            | `36 .. 39` |

### 5.3 校验失败处理

PPS Notify 的 CRC32C 校验失败时：

1. 该报文不得进入时基管理组件
2. PPS 接收线程应静默丢弃并计数
3. 不回 ACK
4. 计入 `pps_crc_error_total`

理由：

* PPS Notify 不是控制事务报文
* 错误的边界事实不能被“勉强消费”
* 也不值得为它做反向应答闭环

## 6 接收与判定规则

### 6.1 PPS 接收线程职责

PPS 接收线程只负责：

1. 收包
2. 基础字段校验
3. CRC32C 校验
4. `PpsSeq / UtcSeconds` 去重与乱序检查
5. 生成内部软件秒边界事件并投递

它不负责：

1. 推进调度周期
2. 直接换表
3. 直接裁定降级
4. 直接广播给数据面

这与秒边界设计说明中的职责分层保持一致。

### 6.2 基础校验顺序

接收顺序冻结为：

1. `Magic` 校验
2. `ProtocolVersion` 主版本校验
3. `PacketType == 0x05` 校验
4. `PayloadLen == 40` 校验
5. `DestID == 0x01` 校验
6. `SourceID` 是否为合法 DACS 阵面
7. `CRC32C` 校验
8. `TimeSource / TimeQuality / EdgeFlags` 合法性校验
9. `PpsSeq / UtcSeconds` 逻辑一致性校验

### 6.3 去重与乱序规则

以 `PpsSeq` 为主键，冻结如下：

* 若 `PpsSeq == last_pps_seq`：视为重复边界，丢弃并计数
* 若 `PpsSeq` 按半区间比较为旧值：视为乱序/旧包，丢弃并计数
* 若 `PpsSeq` 比 `last_pps_seq` 大 1：正常推进
* 若 `PpsSeq` 跨越大于 1：判定为丢边界，记录 `missed_pps_count += delta-1`

### 6.4 UTC 秒值一致性规则

同时检查 `UtcSeconds`：

* 正常情况下，新边界的 `UtcSeconds` 应比上次增加 1
* 若不满足，则标记 `TIME_JUMP`
* 即使 `PpsSeq` 正常，也不能完全忽略 `UtcSeconds` 异常
* 最终是否接受该事件，由时基管理组件裁定

### 6.5 进入时基管理组件的条件

只有在以下条件同时满足时，PPS 接收线程才允许投递：

1. 协议头合法
2. 长度合法
3. CRC 合法
4. `EDGE_VALID = 1`
5. 非重复、非旧序号
6. 基础字段值未越界

## 7 时基管理与调度消费规则

### 7.1 权威周期状态更新

时基管理组件消费有效 PPS Notify 后，更新至少以下状态：

* `pps_seq`
* `current_second`
* `global_epoch`
* `time_source`
* `time_quality`
* `edge_flags`
* `boundary_trust_level`

这与现有“权威全局周期状态”冻结口径一致。

### 7.2 调度中心消费方式

任务调度中心不得“认包换秒”，而应：

1. 读取时基管理组件维护的权威状态
2. 判定是否进入新周期
3. 更新本地 `TargetEffectiveEpoch / GlobalEpoch`
4. 决定是否正常推进、冻结表、续跑旧表、降级或进入安全态

### 7.3 数据面消费方式

数据面不得直接接收 PPS Notify 包。
数据面只允许通过以下两类对象消费边界语义：

1. 共享周期状态
2. 参数快照中的周期上下文

这一点应明确冻结，防止控制职责扩散到数据面。

## 8 异常处理与降级规则

### 8.1 PPS 包丢失

若 `PpsSeq` 检测到跳变，说明存在失边界：

* 时基管理组件记录 `missed_pps_count`
* 若短时少量丢失，可保持当前守时状态并告警
* 若连续超过阈值，调度中心应进入冻结表、旧表续跑、局部降级或安全态

### 8.2 时间质量下降

当 `TimeQuality` 由：

* `Locked -> Holdover`
* `Holdover -> Freerun/Unknown`

时，后端必须记录状态变化，并重新评估后续定时生效能力。现有协议对定时生效已经冻结了：`Holdover` 默认禁止定时生效，`Freerun/Unknown` 必须禁止。

### 8.3 时间来源切换

当 `TimeSource` 发生变化时：

* 必须记录 `TIME_SRC_SWITCH`
* 显控应可见授时来源切换
* 若从 GNSS 降到 PTP / Holdover，应提升告警等级

### 8.4 PPS 包迟到

PPS Notify 到达晚不等于边界无效，但应计入诊断：

```txt
pps_arrival_latency_ms =
CommonHeader.Timestamp(ms) - floor(EdgeLocalTimestampNs / 1_000_000)
```

说明：

* 这里主要用于统计网络通知延迟与抖动
* 迟到的 PPS 包仍可用于“边界事实记录”
* 但是否允许其驱动周期推进，由时基管理组件按策略判定

## 9 显控与日志可观测性

### 9.1 显控最小展示项

显控至少展示：

* 当前 `PpsSeq`
* 当前 `GlobalEpoch`
* 当前授时来源（GNSS / PTP / Holdover）
* 当前时间质量（Locked / Holdover / Freerun）
* 最近一次边界状态（正常 / 跳变 / 重同步 / 失边界）
* 连续失边界次数
* 当前是否处于旧表续跑 / 冻结表 / 降级态

这与现有显控“展示结论和态势”的原则一致。

### 9.2 日志事件

至少打点以下事件：

* `PPS_RX_OK(seq, utc_sec, src, quality)`
* `PPS_DUP(seq)`
* `PPS_OUT_OF_ORDER(seq)`
* `PPS_MISSED(last_seq, new_seq, miss_count)`
* `PPS_TIME_JUMP(last_utc, new_utc)`
* `TIME_SRC_SWITCH(old, new)`
* `TIME_QUALITY_CHANGE(old, new)`
* `PPS_REJECT(reason)`

## 11 冻结表述

> **PPS Notify 报文用于把前端执行侧感知到的物理秒边界，上送为后端控制面可消费的软件秒边界事件。该报文只提供边界事实，不直接承担调度裁决、表生效裁决或数据面驱动职责。PPS 接收线程负责快速接收、校验、去重与事件投递；时基管理组件据此维护权威全局周期状态；任务调度中心基于该状态推进调度周期并裁定后续控制行为；数据面通过共享周期状态与参数快照消费边界语义。该报文提供统一周期语义，不提供硬件级高精度同步保证。**
