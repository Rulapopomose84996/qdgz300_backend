## 还要直接指出两个当前风险

### 风险一：M01->M02 契约还没冻结

这会导致你“接收端详细设计”写了半天，最后在 `RawBlock` 所有权、数据视图、异常标志和生命周期上返工。这个问题应排第一优先级。

### 风险二：旁路落盘很容易污染主链路

环境基线里已经明确：`receiver_app` 运行在 Node 1 / CPU 16-31，原始录制应采用 **NVMe 一级缓冲 + 后台异步转存 `/data`**，录制线程应放在 CPU 0-15 / Node 0，避免侵占数据面核心；同时 `/data` 机械盘阵列持续顺序写入基线约 `374 MB/s`，不适合作为高频连续采集的唯一直接落盘目标。

这意味着：

* 旁路写盘不能挂在 processing thread 上
* 不能让 Node 1 承担机械盘长时间写入压力
* 不能把“录全”作为高于主链路稳定性的目标
