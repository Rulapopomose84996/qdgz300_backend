# 🏗️ 宏观架构缺失内容 - 第 1 部分：系统级核心模块设计

## 全局不变量索引

- 参数生效边界（effective_boundary）与 Static/Dynamic 规则（参照 SSOT）：见 [分块设计文档/顶层设计_3.md](分块设计文档/顶层设计_3.md#ssot-effective-boundary)
- time_quality 与时间红线（参照 SSOT）：见 [分块设计文档/顶层设计_3.md](分块设计文档/顶层设计_3.md#ssot-time-quality)
- SEV 分级与阈值（参照 SSOT）：见 [分块设计文档/顶层设计_2.md](分块设计文档/顶层设计_2.md#ssot-sev-def)

## 📌 概述

本文档完整回答了 " 宏观架构设计的缺失问题 " 第 1 部分，即**系统控制平面的 4 个核心模块**：

1. **ConfigManager** - 配置管理
2. **TimeSyncManager** - 时间同步管理
3. **HealthCheck** - 健康检查与监护
4. **DiagnosticsCenter** - 故障诊断

这 4 个模块在 Node0（Orchestrator 的协作者）上运行，共同构成系统的 " 可靠性基础设施 "。

## 1️⃣ ConfigManager：配置管理

### 1.1 系统定位

**ConfigManager 是系统级模块**，负责：
- 配置来源统一管理（文件/TCP 命令）
- 参数版本管理与版本化快照
- 变更审计与回溯
- 安全分发机制（对数据平面的参数变更分级）

**核心约束**：
- 数据平面（M01/M02/M03/M04）模块不直接读写共享配置对象
- 所有参数变更必须通过 ContentManager 下发，由各模块在安全点应用

### 1.2 参数变更的三层分类机制

所有配置参数必须标注变更等级，决定参数何时生效以及是否需要冲刷/重启：

| 变更等级 | 定义 | 生效时刻 | 需要冲刷？ | 示例参数 |
|---------|------|---------|----------|---------|
| **Hot Reload** | 无需冲刷、无需重启、立即见效 | 帧边界（模块下次读配置） | ❌ | CFAR 阈值、滤波权重、输出频率上限、告警阈值 |
| **Safe-Point Flush** | 需要冲刷局部状态但不重启进程 | 帧末边界或处理周期末 | ✅ | GPU 队列上限、Tracker 融合窗口、调度策略细节 |
| **Restart Required** | 需要进程级重启 | 进入 Standby 或 Shutdown 后 | ✅ 强制重启 | 线程绑核、NUMA 分配、预分配内存池大小、NIC IRQ 配置 |

**一致性对齐**：本节 Hot/Flush/Restart 与 `effective_boundary` 的边界含义对齐，具体原子边界与禁止项见 [分块设计文档/顶层设计_3.md](分块设计文档/顶层设计_3.md#ssot-effective-boundary)。

### 1.3 参数变更的完整生命周期

```txt
【1】发起
    显控或标准化接口发起参数变更请求 → ConfigManager

【2】验证
    ConfigManager执行范围/一致性检查
    + Orchestrator验证系统状态兼容性

【3】分类与分发
    ├─ Hot:    直接更新版本化快照，下一帧自动可见
    ├─ Flush:  Orchestrator下发 APPLY_CONFIG 控制事件
    │          数据面模块在安全点执行冲刷
    └─ Restart: 标记PENDING_RESTART，等待进程重启

【4】确认反馈
    ConfigManager记录版本号、时间戳、应用状态
    反馈给显控/上层系统

【5】日志与回溯
    诊断工具可查询"上次有效配置版本" vs "目标版本"
    快速定位参数冲突或版本不一致
```

### 1.4 快照隔离机制（保证帧一致性）

```c
// 伪代码：快照管理

struct ConfigSnapshot {
    uint32_t version;
    // 各参数字段
    float cfar_threshold;
    int gpu_timeout_ms;
    // ...
};

struct ConfigManager {
    ConfigSnapshot* config_active;      // 当前活跃版本
    ConfigSnapshot* config_next;        // 准备激活的版本

    void on_new_config(const ConfigSnapshot& cfg) {
        config_next = deep_copy(cfg);
        version++;
    }

    // 模块在帧开始时调用
    const ConfigSnapshot* acquire_read_snapshot() {
        return config_active;  // 整个帧都用这个版本
    }

    void release_read_snapshot() {
        // 帧结束，可以触发 config_active = config_next（原子操作）
    }
};
```

**好处**：
- 每帧的参数一致，无中途变化
- 避免临界区竞争
- Hot 参数无性能损耗

### 1.5 参数元表（冻结示例）

最终补充文档应包含一张 " 参数元表 "，定义所有可变参数：

| 参数名 | 所属模块 | 数据类型 | 默认值 | 范围 | 变更等级 | 生效时刻 | 备注 |
|--------|---------|---------|--------|------|---------|---------|------|
| CFAR_THR | M02 | float | 8.0 | [5.0, 15.0] | Hot | 帧边界 | 信号处理阈值，无状态重置 |
| GPU_TIMEOUT_MS | M02 | int | 50 | [10, 200] | Flush | Kernel 完成 | GPU 流水线清空，重启计时器 |
| TRACK_WINDOW_LEN | M03 | int | 10 | [5, 20] | Flush | TrackEpoch 末 | 融合窗口，需重置 Kalman 中间变量 |
| CPU_AFFINITY | M01 | uint64 | 0x060000 | - | Restart | 进程重启 | 绑核策略，运行期不变 |

## 2️⃣ TimeSyncManager：时间同步管理

### 2.1 系统定位

**TimeSyncManager 是独立系统级模块**，唯一职责是：
- 维护系统**权威时间源**的状态机
- 输出 " 时间是否可信 " 的硬信号给 Orchestrator
- 支持时间源丢失、降级、恢复的全生命周期

**冻结约束**（来自系统设计）：
- **Authoritative Time** = 外部授时唯一权威，用于信号处理和航迹生成
- **Wall-clock Time** = 仅用于日志和运维展示
- NTP 不被认可为权威源，仅作为辅助参考
- 与 DACS 协议 V3.1 对齐：仅当 DACS 时间质量为 Q0(Locked) 时允许“秒边界定时生效”，Q1/Holdover 默认禁止，Q2 必须禁止，并要求可观测错误码
- 对显控输出需提供可解释的 `clock_domain` 与 `time_quality` 映射，确保 TrackData 新鲜度判定与后端时间域一致

### 2.2 时间状态机（4 个状态）

| 状态 | 定义 | Orchestrator 决策 | 输出标志 |
|-----|------|------------------|--------|
| **UNSYNC** | 初始化中，未锁定授时源 | 禁止 Running；允许 Init 和 Standby | TIME_INIT |
| **SYNCED** | 已锁定，时间可信 | 允许 Running | TIME_LOCKED |
| **HOLDOVER** | 授时丢失但守时中 | 允许 Degraded（可选 Running+TIME_HOLDOVER 标志） | TIME_HOLDOVER |
| **HOLDOVER_EXPIRED** | 守时预算耗尽，时间不可信 | 强制进 Fault 或 Shutdown | TIME_NOT_AVAILABLE |

**对外映射**：time_state 与 TrackData `time_quality` 的对外映射、holdover 红线与停止输出判据见 [分块设计文档/顶层设计_3.md](分块设计文档/顶层设计_3.md#ssot-time-holdover)。

**time_quality 枚举映射表（SSOT 对齐 顶层设计_4.md A.3）**：

| 内部 time_state | 对外 time_quality | V3.1 TimeQuality |
|-----------------|-------------------|------------------|
| UNSYNC | FREERUN | Q2 |
| SYNCED | LOCK | Q0 |
| HOLDOVER | HOLDOVER | Q1 |
| HOLDOVER_EXPIRED | FREERUN | Q2 |

### 2.3 TimeSyncManager 的输出接口（给 Orchestrator）

模块对外只发布这些 " 可裁决的事实 "，不暴露算法细节：

```json
{
  "time_state": "SYNCED",              // 当前时间状态
  "auth_time_ok": true,                 // 权威时间是否可用于处理
  "dacs_time_quality": "Q0",            // Q0/Q1/Q2，对应DACS协议冻结语义
  "time_source": "GNSS|PTP|LOCAL",      // 时间来源，用于clock_domain映射
  "clock_domain": "GNSS",              // 对外输出域，映射到显控协议
  "holdover_budget_ms": 3500,           // 守时剩余预算（当HOLDOVER时有效）
  "offset_estimate_us": 0.5,            // 时间偏移估计值（诊断用）
  "drift_estimate_ppm": 1.2,            // 频率漂移估计（诊断用）
  "last_sync_time": "2026-02-25T14:30:00.000Z"  // 最后同步时刻
}
```

### 2.4 时间故障的根因诊断框架

| 故障现象 | 根因检查优先级 | 诊断手段 |
|---------|-----------------|---------|
| **Init 无法进 Standby** | 1. 授时源不可达？<br/>2. 偏差太大？<br/>3. 漂移太快？ | 查看 auth_source_reachable<br/>offset_estimate > threshold<br/>drift_estimate 判断 |
| **Running 中突然进 Degraded** | 1. 授时丢失？<br/>2. 质量劣化？<br/>3. 信号中断？ | 检查 TIME_SOURCE_LOST 事件<br/>offset 跳变监控<br/>timestamps 停止增长 |
| **TrackData 输出中断** | 需与 HealthCheck.pipeline_health 对比：<br/>- 时间不可信 → 根因是 TIME<br/>- 其他链路健康 → 根因在 M01-M04 | 时间状态时间线 + 管道关键指标对齐分析 |

### 2.5 Orchestrator 的时间校验规则（必须冻结）

```txt
// Orchestrator 中的状态转移规则

if TimeSyncManager.time_state == UNSYNC:
    禁止 Init → Running 转移
    允许 Init → Standby（作为"等待授时"中间态）

if TimeSyncManager.time_state == SYNCED:
    允许 Standby → Running

if TimeSyncManager.time_state == HOLDOVER:
    if elapsed_holdover > holdover_budget_ms:
        强制进 Fault
    else:
        允许进 Degraded（可选进Running+输出TIME_HOLDOVER标志）

// DACS 时间质量对“定时生效”的硬门限（V3.1）
if TimeSyncManager.dacs_time_quality != Q0:
  禁止“秒边界定时生效”，仅允许“立即生效”或进入Degraded
  并在控制回执/显控状态中显式标记原因

if TimeSyncManager 在<1s内转移两次（如SYNCED→HOLDOVER→SYNCED）:
    认为时间源不稳定，进 Degraded 且锁定不自动恢复
    (需显控手动干预)
```

## 3️⃣ HealthCheck：健康检查与监护

### 3.1 核心职责

HealthCheck 分为两个层面：

| 层面 | 目标 | 触发时机 | 输出 |
|-----|------|---------|------|
| **Boot Health** | 验证系统初始化完整性，确保所有冻结约束满足 | Init 阶段一次性检查 | 通过/失败 + 详细失败原因 |
| **Runtime Health** | 连续监护关键指标，转换为告警与状态转移驱动 | 运行期持续轮询（100ms 周期） | 告警事件 + 系统状态建议 |

### 3.2 Boot Health 检查清单

```txt
【1】线程绑核验证
    读取CPU affinity bitmap，与设计拓扑逐项比对
    失败 → "Expected CPU16, Got CPU17"，禁止进Standby

【2】GPU设备与Stream创建
    尝试创建3个CUDA stream（流水线约束）
    失败 → 设备不可达，禁止进Standby

【3】SPSC队列深度与预分配
    遍历所有队列，确认容量 ≥ StartupConfig.min_queue_depth
    失败 → "Raw队列预分配不足"，禁止进Standby

【4】时间前置条件
  检查 TimeSyncManager.time_state ≠ UNSYNC
  若 dacs_time_quality != Q0，仅允许进入Standby/Degraded，禁止进入Running并禁止定时生效

【5】接收链路验证
  尝试从3路UDP接收并重组1个CPI（timeout 500ms，允许IncompleteFrame）
  失败 → "UDP_0超时"，列出具体哪路失败

【6】控制通道就绪
    TCP监听端口是否已绑定，是否能accept连接
    失败 → "TCP显控通道无法监听"
```

**设计原则**：Boot 检查是 " 配置一致性 + 通路可达性 "，不涉及算法效能，避免虚报。

### 3.3 Runtime Health 监护与告警阈值

HealthCheck 持续轮询关键指标，按级别反馈告警：

| 指标 | 监护项 | 预告警 (WARN) | 确认告警 (CRITICAL) | 故障条件 (FAULT) |
|-----|--------|------------|------------------|-----------------|
| **TrackData 输出** | trackdata_gap_ms | 300-500ms | ≥500ms (T1) | ≥2000ms (T2) |
| **GPU 处理** | kernel_timeout_count | count 变化 | count > 0 | 连续超时 + 延迟↑ |
| **M01-M04 队列** | queue_depth | depth > 70% 容量 | depth > 90% 容量 | depth 溢出 + 背压 |
| **系统状态** | 转移完整性 | 冗余状态转移 | 异常转移链 | 3 次以上异常 |

### 3.4 Runtime 告警的 " 响应延迟 " 设计（避免瞬时抖动）

```txt
规则：采用"确认阈值"机制

状态流转：A(Running)
  ↓ [HealthCheck异步轮询，100ms周期]
  └─ 检测到 trackdata_gap_ms = 550ms
     → 记录为 WARN 级，计数+1（但不转状态）
  ↓ [继续轮询]
  └─ trackdata_gap_ms = 520ms  ← 恢复，WARN计数归零
  ↓ 或
  └─ trackdata_gap_ms = 680ms
     → 连续2次CRITICAL（间隔<100ms），确认故障
  ↓
  └─ Orchestrator 触发 Running → Degraded 转移

恢复条件（从Degraded回Running）：
  - 连续帧间隔 < 500ms
  - 且保持 ≥ 5s 无新告警
  - 由Orchestrator决策是否自动恢复（可配置）
  > 对齐 `00_顶层架构.md` §2.7 SSOT 定义
```

**好处**：避免瞬时抖动、有缓冲空间、条件明确

### 3.5 System Warmup（预热）条件

**不使用固定帧数，改用时间 + 连续性条件**：

```txt
Warmup的3个并行条件（全部满足才转向Standby）：

1. 连续输入条件
  └─ 连续 ≥ 1000ms 未中断收到CPI重组输出（允许IncompleteFrame）
   └─ 用 warmup_start_timestamp 和 last_cpi_timestamp 比对

2. 处理链路通畅条件
   └─ 至少产生 1 个 TrackFrame（frame_seq 自递增）
   └─ 证明 M03 → M04 通路畅通

3. 输出稳定性条件（可选，但推荐）
   └─ TrackFrame 输出率 ≥ 0.9Hz（连续2帧）
   └─ 证明M03融合逻辑已进稳态

实现要点：
    不依赖具体帧结构
    与采样率/帧率无关
    纯时间与计数的组合判断
```

## 4️⃣ DiagnosticsCenter：故障诊断（V1 轻量化）

### 4.1 系统定位与 V1 范围

**DiagnosticsCenter 是故障自动化诊断框架**，采用**最轻量化实现**：

- **输入**：来自 HealthCheck、Orchestrator、ConfigManager、TimeSyncManager 的事件与指标
- **处理**：应用简化规则库，推测最可能的根因
- **输出**：单个 JSON 诊断包（<50KB）或 ZIP（JSON + Markdown 说明）

**V1 范围**：JSON 数据导出 + 5 个基础规则库 + 推荐行动建议
**V1.5 可选**：包装成 HTML 可视化报告（30 min 额外投入）
**V2 迭代**：PNG 图表生成 + 规则库扩展

### 4.2 关键指标采样结构（应持续采样到环形缓冲）

### 4.4 故障诊断规则库（V1 最小集 - 5 个模式）

**规则库的设计**：通过特征匹配识别故障模式并推测根因

#### **规则 1：GPU 过载/超时**

```txt
特征：
  - M02.kernel_timeout_count > 0（绝对触发）
  - M02.gpu_utilization ≥ 90% 或 gpu_temp ≥ 85°C（加强条件）
  - M02.queue_depth 持续上升或溢出

根因信度：90%

诊断文本：
  "GPU处理能力不足或散热故障。GPU温度过高会触发自动降频，导致处理延迟。
   请检查：(1)GPU散热风扇是否工作；(2)是否有其他进程抢占GPU；(3)算法复杂度是否超预期。"

推荐行动：
  - 检查GPU风扇运行状态与温度传感器读数
  - 用 nvidia-smi 查看GPU占有进程，排除干扰
  - 临时降低CFAR阈值或融合分辨率，观察性能恢复
```

#### **规则 2：时间源丢失**

```txt
特征：
  - TimeSyncManager.time_state 从SYNCED→HOLDOVER或UNSYNC（绝对触发）
  - TrackData延迟或中断紧跟其后（<200ms内）
  - offset_estimate或drift_estimate出现跳变

根因信度：95%

诊断文本：
  "权威时间源丢失，导致系统进入守时或不可信状态。
   信号处理依赖权威时间，一旦丢失，航迹融合无意义。
   请检查：(1)GPS/PPS接线连接；(2)授时源设备是否掉电；(3)光纤/网络授时是否通路中断。"

推荐行动：
  - 检查GPS天线是否有信号（LED指示灯）
  - 检查PPS线是否完整或断裂
  - 确认授时源设备工作状态（ping地址、查看web界面）
  - 必要时重启授时源设备
```

#### **规则 3：网络丢包/拥塞**

```txt
特征：
  - M01.cpi_reorg_fail_count 突增或非零（绝对触发）
  - 或 M04.trackframe_udp_fail_count > 0
  - 但 M02/M03处理正常（无GPU超时、Track融合率正常）

根因信度：80%

诊断文本：
  "网络链路发生丢包或拥塞。UDP接收侧的CPI重组失败表明数据包到达不完整。
  这可能是物理链路故障、交换机端口速率协商失败或突发流量冲击。
  请检查：(1)网线连接；(2)交换机端口速率（应10GbE）；(3)是否有其他网络流量冲击；(4)显控UDP是否超1200B导致IP分片。"

推荐行动：
  - 用 ethtool 检查网卡速率是否降速（eth0应为10GbE）
  - 用 iperf 或 ping 进行网络诊断，看包丢失率
  - 替换网线重试
  - 检查交换机端口配置与对端设备速率协商
  - 检查 M04 TrackData 单包长度是否严格 <= 1200B 并启用超限截断标记
```

#### **规则 4：参数修改导致异常**

```txt
特征：
  - ConfigManager.config_applied_version 刚更新（距故障<60s）
  - 故障表现为：Track数异常、融合失败率↑、输出异常
  - 其他责任链路（时间、网络、GPU）正常

根因信度：70%

诊断文本：
  "近期参数更新与故障事件在时间上强相关。
   参数范围错误或参数组合不兼容可能触发算法异常。
   请比对故障前后的参数差异。"

推荐行动：
  - 查询config版本历史，找到故障前最后一个稳定版本
  - 对比新旧版本的参数差异（重点检查范围类/权重类参数）
  - 回滚到稳定版本并重启系统
  - 在实验室环境中逐参数调试，确认合理范围
```

#### **规则 5：启动自检失败**

```txt
特征：
  - HealthCheck.boot_check_failed = true
  - 故障发生在Init阶段（从未进Standby）
  - failure_code 标记具体失败项

根因信度：100%（启动检查失败即是根因本身）

诊断文本：
  "系统初始化自检失败，具体原因见诊断包中boot_report字段。
   常见原因包括：线程绑核不匹配、网络不通、GPU不可用、时间源未就绪。"

推荐行动：
  - 依据诊断包的boot_report逐项排查
  - 检查当前物理配置是否与设计拓扑匹配（CPU绑核、网卡位置）
```

### 5.2 核心交互定义

#### **交互 1：参数变更的安全应用**

```txt
步骤：
1. 显控或配置源发起参数变更 → ConfigManager
2. ConfigManager验证范围、分类等级
3. 依据等级分发：
   - Hot: 写版本化快照，M01-M04帧边界自动读取新版本
   - Flush: Orchestrator 下发 APPLY_CONFIG，模块在安全点冲刷
   - Restart: 标记PENDING，等进程重启
4. ConfigManager反馈成功/失败状态给显控
5. DiagnosticsCenter 记录参数变更事件（便于故障追溯）
```

#### **交互 2：时间源故障处理**

```txt
步骤：
1. TimeSyncManager 检测到 time_state 转移（如SYNCED→HOLDOVER）
2. 立刻通知 Orchestrator
3. Orchestrator 根据转移类型决策：
   - SYNCED→HOLDOVER: 进Degraded + 继续输出TrackData(打TIME_HOLDOVER标志)
   - SYNCED→UNSYNC: 禁止Running
   - HOLDOVER→HOLDOVER_EXPIRED: 进Fault
4. HealthCheck 监护时间恢复（offset + drift指标），若恢复发起Degraded→Running
5. DiagnosticsCenter 记录时间转移时间线（用于故障诊断）
```

#### **交互 3：健康监护与故障诊断**

```txt
步骤：
1. HealthCheck 持续轮询关键指标，生成告警事件（ts, severity, module, message）
2. 告警事件推送给 Orchestrator（决策状态转移）和 DiagnosticsCenter（诊断材料）
3. DiagnosticsCenter 聚合：
   - HealthCheck 的告警时间线
   - Orchestrator 的状态转移时间线
   - ConfigManager 的参数变更时间线
   - TimeSyncManager 的时间状态时间线
   - 过去5分钟的关键指标采样
4. 应用规则库，推测根因
5. 生成诊断包（JSON格式），供显控/工具消费
```

## 6️⃣ 协议适配层设计（FPGA V3.1 与显控 V1 一致性要求）

### 6.1 设计动机与风险背景

根据《后端架构、射频前端 FPGA 与显控一致性耦合分析报告》，当前后端设计与已定版 FPGA 协议 V3.1 存在两处**最高优先级不一致**，若不修正将导致联调失败：

| 不一致类型 | 原设计 | 协议 V3.1 要求 | 风险等级 |
|-----------|--------|---------------|---------|
| **数据重组超时** | T_reassembly=10ms 丢弃 | T_reasm=100ms 补零输出 + 标记 IncompleteFrame | P0/阻断 |
| **显控 UDP 容量** | 未限制单包大小 | MaxUdpPayloadBytes=1200B，超限截断并标记 | P0/阻断 |

为此，必须在后端数据面（M01 Receiver、M04 Gateway）与控制面（Orchestrator）引入**协议适配层**，作为 FPGA V3.1 和显控 V1 协议的唯一事实源实现。

### 6.2 M01 Receiver 协议适配层（数据平面入口）

#### 6.2.1 重组引擎冻结规范（V3.1 强制约束）

```cpp
// M01 Receiver 重组引擎参数（冻结，不可配置）
const uint32_t T_REASM_MS = 100;              // V3.1 冻结：100ms 超时
const uint32_t MAX_REASM_BYTES_PER_KEY = 16*1024*1024;  // 单帧重组上限
const uint32_t MAX_TOTALFRAGS = 1024;          // 最大分片数

// 重组键定义（V3.1 第5.2节）
struct ReassemblyKey {
    uint16_t source_id;         // 来源标识
    uint32_t control_epoch;     // 控制纪元（纪元变化需清空重组缓存）
    uint32_t frame_counter;     // 搜索帧计数
    uint16_t beam_id;           // 波束ID
    uint16_t cpi_index;         // CPI计数

    bool operator==(const ReassemblyKey& other) const;
};

// 重组结果标记
struct ReassembledFrame {
    std::vector<uint8_t> payload;
    bool is_complete;              // false 表示存在缺片
    std::vector<uint16_t> missing_frags;  // 缺片索引列表（用于诊断）
    ExecutionSnapshot exec_snapshot;      // 尾包携带的执行快照（V3.1 §5.3.4）
    uint64_t reassembly_time_us;          // 重组完成时间戳
};
```

#### 6.2.2 缺片补零策略（关键差异点）

**冻结规则**（V3.1 §5.3.3）：
- **超时到期（100ms）时必须输出帧**，即使缺片也不丢弃
- **缺失的分片按分片大小补零**（例如 FragIndex=3 缺失，则在对应位置写入 0x00）
- **标记 `is_complete=false` 和 `missing_frags` 列表**，供 M02 信号处理识别

**实现要点**：
```cpp
void ReassemblyEngine::on_timeout(const ReassemblyKey& key) {
    auto& ctx = reasm_contexts_[key];

    // 检查完整性
    if (ctx.received_frags.size() < ctx.total_frags) {
        ctx.is_complete = false;

        // 补零缺失的分片
        for (uint16_t i = 0; i < ctx.total_frags; ++i) {
            if (!ctx.received_frags.contains(i)) {
                ctx.missing_frags.push_back(i);
                // 补零（假设每片固定大小，尾片除外）
                size_t frag_size = (i == ctx.total_frags - 1)
                    ? ctx.tail_frag_payload_bytes
                    : DEFAULT_FRAG_SIZE;
                ctx.payload.resize(ctx.payload.size() + frag_size, 0x00);
            }
        }
    }

    // 输出到 M02（即使不完整）
    emit_frame(ctx.as_reassembled_frame());

    // 更新诊断计数
    stats_.incomplete_frame_count++;
}
```

#### 6.2.3 Execution Snapshot 提取（控制闭环验证）

**冻结规则**（V3.1 §5.3.4）：
- 尾包（FragIndex == TotalFrags-1）携带 Execution Snapshot（实际执行参数快照）
- 后端必须提取并转发给 Orchestrator，用于验证"控制下发值 vs 前端实际生效值"

**闭环验证流程**：
```txt
步骤1：Orchestrator 下发 Control(SequenceNumber=123, BeamAzimuth=45.0°)
步骤2：M01 接收 Data 尾包，提取 Execution Snapshot { BeamAzimuth=45.0° }
步骤3：Orchestrator 比对：
       - 一致 → 无告警
       - 不一致 → 生成 CONTROL_MISMATCH 告警，推送给 DiagnosticsCenter
```

#### 6.2.4 心跳优先队列（遥测平面保活）

**冻结规则**（V3.1 §6.2）：
- Data(0x03) 与 Heartbeat(0x04) 复用 UDP/9999
- 接收端必须按 `PacketType` 分类进入**优先队列**，优先处理心跳

**实现策略**：
```cpp
// 接收线程入口
void ReceiverThread::on_packet_received(uint8_t* buf, size_t len) {
    auto pkt_type = extract_packet_type(buf);

    if (pkt_type == 0x04) {
        // 心跳立即进入高优先级队列
        heartbeat_queue_.enqueue(buf, len);
    } else if (pkt_type == 0x03) {
        // 数据进入普通队列
        data_queue_.enqueue(buf, len);
    }

    // 建议使用两个独立线程处理，避免数据洪峰饿死心跳
}
```

**DSCP 配置建议**（可选）：
- 要求 DACS 侧对 Heartbeat 设置 DSCP=EF（Expedited Forwarding）
- 交换机支持时启用 QoS 优先队列

### 6.3 M04 Gateway 协议适配层（显控数据面输出）

#### 6.3.1 TrackData 单包限长策略（V1 强制约束）

**冻结规则**（显控协议 V1）：
- `MaxUdpPayloadBytes = 1200B`（含 Protobuf 序列化后的完整数据）
- **禁止分片**：单个 datagram 必须承载完整 TrackData
- **超限必须截断**并显式标记 `is_truncated=true`、`drop_reason`、`dropped_count`

**截断策略示例**：
```cpp
// 按威胁/置信度/距离排序后选择性输出
struct TrackDataBuilder {
    void add_tracks(const std::vector<Track>& tracks) {
        // 按优先级排序（威胁高、距离近优先）
        auto sorted = sort_by_priority(tracks);

        size_t budget = 1200;  // UDP payload 上限
        size_t used = base_size();  // 基础字段占用

        for (const auto& tk : sorted) {
            size_t tk_size = estimate_track_size(tk);
            if (used + tk_size > budget) {
                // 超限，标记截断
                proto_.set_is_truncated(true);
                proto_.set_drop_reason("PAYLOAD_SIZE_LIMIT");
                proto_.set_dropped_count(sorted.size() - proto_.tracks_size());
                break;
            }
            proto_.add_tracks()->CopyFrom(tk);
            used += tk_size;
        }
    }
};
```

#### 6.3.2 全量快照语义（丢包不漂移）

**冻结规则**（显控协议 V1）：
- TrackData 为**全量快照**（非增量），丢包不会导致状态漂移
- 每帧必须携带 `frame_seq`（单调递增）、`backend_instance_id`、`clock_domain`、`data_timestamp(ns)`

**帧序号管理**：
```cpp
class GatewayOutputManager {
  private:
    uint64_t frame_seq_ = 0;          // 单调递增，进程生命周期内不重置
    std::string backend_instance_id_; // 后端实例ID，重启后变化

  public:
    void send_trackdata(const TrackSnapshot& snapshot) {
        TrackDataProto proto;
        proto.set_frame_seq(frame_seq_++);
        proto.set_backend_instance_id(backend_instance_id_);
        proto.set_clock_domain(map_time_domain());
        proto.set_data_timestamp_ns(snapshot.timestamp_ns);

        // ... 填充 tracks

        udp_send(proto.SerializeAsString());
    }
};
```

### 6.4 Orchestrator 控制平面适配层（幂等桥接）

#### 6.4.1 三方幂等键桥接表

**冻结约束**：
- 显控侧幂等键：`command_id`（TTL ≥ 10min）
- DACS 侧幂等键：`(SourceID, ControlEpoch, SequenceNumber)`
- 后端必须维护**桥接映射表**，聚合多阵面 Ack 结果

```cpp
// 幂等桥接条目
struct ControlBridgeEntry {
    std::string hmi_command_id;         // 显控 command_id
    std::vector<DacsControlTx> dacs_txs; // 对应的 DACS 控制事务

    struct AckAggregation {
        int total = 0;
        int success = 0;
        int not_applied = 0;
        int applied_late = 0;
        std::vector<std::string> errors;
    } ack_status;

    uint64_t create_time_ms;
    bool is_final = false;              // 是否已最终判定
};

// Orchestrator 维护映射表
class IdempotenceBridge {
  private:
    std::map<std::string, ControlBridgeEntry> bridge_map_;
    const uint64_t TTL_MS = 10 * 60 * 1000;  // 10分钟

  public:
    // 显控发起控制
    void on_hmi_control_request(const std::string& cmd_id, const ControlParams& params) {
        ControlBridgeEntry entry;
        entry.hmi_command_id = cmd_id;

        // 生成对应的 DACS 控制事务（可能是多个阵面）
        for (auto& dacs : dacs_instances_) {
            DacsControlTx tx;
            tx.source_id = BACKEND_SOURCE_ID;
            tx.control_epoch = current_control_epoch_;
            tx.sequence_number = next_seq_++;
            tx.payload = encode_params(params);

            entry.dacs_txs.push_back(tx);
            entry.ack_status.total++;
        }

        bridge_map_[cmd_id] = entry;

        // 发送到各 DACS
        for (auto& tx : entry.dacs_txs) {
            send_dacs_control(tx);
        }
    }

    // DACS 返回 Ack
    void on_dacs_ack(uint32_t seq_num, AckResult result) {
        // 找到对应的 command_id
        for (auto& [cmd_id, entry] : bridge_map_) {
            for (auto& tx : entry.dacs_txs) {
                if (tx.sequence_number == seq_num) {
                    // 聚合结果
                    if (result == AckResult::SUCCESS) {
                        entry.ack_status.success++;
                    } else if (result == AckResult::NOT_APPLIED) {
                        entry.ack_status.not_applied++;
                    } else if (result == AckResult::APPLIED_LATE) {
                        entry.ack_status.applied_late++;
                    }

                    // 判定最终结果
                    if (entry.ack_status.success + entry.ack_status.not_applied
                        + entry.ack_status.applied_late >= entry.ack_status.total) {
                        entry.is_final = true;
                        send_hmi_response(cmd_id, aggregate_result(entry));
                    }
                    return;
                }
            }
        }
    }
};
```

#### 6.4.2 控制重传一致性保障

**冻结规则**（V3.1 §4.3）：
- RTO = 2500ms、MAX_RETRY = 3、指数退避
- **重传包必须保持 `SequenceNumber` 和 `Payload` 完全一致**

```cpp
class ControlRetransmitter {
  private:
    struct PendingControl {
        DacsControlTx tx;
        uint64_t send_time_ms;
        int retry_count = 0;
        std::vector<uint8_t> frozen_payload;  // 冻结的 payload
    };

  public:
    void on_timeout(uint32_t seq_num) {
        auto& pend = pending_map_[seq_num];

        if (pend.retry_count >= MAX_RETRY) {
            // 失败归因
            notify_failure(seq_num, "MAX_RETRY_EXCEEDED");
            pending_map_.erase(seq_num);
            return;
        }

        // 重传：SequenceNumber 和 Payload 完全一致
        pend.retry_count++;
        pend.send_time_ms = get_time_ms();
        send_dacs_control_raw(pend.frozen_payload);  // 使用冻结的原始包
    }
};
```

#### 6.4.3 时间质量门控（定时生效前置条件）

**冻结规则**（V3.1 §7.2.3）：
- 仅当 `dacs_time_quality == Q0` 时允许"秒边界定时生效"
- Q1/Q2 必须拒绝或改为"立即生效"，并在回执中标记原因

```cpp
class OrchestratorTimedApply {
  public:
    bool can_apply_on_second_boundary() {
        auto time_quality = time_sync_mgr_->get_dacs_time_quality();

        if (time_quality != TimeQuality::Q0_LOCKED) {
            // Q1/Q2 禁止定时生效
            log_warning("DACS time_quality != Q0, timed-apply disabled");
            return false;
        }

        return true;
    }

    void schedule_timed_control(const ControlParams& params, uint64_t apply_time_s) {
        if (!can_apply_on_second_boundary()) {
            // 降级为立即生效，并通知显控
            send_hmi_notification("TIME_QUALITY_DEGRADED",
                "Timed-apply not available, fallback to immediate apply");
            apply_control_immediate(params);
            return;
        }

        // 正常定时生效流程
        enqueue_timed_control(params, apply_time_s);
    }
};
```
