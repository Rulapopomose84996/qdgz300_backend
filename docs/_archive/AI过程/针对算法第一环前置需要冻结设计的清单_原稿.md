现有材料已经能确认三件事：后端总链路大致是“数据接收/重组 → 脉压 → 多普勒 → CFAR → 融合”，协议侧冻结了分片重组与补零规则，工程侧也把接收与信号处理拆成了 M01/M02 两个模块；但这些材料还不足以唯一确定“第一环”的研究对象、输入定义和验收口径。

### 我已经确认的当前不一致 / 未闭合点

1. **“第一环”口径不统一**
   研究报告把链路写成“Data 重组/解交织 → 脉冲压缩 → 多普勒 → CFAR → 融合”，而项目骨架把“接收/重组”放在 M01，把 FFT/CFAR 等信号处理放在 M02。也就是说，系统口径下第一环可能是“重组+解交织”，算法口径下第一环可能是“距离向匹配滤波/脉压”。这件事目前没有冻结。

2. **PRI / PRF 与 CPI / N 的语义存在冲突**
   研究报告采用了 `CPI=50 ms, N=20`，若按 CPI 内均匀发脉冲理解，则 `PRI≈2.5 ms`；但协议里的 `PulsePeriod` 量化单位是 `1 μs`，单字节范围 `0~255 μs`，两者明显对不上。报告本身也把这一点标成了风险项。这个不关掉，多普勒刻度、速度模糊区、相干性分析都会漂。

3. **采样率是需求/报告假设，不是协议显式字段**
   研究报告按 `Fs=25 MHz` 建模，但协议当前显式可控的是带宽、脉宽、周期、积累点数、样点数，采样率并没有被明确作为协议字段冻结。第一环如果是脉压，这会直接影响参考信号生成、距离刻度和滤波器长度。

4. **`ChannelMask` / `DataType` 的工程依赖关系没有彻底闭合**
   协议重组键、RAW 长度计算、补零长度计算都依赖 `ChannelMask` 和复数数据类型；报告则把 `ChannelMask=0x000F`、`DataType=Int16` 视为系统集成假设。也就是说，算法研究当前用的是“假定输入格式”，不是“完全由协议自描述的输入格式”。

5. **极化口径未统一**
   报告按“双极化能力、当前先做 V 极化”来写，但协议里没有看到清晰的极化表达字段；报告也把这一点列为未决项。若第一环研究要定义输入张量或通道布局，这一点必须明确。

6. **`SampleCountFixed` 与 `Short/LongCodeSampleCount` 的关系未说明**
   协议一方面要求非尾包 `SampleCount == SampleCountFixed` 才能支持“按分片补零”，另一方面 Control 又给了 `ShortCodeSampleCount` / `LongCodeSampleCount`。但当前材料没有明确：`SampleCountFixed` 是不是由长/短码样点数推导、何时切换、长短组合模式怎么处理。这个不讲清，第一环输入长度都不稳定。

7. **Snapshot 依赖与丢尾包场景之间有闭环缺口**
   协议规定 `Execution Snapshot` 只在尾包携带；同时又允许 `T_reasm=100 ms` 超时后补零输出不完整帧，并丢弃迟到分片。研究报告希望利用 Snapshot 做闭环一致性验证，但如果尾包丢了，算法侧可能拿到“可输出的帧”却拿不到 Snapshot。这个场景当前没有明确处理策略。

8. **波形范围没有冻结**
   协议支持 LFM / NLFM / 二相码等多种波形；研究报告当前默认按 LFM 脉压建模。若第一环是匹配滤波/脉压，就必须先确认“本期只研究 LFM”，否则参考信号生成和性能指标都不是一个问题。

9. **时间同步口径没有对齐到研究所需级别**
   协议对 DACS 的秒边界定时生效、GNSS/PTP/XO 降级恢复定义得很细；服务器环境基线却说明当前服务器时钟处于 `Orphan Mode (stratum 10)`，需要注意与标准时间的长期对齐。它们不一定直接冲突，但对于多阵面脉冲级时间对齐、离线数据回放、时序验收来说，当前口径不够闭合。

10. **项目骨架不是实现现状，不能直接当成代码依据**
    骨架文档已经明确说它是“规划视图，不是代码现状清单”。所以如果你后面希望我研究“第一环的落地实现”，还需要真实源码树、CMake、生效分支和当前 stub/空实现清单。
