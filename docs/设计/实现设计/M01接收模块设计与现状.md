# M01 接收模块设计与现状

> 依据范围：`src/m01_receiver/`、`include/qdgz300/m01_receiver/`、相关配置文件。  
> 原则：本文描述当前代码已经体现出的模块边界和运行主线；若与历史设计稿冲突，以代码为准。

## 1. 模块职责

M01 的职责是：

1. 从单阵面或三阵面 UDP 入口收包
2. 解析并校验协议头
3. 通过 `RxStage` 解耦热路径和后续处理
4. 分发 `DATA / HEARTBEAT`
5. 对数据执行分片重组与重排
6. 将有序输出投递给下游接口

M01 不承担：

- M02 GPU 信号处理
- M03 航迹处理
- M04 网关协议编码
- 系统级 Orchestrator 状态机裁决

## 2. 当前主路径

```text
UdpReceiver / ArrayFaceReceiver
  -> RxStage::on_packet()
  -> SPSCQueue<RxEnvelope>
  -> processing thread
  -> Dispatcher
  -> Reassembler
  -> Reorderer
  -> DeliveryInterface
```

收包热路径只负责解析、校验、封装 `RxEnvelope` 和入队；后续处理由每阵面独立 processing thread 完成。

## 3. 关键实现点

### 3.1 网络层

- 支持单阵面和三阵面配置
- 可按阵面绑定 IP、CPU 和来源过滤
- 支持捕获 hook 与抓包落盘

### 3.2 `RxStage`

- 是热路径的分层边界
- 采用 `drop_oldest_push`，不阻塞等待
- 负责把原始报文转成轻量的 `RxEnvelope`

### 3.3 `Dispatcher`

- 当前主路径是同步直通分发
- `DATA` 与 `HEARTBEAT` 走不同处理分支
- 零拷贝路径在 processing thread 中实际参与运行

### 3.4 `Reassembler`

- 使用完整协议键做重组
- 支持零拷贝缓存和超时输出不完整帧
- 支持超时后冻结键，丢弃迟到分片

### 3.5 `Reorderer`

- 负责按序列号恢复输出顺序
- 输出 `OrderedPacket`
- 允许在超时场景下做有界冲刷

## 4. 当前接口问题

- M01 与 M02 之间的 `RawBlock` 结构与所有权模型尚未完全统一。
- 真实硬件规模下的 payload 上限仍待联调确认。
- 部分设计稿使用的术语和现状代码已不一致，应以后者为准。
