# QDGZ300 服务器部署和压测指南

> 适用环境：麒麟 V10 ARM64 服务器  
> 当前有效方案：共享离线包目录 + 共享依赖缓存目录 + 正式构建脚本入口

## 1. 目录约定

项目源码目录：

```text
/home/devuser/WorkSpace/qdgz300_backend
```

共享离线包目录：

```text
/home/devuser/WorkSpace/ThirdPartyCache/qdgz300_backend/archives
```

服务器原生共享依赖缓存目录：

```text
/home/devuser/WorkSpace/ThirdPartyCache/qdgz300_backend/build/native-aarch64
```

仓库内 `deps_offline/` 不再是默认流程的一部分。

## 2. 首次部署

### 2.1 拉取代码

```bash
cd /home/devuser/WorkSpace/qdgz300_backend
git pull --ff-only origin refactor/kylin-v10-arm64-cuda-clang18
```

### 2.2 构建

```bash
cd /home/devuser/WorkSpace/qdgz300_backend
bash scripts/build/build_production.sh
```

该脚本会自动：

1. 校验 `cmake`、`ninja`、CoreX clang 环境
2. 调用 `scripts/prepare_native_deps.sh` 准备共享依赖缓存
3. 自动清理路径失配的旧 `build_production` 缓存
4. 配置并构建主工程
5. 通过 `scripts/test/test_unit.sh` 执行单元测试

### 2.3 安装

```bash
cd /home/devuser/WorkSpace/qdgz300_backend
sudo bash deploy/install.sh build_production
sudo bash scripts/ops/enable_services.sh
sudo bash scripts/ops/start_services.sh
```

### 2.4 验收

```bash
cd /home/devuser/WorkSpace/qdgz300_backend
sudo bash scripts/ops/check_services.sh
sudo journalctl -u qdgz300-receiver.service -n 50 --no-pager
```

PCAP 落盘链路说明：

- `receiver_app` 只负责把原始报文写到 NVMe spool：`/opt/qdgz300_backend/data/receiver_spool`
- `qdgz300-spool-mover.service` 持续把封口 `.pcap` 搬到 HDD archive：`/data/qdgz300/receiver/archive`
- `.pcap.part` 表示当前仍在写入，mover 不会触碰

## 7. 闭环结论

截至当前版本，以下链路已验证通过：

1. 服务器原生共享依赖缓存准备
2. 服务器原生构建
3. 单元测试脚本入口
4. 安装脚本入口
5. systemd 服务启停链路
6. WSL 交叉共享缓存与构建链路

因此后续服务器部署、回归、压测都应以本文档中的正式脚本入口为准，不再回退到仓库内离线目录流程。
