# 数据流、控制流与状态契约

## 1. 数据流主线

当前设计主线是：

```text
M01 Receiver
  -> M02 SignalProc
  -> M03 DataProc
  -> M04 Gateway
  -> HMI / 上级系统
```

其中 M01 负责收包、校验、重组和重排，后续模块分别负责信号处理、航迹处理与对外输出。

## 2. 控制流主线

控制流由 Orchestrator 统一裁决：

- 接收外部控制命令
- 根据系统状态和事件证据决定允许、拒绝、延迟或降级
- 统一发布状态转移、告警与诊断触发

## 3. 系统状态集合

对外 `system_state` 冻结为五态：

- `Init`
- `Standby`
- `Running`
- `Degraded`
- `Fault`

`Shutdown` 属于生命周期事件，不属于对外状态值域。

## 4. TrackData 契约

- 对外采用帧级一致的全量快照语义。
- 同一帧内所有 Track 共享相同的 `frame_seq`、`data_ts` 和系统质量口径。
- 允许降频，但降频后的每一帧仍应是可直接整帧替换的快照。
- V1 不支持 partial snapshot 作为正式输出语义。

## 5. 时间与质量口径

- 系统使用 `data_ts / ingest_ts / output_ts` 三时间戳模型。
- 对外时间质量以 `LOCK / HOLDOVER / FREERUN` 语义表达。
- `TrackData` 输出间隔是运行态健康度和故障转移的重要输入。

## 6. 关键阈值

- `T1_trackdata_gap = 500ms`：进入 `Degraded` 的默认阈值
- `T2_trackdata_gap = 2s`：进入 `Fault` 的红线阈值
- `T_reasm = 100ms`：协议重组超时基线

## 7. 命令与参数生效边界

- 热路径相关参数必须在安全边界生效，不允许“半帧切换”或“半 CPI 切换”。
- 运行期命令由 Orchestrator 串行化和审计。
- 同一业务命令必须支持幂等处理和明确的 ACK / NACK 语义。
