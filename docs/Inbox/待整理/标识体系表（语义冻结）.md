# 标识体系表（语义冻结）

## 一、系统角色与总体对象

| 中文名          | 语义说明                                              | 备注                               |
| ------------ | ------------------------------------------------- | -------------------------------- |
| 阵面           | 雷达的单个固定阵面，是物理执行与采样来源的基本单元。当前系统共 3 个阵面。            | 可理解为 Face；后续若写英文，建议统一映射为 `Face`。 |
| 前端           | 靠近阵面侧的硬件执行部分，至少包含 ADC、FPGA，以及被 FPGA 控制的收发与时序执行链路。 | 后端不感知其更细内部实现。                    |
| FPGA         | 前端侧核心可编程逻辑，负责控制表接收/缓存/生效、原始数据组包并上送服务器。            | 是“更接近真实执行边界”的一侧。                 |
| 服务器          | 后端计算平台，同时承担数据处理节点与控制调度节点两个角色。                     | 不是单纯转发器，而是系统中枢。                  |
| 显控           | 人机交互与显示控制端，负责任务意图输入、状态展示、人工干预。                    | 文档中也出现 HMI 说法；建议中文正文统一写“显控”。     |
| 全局调度视图 / 总计划 | 服务器内部面对整部雷达形成的一份统一调度视图。                           | 对外不会直接以这一对象下发，而是拆成三份阵面执行表。       |
| 阵面执行表        | 从全局调度视图拆分出来、面向单个阵面的具体执行表。                         | 逻辑上一套全局计划，物理上三份阵面执行表。            |
| 系统级统一时间语义    | 控制流与数据流共同遵守的一套周期、边界、生效口径。                         | 不是“服务器自己猜时间”，而是由软件秒边界事件驱动建立。     |

## 二、时间、边界与周期语义

| 中文名      | 语义说明                                    | 备注                            |
| -------- | --------------------------------------- | ----------------------------- |
| 根时基来源    | 系统首选的最上层授时来源。当前冻结为北斗 GNSS。              | 这是“授时根”，不是运行时直接触发源。           |
| 降级时间来源   | 根时基不可用时的降级来源。当前冻结为 PTP / 本地晶振 holdover。 | 是守时与降级口径，不等于主触发逻辑。            |
| 运行时直接触发源 | 服务器控制动作在运行时实际消费的边界输入。                   | 当前冻结为 FPGA 上送的软件秒边界事件。        |
| 秒边界      | 系统级“大边界”，用于换表、换周期、统一对齐。                 | 不建议再用“服务器本地秒”去替代它。            |
| 软件秒边界事件  | 前端感知到物理边界后，上送后端并在控制面被承认的逻辑秒边界事件。        | 这是当前文档里的核心冻结对象；它不是硬件 1PPS 本体。 |
| 物理边界     | 前端执行域内真实感知到的边界，例如 FPGA 侧 1PPS / 秒边界。    | 属于前端执行域，不属于服务器本地软件时钟域。        |
| 调度大周期    | 控制面的大周期单位。当前默认定义为 1 秒。                  | 可与 `Schedule Epoch` 对应。       |
| 全局周期     | 系统在统一时间语义下推进的逻辑周期。                      | 通常与 1 秒调度大周期对应。               |
| 生效边界     | 某张执行表从待生效变为实际生效的边界。                     | 当前冻结口径是：在软件秒边界统一生效。           |
| 表内推进边界   | 执行表在秒内继续往前推进时所依据的边界。                    | 当前冻结为 CPI 边界。                 |
| CPI 边界   | 秒内小边界，负责表内推进和数据处理对齐。                    | 秒边界负责换表；CPI 边界负责按表推进。         |
| 时间戳      | 用于记录、追踪、回放、对时的时间信息。                     | 明确不直接触发生效。                    |
| 权威全局周期状态 | 由时基管理组件维护的统一周期状态集合。                     | 其他模块不应各自再维护另一套“自己的周期状态”。      |
| 当前秒号     | 当前逻辑秒的编号或时间语义。                          | 属于权威全局周期状态的一部分。               |
| PPS 序号   | 软件秒边界事件的顺序编号。                           | 用于去重、乱序、失边界检测。                |
| 授时状态     | 当前处于哪种授时/守时状态，例如锁定、holdover、freerun。    | 用于调度可信度判断和状态展示。               |
| 当前周期可信性  | 当前逻辑周期是否可被信任、是否存在跳变/降级。                 | 建议作为权威周期状态的一部分保留。             |

## 三、接口与链路对象

| 中文名             | 语义说明                               | 备注                 |
| --------------- | ---------------------------------- | ------------------ |
| 上行原始数据接口        | 前端向服务器上送原始采样数据的正式接口。               | 5 类正式接口之一。         |
| 下行调度执行表接口       | 服务器向前端下发周期性执行表的正式接口。               | 服务搜索/跟踪/精跟主链。      |
| 管理/应急命令接口       | 服务器向前端下发非周期性管理动作的正式接口。             | 用于启停、复位、安全态、测试模式等。 |
| ACK / 遥测 / 状态接口 | 前端向服务器回传接收确认、生效确认、心跳、遥测和状态的正式接口。   | 5 类正式接口之一。         |
| 时基边界事件接口        | 前端向服务器控制面上送边界事实的正式接口。              | 对应软件秒边界事件。         |
| 上行数据流           | 从前端原始采样到服务器处理再到结果发布的主数据链。          | 属于数据面主链。           |
| 下行控制流           | 从显控任务意图到控制策略、调度定版、下发表、ACK 回收的主控制链。 | 属于控制面主链。           |
| 全双工共链路          | 上行原始数据与下行控制当前默认共用同一 10G 全双工物理链路。   | 当前不额外假设独立控制链路。     |
| 原始包镜像流          | 接收模块旁路输出的原始包副本流。                   | 用于取证、回放、协议排障。      |
| 统一控制链路视图        | 控制网关向调度中心和显控输出的控制链路综合状态视图。         | 包括发送、ACK、遥测、超时等。   |

## 四、控制面对象与控制语义

| 中文名     | 语义说明                                  | 备注                          |
| ------- | ------------------------------------- | --------------------------- |
| 调度执行表   | 服务器周期性生成并提前下发给前端的执行对象。                | 是主控制对象，不是普通“命令”。            |
| 管理/应急命令 | 面向启停、复位、安全态等管理动作的非周期性控制对象。            | 与调度执行表正式分离。                 |
| 当前生效表   | 前端当前正在执行的那张表。                         | 对外语义建议统一写 `current`。        |
| 待生效表    | 已装载但尚未在边界生效的下一张表。                     | 对外语义建议统一写 `next`。           |
| 服务器侧预备表 | 服务器本地可保留的一张额外预备表语义。                   | 对应 `queued`；不要求 FPGA 真三缓冲。  |
| 候选安排    | 控制策略模块基于数据结果和任务意图生成的下一周期候选控制安排。       | 不是最终定版。                     |
| 定版执行表   | 经任务调度中心批准并编号后的正式执行表。                  | 才允许进入下发流程。                  |
| 生效秒序号   | 某张表计划在哪个逻辑秒周期生效。                      | 常与目标生效周期联动。                 |
| 表版本号    | 执行表的版本标识。                             | 三阵面共享全局表版本口径，但每面可有各自实例。     |
| 旧表续跑    | 新表未及时到达时，允许当前表在限定窗口内继续运行。             | 属于可用性优先的工程策略。               |
| 局部降级运行  | 局部阵面异常时，系统不立即整机停摆，而进入受控降级。            | 长期异常再收紧到安全态。                |
| 安全态     | 在严重异常下进入的受控安全运行状态。                    | 一般优先保证停发射或受控发射，同时保留接收与诊断能力。 |
| 模式状态机   | 控制面对待机、搜索、跟踪、测试、降级、安全态等模式进行管理的状态机。    | 由任务调度中心负责。                  |
| 边界裁决    | 根据时基状态、表状态、ACK 状态判断是否进入新周期、是否允许生效的决策。 | 不能下放给 PPS 接收线程。             |

## 五、数据面对象与闭环对象

| 中文名         | 语义说明                                          | 备注                   |
| ----------- | --------------------------------------------- | -------------------- |
| 完整 CPI 数据单元 | 数据面后续处理的标准输入单位。                               | 接收模块输出给信号处理模块的主对象。   |
| 参数快照        | 由接收模块前移挂载、贯穿后续链路的权威上下文锚点。                     | 是当前设计里非常关键的冻结对象。     |
| 控制建议输入对象    | 数据处理模块输出给控制面的候选控制输入。                          | 它不是执行表本身。            |
| 闭环链路        | 从完整 CPI 数据单元，经控制建议输入对象、候选安排，最终形成下一周期定版执行表的链路。 | 当前冻结为搜索/跟踪/精跟统一闭环模型。 |
| 处理质量标记      | 信号处理或数据处理阶段对结果质量附加的标记。                        | 不应替代参数快照。            |
| 接收健康状态      | 接收模块输出的健康统计与异常统计。                             | 用于监控、诊断和控制面辅助判断。     |
| 数据面主链       | 接收、重组、信号处理、数据处理到结果发布的实时链路。                    | 不应侵入复杂控制逻辑。          |
| 数据旁路链       | 从接收模块尽早分流出去的录制与取证链路。                          | 不得反压主链。              |

## 六、模块与职责名词

| 中文名               | 语义说明                                             | 备注                       |
| ----------------- | ------------------------------------------------ | ------------------------ |
| 接收模块              | 上行数据进入服务器后的第一个业务模块，负责收包、基础校验、业务重组、形成完整 CPI 数据单元。 | 位于数据面。                   |
| 信号处理模块            | 对完整 CPI 数据单元进行算法级处理的模块。                          | 消费参数快照，不重新发明同义副本。        |
| 数据处理模块            | 对信号处理结果进行目标级逻辑处理、融合和结果整理的模块。                     | 输出控制建议输入对象。              |
| 上行数据流网关           | 将数据面结果转成面向外部消费者的发布格式并输出。                         | 是对外发布边界，不是算法模块。          |
| 控制策略模块            | 基于数据结果、任务意图和资源约束生成下一周期候选安排的模块。                   | 有生成权，没有最终定版权。            |
| 下行控制网关            | 负责显控控制请求标准化接入、执行表封装发送、ACK/遥测/状态回收。               | 不负责定版。                   |
| 任务调度中心            | 控制流主脑与最终定版权持有者。                                  | 负责状态机、时基管理、生效边界、定版、降级裁决。 |
| PPS 接收线程          | 接收前端上送的 PPS 通知包并形成软件秒边界事件的轻量接入线程。                | 职责是接入，不是裁决。              |
| 时基管理组件            | 维护权威全局周期状态的核心组件。                                 | 任务调度中心读取它，而不是直接“认包换秒”。   |
| 显控接口模块            | 控制面对显控暴露的接入边界。                                   | 负责任务意图输入与状态展示交互。         |
| ACK / 遥测 / 状态汇聚模块 | 汇总前端上报的控制状态对象。                                   | 可在控制网关或调度中心视角统一呈现。       |
| 原始数据录制链路          | 用于原始包镜像录制、回放、排障和离线分析的辅助链路。                       | 主录制对象是原始包镜像流。            |
| 基础设施建设            | 提供日志、指标、告警、配置、证据沉淀等通用能力。                         | 不参与主业务决策。                |

## 七、ACK、状态与可观测性对象

| 中文名    | 语义说明                         | 备注                   |
| ------ | ---------------------------- | -------------------- |
| 接收确认   | 表示报文已到达、基础校验通过、已进入前端缓存。      | 建议对应 `RxAck`。        |
| 生效确认   | 表示某张表已在约定时间边界正式生效。           | 建议对应 `ApplyAck`。     |
| 心跳     | 前后端存活与基本链路可用性的周期性状态对象。       | 可归入 ACK / 遥测 / 状态接口。 |
| 遥测     | 前端运行态、资源态、授时态等上报信息。          | 与 ACK 不同，偏运行信息。      |
| 状态码    | 用于表达接收、生效、错误、异常、告警等结果的编码化对象。 | 当前应先冻结语义，再冻结编码。      |
| 告警摘要   | 面向显控展示的异常结论层信息。              | 显控看结论，日志看过程。         |
| 过程证据   | 用于回放、追责、排障的详细过程记录。           | 建议下沉到日志与基础设施层。       |
| 可观测性分层 | “显控展示结论与态势，日志保留过程与证据”的分层原则。  | 当前已在上下文文档中冻结。        |

## 九、缓冲、执行与切换语义

| 中文名       | 语义说明                       | 备注                          |
| --------- | -------------------------- | --------------------------- |
| `current` | 前端当前正在执行的控制表语义。            | 对外正式缓冲语义之一。                 |
| `next`    | 前端已经装载、等待在边界切换生效的下一张控制表语义。 | 对外正式缓冲语义之一。                 |
| `queued`  | 服务器侧预备表语义。                 | 不是前端对外正式冻结语义；不要求 FPGA 真三缓冲。 |
| 提前下发      | 执行表在生效边界到来前提前发送到前端 `next`。 | 当前总原则之一。                    |
| 边界生效      | 不在最后时刻临时发控制，而是在边界到来时切换生效。  | 与软件秒边界事件配套。                 |
| 秒内按表推进    | 秒内不任意热更新，而是按表内定义推进执行。      | CPI 边界承担推进角色。               |

## 十、容易混淆、必须统一口径的词

| 中文名                | 语义说明                               | 备注                         |
| ------------------ | ---------------------------------- | -------------------------- |
| 秒边界 vs 服务器本地时间     | 秒边界是系统级逻辑边界；服务器本地时间只是本机时钟读数。       | 两者不能混用。                    |
| 软件秒边界事件 vs 硬件 1PPS | 前者是后端软件消费的逻辑事件；后者是物理授时脉冲。          | 文档明确前者不等于后者。               |
| 调度执行表 vs 管理/应急命令   | 前者服务周期性搜索/跟踪主链；后者服务启停、复位、安全态等管理动作。 | 必须硬拆分。                     |
| 候选安排 vs 定版执行表      | 候选安排只是待审批结果；定版执行表才可编号、下发、生效。       | 生成权与定版权必须分离。               |
| 参数快照 vs 处理质量标记     | 参数快照是权威上下文；处理质量标记只是附加结果说明。         | 不能互相替代。                    |
| 数据流 vs 控制流         | 数据流负责实时数据处理；控制流负责模式、边界、定版、ACK、降级。  | 两者必须分治。                    |
| 全局计划 vs 阵面执行表      | 全局计划是内部统一调度视图；阵面执行表是对外实际下发对象。      | 逻辑上一套，物理上三份。               |
| 接收确认 vs 生效确认       | 接收确认表示已收到并装载；生效确认表示已正式切换生效。        | 建议统一为 RxAck / ApplyAck 语义。 |
