## 1. 术语与状态标记

### 1.1 状态标记

- **[已明确]**：当前对话中已被明确说明、可作为当前上下文基线使用
- **[暂定]**：已有明显倾向，但仍可能在后续设计中调整
- **[待决策]**：当前尚未冻结，需要后续专门讨论

### 1.2 术语说明

- **阵面（Face）**：雷达的单个固定阵面；当前系统共 3 个阵面
- **前端**：靠近阵面侧的硬件执行部分，至少包含 ADC、FPGA，以及被 FPGA 控制的 TR/时序/收发参数执行链路
- **FPGA**：前端侧负责控制表接收/缓存/生效、原始数据组包并上送服务器的核心可编程逻辑
- **服务器**：后端计算平台，同时承担**数据处理节点**与**控制调度节点**两个角色
- **显控（HMI）**：人机交互/显示控制端，当前设计为服务器本机上的独立软件进程，未来可外接显示终端或对接数据中心
- **CPI / PRT / Beam**：作为协议与调度中的基础时间/执行单元使用，当前项目对它们已有约定，但部分递增/生效规则尚未全部冻结
- **波位安排表 / 波位控制表 / Schedule**：服务器下发给前端阵面/FPGA 的周期性控制执行表
- **RxAck**：接收确认
- **ApplyAck**：生效/执行确认

## 3. 系统总体拓扑（宏观）

### 3.1 物理系统形态 [已明确]

当前雷达系统为：

- **3 个固定阵面**
- 每个阵面覆盖约 **120° 扫描范围**
- 三个阵面分别独立产生原始雷达回波采样数据
- 每个阵面的前端都包含 ADC 采样与 FPGA 逻辑
- FPGA 负责将前端采样数据组装为上行 UDP 数据包并发往服务器
- 服务器通过 10G 光口分别接收 3 个阵面的数据

### 3.2 服务器网口拓扑 [已明确]

服务器具备一块：

- **4 口 10Gbps 全双工网卡**
- 单口速率上限：**10Gbps**
- 其中：
  - **1 号口 ↔ 1 号阵面**
  - **2 号口 ↔ 2 号阵面**
  - **3 号口 ↔ 3 号阵面**
  - **4 号口预留**

工程现场命名基线（已有资料）：

- `receiver0` → 1 号阵面
- `receiver1` → 2 号阵面
- `receiver2` → 3 号阵面
- `receiver3` → 预留

### 3.3 当前上行主链路 [已明确]

单个阵面的上行主链路为：

**前端 ADC 采样 → FPGA 组包 → UDP/IP → 服务器 10G 网卡 → 接收模块 → 信号处理模块 → 数据处理模块 → 网关/USP → 显控**

三个阵面同理，三路并行进入服务器。

### 3.4 当前下行主链路 [暂定趋于明确]

当前倾向的下行主链路为：

**显控/任务输入 → 服务器控制面 → 波位安排表生成 → 下发模块 → 与对应阵面共用同一 10G 全双工链路 → FPGA 接收并缓存 → 在约定时间边界生效**

说明：

- 当前倾向于 **上下行共用同一 10G 光口**
- 这是可行的，因为网口为**全双工**
- 下行控制不额外要求独立物理口
- 逻辑上依靠地址/逻辑 ID / 协议头进行区分

## 4. 前后端接口边界（极重要）

### 4.1 服务器视角的接口边界 [已明确]

作为后端开发者，当前不要求详细关心前端内部更深层的实现结构。  
对后端而言，前端的**可见接口边界**可收敛为两类：

#### A. 上行接口（前端 → 服务器）

- 前端 FPGA 负责将采样后的原始雷达数据组装为**自定义协议的 UDP 数据包**
- 服务器面向的输入单元是：**一个个 UDP 数据包**
- 后端只需认定：
  - 数据由前端 FPGA 组包
  - 协议字段由前后端共同约定
  - UDP 包是服务器的上行数据输入边界

#### B. 下行接口（服务器 → 前端）

- 服务器周期性生成**波位安排表 / 波位控制表**
- 服务器将控制表按约定协议发送给前端
- 前端 FPGA 接收控制表后，缓存并在约定边界生效
- 控制表会进一步作用到：
  - TR 组件
  - 时序控制
  - 发射参数
  - 接收参数
  - 波形/脉冲相关参数

### 4.2 松耦合原则 [已明确]

前后端设计应尽量保持**松耦合**：

- 前端对后端暴露：
  - 上行原始 UDP 数据接口
  - 下行波位安排表接口
  - 必要的 ACK / 状态 / 错误回传接口
- 后端无需感知前端全部内部模块细节
- 前端无需感知后端内部信号处理链路细节

该原则有利于：

- 协议边界清晰
- 联调职责清晰
- 前后端独立演进
- 替换/升级某一侧时降低耦合风险

## 5. 服务器硬件与资源分区基线

### 5.1 CPU/NUMA 资源规划 [已明确]

服务器为 **32 核 CPU**，并划分为两组用途：

- **Node 1 / CPU 16-31**：承担实时数据面
- **Node 0 / CPU 0-15**：承担非实时管理面

### 5.2 物理/逻辑归属 [已明确]

当前已知工程基线：

- 10G 数据网卡位于 **Node 1**
- GPU 计算卡位于 **Node 1**
- 所有实时数据处理相关链路尽量驻留在 **Node 1**
- 所有非实时链路尽量放在 **Node 0**

### 5.3 数据面 / 管理面分离 [已明确]

当前服务器采取明确的“双平面”思路：

#### 数据面（实时）
负责：

- 网卡收数
- 数据接收与基础校验
- 重组/组帧
- 信号处理
- 数据处理
- 实时主链路

#### 管理面（非实时）
负责：

- 调度
- 日志
- 监控
- 显控相关控制逻辑
- 旁路落盘的异步转存
- 其他非实时系统服务

该分离的目的在于：

- 确保实时链路不被管理任务污染
- 保持 NUMA 亲和性
- 降低调度抖动与系统底噪
- 给后续性能优化留下空间

## 6. 上行原始数据流（当前理解）

### 6.1 上行链路的本质 [已明确]

上行链路承载的是：

- **前端 ADC 采样之后的原始雷达数据**
- 这些数据由 FPGA 组装为自定义 UDP 数据包
- 服务器将其作为原始数据接收并进入后续处理链

### 6.2 FPGA 在上行中的角色 [已明确]

FPGA 在上行数据链路中的职责包括：

1. 接收 ADC 采样数据
2. 按双方约定的协议格式进行组包
3. 填充协议字段（如 CPI/PRT/通道/包计数等）
4. 将组包结果通过 UDP/IP 发送给服务器

因此：

- **上行原始数据包是 FPGA 发出的**
- 协议由前后端共同定义
- 服务器直接面对 FPGA 生成的数据包

### 6.3 当前宏观阶段对字段细节的态度 [已明确]

虽然已有若干字段草案与示例，但当前架构阶段：

- **不强求所有字段细节冻结**
- 只需确认：
  - UDP 是输入边界
  - 协议可扩展
  - 字段由双方约定
  - 服务器按“包 → 重组 → 处理”思路工作

### 7.1 波位控制表早期简化约定（资料项）

已有一版较早期/较简化的控制表背景：

- 控制表长度：**2048 字节**
- 包头：`0x55AAFF00`
- CPI 计数：2 字节
- 其余字节先填 0

该版本可视作控制表载荷雏形，不应简单等同于完整成熟的控制协议。

### 7.2 上行 UDP 数据包早期简化约定（资料项）

已有一版较早期/较简化的上行数据报文背景：

- 单个 UDP 包长度：**5120 字节**
- 一个 PRT 对应 **16 个 UDP 包**
- 4 个通道，每个通道 4 包
- 包头：`0x55AAFF03`
- 包内带：
  - CPI 计数
  - PRT 计数
  - 通道计数
  - 包计数
  - IQ 点数
  - 包尾
- 数据区有效长度：**5104 字节**
- IQ 按 `I/Q` 顺序组织

## 8. 下行波位控制流（当前理解）

### 8.1 下行控制的本质 [已明确]

下行链路承载的是：

- 服务器生成的**周期性波位安排表**
- 这些表不是“随手发一个控制命令”，而是**按周期下发的一张执行表**

### 8.2 当前设计倾向 [已明确]

当前较明确的设计倾向如下：

1. **下行控制与上行数据共用同一 10G 全双工链路**
2. **由服务器周期性下发一张表**
3. 三个阵面各自获得与自身相关的控制信息
4. 三个阵面在物理执行上是独立的，但在时间上要求严格同步
5. 授时基准来自 **北斗 GNSS**
6. 前端 FPGA 采用**双缓冲**思想

### 8.3 服务器与三阵面的表关系 [当前已形成清晰倾向]

虽然三阵面的执行内容不同，但更合理的服务器内部组织方式是：

- 服务器内部先维护一份**统一的全局调度视图/总计划**
- 然后根据阵面差异，拆分/生成：
  - Face1 执行表
  - Face2 执行表
  - Face3 执行表

因此建议认知为：

> **服务器内部统一调度，对外按阵面独立下发表。**

这比“完全无关联地独立生成三份表”更利于保持全局一致性。

### 8.3.1 全局周期号与阵面执行表的关系 [建议冻结]

当前更合理且建议直接冻结的设计是：

- **一个雷达，三个阵面**
- 三个阵面**共享同一个 Global Schedule ID / Global Epoch**
- 三个阵面**共享同一个 GNSS 秒边界**
- 三个阵面**共享同一个全局表版本号**
- 但每个阵面保留各自独立的 **FaceSchedule / 阵面执行表**

可将其理解为：

> **逻辑上一个全局计划，物理上三份阵面执行表。**

相较于“三个阵面各自维护独立周期号”的方案，这种组织方式更有利于：

- 显控侧统一展示当前运行态
- 后端日志按同一周期边界对齐
- ApplyAck 按全局版本聚合判断
- 故障回滚、告警聚合与问题定位
- 后续多阵面协同搜索/协同跟踪扩展

### 8.4 下行表的发送方式 [暂定]

当前倾向：

- 阵面专属表走**单播/定向发送**
- 不建议把“三份不同内容的阵面表”依赖广播实现
- 广播/组播更适合：
  - 全局时间通知
  - 统一停机/安全态通知
  - 心跳/发现类消息
  - 共性控制消息

### 8.5 ACK 需求 [已明确]

当前明确需要 ACK。  
比较合理的语义分层为：

#### 1）RxAck（接收确认）
表示：

- 报文已到达
- 基础校验通过
- 已进入前端缓存/next 表缓冲

#### 2）ApplyAck（生效确认）
表示：

- 某张表已在约定时间边界正式生效
- 可附带表版本号/生效时间/错误码

### 8.6 控制协议底层承载 [待决策]

当前用户倾向过几种可能：

- TCP
- 更高级的基于 UDP 的 UCIP（表述为思路方向）
- 自定义协议内自带 ACK 字段

当前尚未冻结底层选择。  
但**无论最终底层用 TCP 还是 UDP 类方案，ACK 语义本身都建议保留。**

### 8.7 控制表的连续续跑与容错策略 [建议冻结]

当前建议将前端控制表缓冲与生效语义明确为：

- **当前生效表：`current`**
- **待生效表：`next`**
- **可选预备表：`queued`**

在此基础上，建议允许如下容错行为：

- 若新表未按时到达，允许 `current` 在**限定时间窗**内继续运行多个周期
- 系统继续维持发射/接收主流程，避免因一次控制抖动导致整体掉空
- 同时上报“控制更新超时”或“陈旧表续跑”类告警
- 若超出续跑上限，再进入降级或安全态

更合理的处理顺序建议为：

> **先保旧表 → 再告警 → 再超时降级 → 最后进入安全态。**

该策略更符合雷达系统对连续运行稳定性的要求，也比“单次控制超时立即停摆”更稳健。

## 10. 显控在整个控制流中的角色（当前已理顺）

### 10.1 显控不应承担的职责 [已明确]

显控**不应**成为：

- 逐波束手工操作员
- 每秒直接拼三份波位执行表的人工工具
- 直接驱动每个 Beam 实时变化的“低级执行器”

如果这样设计，会导致：

- 人工依赖过重
- 难以闭环
- 无法自动跟踪/自动调度
- 后续扩展性极差

### 10.2 显控更合理的职责 [已明确]

显控更适合承担：

1. **任务与模式输入**
   - 搜索模式
   - 跟踪模式
   - 测试模式
   - 通信检查
   - 增益/标定等
2. **任务级参数输入**
   - 搜索扇区
   - 目标列表
   - 优先级
   - 工作约束
3. **状态展示**
   - 执行状态
   - 阵面状态
   - 当前模式
   - 表版本/ACK/错误
   - 处理结果/态势信息
4. **人工干预**
   - 启停
   - 模式切换
   - 安全停机
   - 参数切换

### 10.3 服务器控制面的角色 [已明确]

服务器控制面才是：

- 调度器
- 模式状态机承载者
- 波位表生成器
- 阵面表拆分器
- 前后端控制协议的终止点
- ACK / 状态汇聚点

### 10.4 一个简洁的职责定义（推荐冻结）

> **显控负责任务与模式输入、状态展示与人工干预；服务器控制面负责将任务意图转换为周期性波位安排表并下发至各阵面；前端阵面负责在统一时间边界按表执行并回传状态。**

### 10.5 显控下发的最小对象 [建议冻结]

显控侧更合理的下发对象，不应是逐波束执行表本身，而应是“任务意图组合”。  
建议显控的最小输入对象定义为：

- **工作模式**
- **任务模板**
- **目标列表**
- **跟踪/搜索策略参数**
- **资源约束参数**
- **人工干预命令**

而不建议显控持续直接下发：

- 三个阵面的逐秒波位表
- 逐波束 Az / El / 脉宽 / 频点等执行层参数

原因在于，当前已知的 32 字节波束项中已包含大量执行层字段，例如：

- BeamID
- BeamWorkStatus
- AzimuthAngle / ElevationAngle
- WorkFreqIndex
- MGC 增益
- 周期组合模式
- 长短码波形
- 带宽 / 脉宽 / 周期
- 积累点数
- 采样点数
- 数据率

这些字段明显更适合作为**服务器调度器生成给前端执行器的表项**，而不是显控层持续手工输入的对象。

因此建议将以下表述作为固定结论：

> **显控下发任务意图，服务器生成执行表，前端按表执行。**

### 10.6 显控状态展示与日志下沉原则 [建议冻结]

显控侧建议展示“运行态摘要”，而将详细过程证据下沉到日志系统。

#### 显控应展示的最小状态集

建议至少包括：

- 当前全局周期号 / 表版本号
- 当前生效时间戳
- Face1 / Face2 / Face3 当前 Apply 状态
- `next` 表是否已装载
- 当前是否处于旧表续跑状态
- 上次下发是否接收成功
- 上次 Apply 是否成功
- 告警摘要
- 当前授时状态（GNSS / NTP / Holdover）

#### 日志系统保留的过程细节

建议至少保留：

- 表校验失败原因
- 非法字段定位
- 具体哪个 face 超时
- ACK 明细
- 表切换时刻
- 降级原因
- 失效计数
- 连续重试统计

也就是：

> **显控看“结论和态势”，日志看“过程和证据”。**

## 11. 时间同步与生效边界

### 11.1 严格同步要求 [已明确]

三个阵面要求严格同步。  
这意味着：

- 不能把阵面控制视为完全互不相关的三套系统
- 必须存在统一时间基准
- 必须明确定义控制表“何时生效”

### 11.2 授时体系 [已明确]

当前授时/守时体系为三层降级：

1. **北斗 GNSS 外部统一授时**（首选）
2. **NTP**（GNSS 失效后的降级）
3. **本地晶振 holdover**（进一步守时）

### 11.3 调度周期与执行粒度 [暂定已基本收敛]

结合当前协议草案、授时条件与整体调度思路，当前更合理的宏观冻结建议为：

- **大周期 / Schedule Epoch = 1 秒**
- **表内执行边界 = CPI**
- **波束推进粒度 = Beam / PRT**

也就是说：

- **秒边界**：决定切换哪一张表、进入哪一个全局周期
- **CPI 边界**：决定表内往前推进到哪个执行段
- **PRT / 包级**：承载数据生产、发送、接收与重组

这个定义与当前系统条件天然匹配：

- 已具备北斗 GNSS 的**秒脉冲**作为统一外部授时基准
- 上行协议中已经显式携带 **CPI / PRT / 通道计数 / 包计数**
- 当前已知控制表长度暂定为 **2048 字节**
- 当前已知上行原始数据 UDP 包长度为 **5120 字节**
- 当前约定 **每个 PRT 对应 16 个 UDP 包**
- 当前约定数据区有效长度为 **5104 字节**

这些背景已经足以支撑“**秒级调度 + CPI/PRT 执行**”的宏观模型。

### 11.4 当前建议的认知框架

建议将以下表述作为当前阶段的统一理解：

> **大周期按 1 秒定义；秒边界负责换表/换周期，CPI 边界负责表内推进，PRT/包级负责数据生产与接收重组。**

## 12. 控制流是否闭环：跟踪模式下的动态调整

### 12.1 用户的真实疑问 [已明确]

用户提出一个关键问题：

- 在目标跟踪场景中，是否需要根据前一个波位获取的雷达回波结果，临时调整后续波位安排？

### 12.2 当前合理答案（架构层面）

是的，**跟踪模式通常需要闭环调整**。  
但这不意味着显控人工逐次干预，而意味着：

- 数据处理/跟踪模块输出目标状态估计
- 服务器控制面读取这些结果
- 在下一个调度周期中自动修正后续波位表
- 再下发给前端

### 12.3 控制闭环的合理路径

**上行原始数据 → 信号处理 → 数据处理/目标状态估计 → 服务器控制面重规划 → 新周期波位表 → 前端执行**

因此：

- 显控是上层任务输入者与观察者
- 服务器控制面才是自动闭环调度执行者

## 13. 服务器内部宏观模块划分（当前建议）

### 13.1 数据面模块（实时链）

建议服务器内部宏观数据面模块至少包括：

1. **接收模块（Receiver）**
   - 网卡收数
   - 基础协议解析
   - 基础合法性校验
   - 包级接收统计
2. **重组/组织模块**
   - 按约定对包进行组织

3. **信号处理模块（Signal Processing）**
   - 主算法链
   - GPU 加速部分在此阶段承载
4. **数据处理模块（Data Processing）**
   - 检测/估计/后处理/目标级输出
5. **网关/USP 模块**
   - 输出到显控或外部系统
6. **原始数据旁路落盘分支**
   - 从接收端尽早分流
   - 用于调试与算法分析保留原始资料

### 13.2 控制面模块（非实时/较低实时）

建议服务器内部控制面模块至少包括：

1. **显控接口模块**
2. **模式状态机**
3. **任务管理模块**
4. **波位表生成器**
5. **阵面表拆分与下发模块**
6. **ACK / 遥测 / 状态汇聚模块**
7. **日志 / 监控 / 运维模块**

### 13.3 角色总结

服务器当前同时扮演两个角色：

#### A. 数据处理节点
承担三阵面原始数据接收与主处理链

#### B. 控制调度节点
承担模式管理、表生成、表下发、状态汇聚

## 14. 原始数据旁路落盘策略

### 14.1 落盘位置与时机 [已明确]

当前明确倾向：

- **原始数据旁路落盘放在接收端**
- 即：**一拿到 UDP 包就尽快收下来**
- 目的是保存最原始、最靠近现场的采集数据

### 14.2 落盘层级 [已明确]

当前倾向的落盘策略为：

1. **UDP 包到达接收端即旁路分流**
2. 使用 **NVMe 作为一级缓存**
3. 后台异步转存到 `/data` 数据盘

### 14.3 这样做的原因

- 现场机械盘/大容量盘写入能力有限
- 不能让原始录制阻塞实时主链路
- NVMe 更适合作为瞬时吸收层
- 后台异步转存更适合长期留存

### 14.4 与前述资源分区的关系

该策略也符合：

- 实时数据面尽量不被长期写盘拖慢
- 非实时后台转存放在管理面处理

## 15. 显控部署与后续扩展需求

### 15.1 当前显控部署形态 [已明确]

当前设计下：

- 显控是**本机软件模块**
- 以**独立进程**方式存在
- 不与信号处理模块合并为一个进程

### 15.2 为什么当前仍保留网关/外部输出思路 [已明确]

虽然当前显控在服务器本机上运行，理论上本地 IPC 也可完成联通，但工程上仍保留网关/对外输出策略，是因为后续有扩展需求：

1. 调试时在本机临时查看
2. 外接外部电脑显示
3. 对接更大的数据中心/服务中心

因此当前更合理的理解是：

- **显控当前是本机独立进程**
- **系统未来具备本地/外接/中心化三种输出模式的扩展可能**
- 因此网关/USP 仍然是合理存在的结构

## 16. 当前架构上已经较清晰的原则（建议冻结）

以下原则已在当前讨论中具有较强一致性，建议作为后续设计基线：

### 16.1 接口边界原则
- 服务器上行输入边界是**前端 FPGA 发来的 UDP 原始数据包**
- 服务器下行输出边界是**发往前端 FPGA 的波位安排表**

### 16.2 松耦合原则
- 前后端通过协议接口解耦
- 后端不深入依赖前端内部全部实现细节
- 前端不依赖后端内部实现细节

### 16.3 三阵面控制组织原则
- 服务器内部统一调度
- 对外按阵面独立下发表

### 16.4 显控职责原则
- 显控负责“任务/模式”
- 服务器负责“执行表生成与下发”
- 前端负责“按表执行”

### 16.5 资源分区原则
- 实时数据面尽量驻留 Node 1
- 非实时管理面尽量驻留 Node 0

### 16.6 原始数据保留原则
- 接收端尽早旁路保存原始数据
- NVMe 一级缓存 + 异步转存

### 16.7 同步原则
- 三阵面在时间上严格同步
- 授时基于北斗 GNSS，具备 NTP 与 holdover 降级

### 16.8 缓冲原则
- 前端 FPGA 明确采用双缓冲（current / next）思想

### 16.9 ACK 原则
- 建议区分 RxAck 与 ApplyAck 两级确认

### 16.10 调度周期原则
- 大周期默认按 **1 秒** 定义
- 秒边界负责切换全局调度周期与控制表
- CPI 负责表内执行推进
- PRT / 包级负责数据生产、发送、接收与重组

### 16.11 全局周期号原则
- 一个雷达的三个阵面共享同一个 Global Schedule ID / Global Epoch
- 共享同一个 GNSS 秒边界
- 共享同一个全局表版本号
- 对外表现为三份阵面执行表，对内保持统一全局计划

### 16.12 容错续跑原则
- 前端控制表建议采用 `current / next / queued` 语义
- 新表未到时允许旧表在限定时间窗内续跑
- 优先保连续运行，再逐步告警、降级、进入安全态

### 16.13 状态展示分层原则
- 显控展示表级结论、运行态势与告警摘要
- 详细校验、ACK、超时与降级证据下沉到日志系统

## 18. 当前建议的系统级认知（可直接供新 AI 对话使用）

下面这段可作为简洁背景说明直接复用：

> 当前雷达系统由 3 个固定阵面组成，每个阵面覆盖约 120°。前端 ADC 采样后的原始数据由前端 FPGA 按双方自定义协议组装为 UDP/IP 数据包，经各自对应的 10G 光口发送到服务器。服务器具备 4 口 10G 全双工网卡，其中 3 口分别对应 3 个阵面，1 口预留。服务器为 32 核双 NUMA 架构，Node 1 负责实时数据面（网卡/GPU/接收/信号处理），Node 0 负责非实时管理面（调度、日志、监控、异步转存等）。服务器内部同时承担“数据处理节点”和“控制调度节点”两个角色。当前调度大周期建议按 1 秒定义，并与北斗 GNSS 秒脉冲对齐；秒边界负责全局换表/换周期，CPI 边界负责表内推进，PRT/包级负责数据生产与接收重组。三个阵面共享同一个 Global Schedule ID / Global Epoch、同一个全局表版本号与同一个秒边界，但各自持有独立的 FaceSchedule，形成“逻辑上一套全局计划、物理上三份阵面执行表”的组织方式。上行主链路为：前端采样 → FPGA 组包 → UDP/IP → 服务器接收模块 → 信号处理模块 → 数据处理模块 → 网关/USP → 显控。下行主链路当前倾向为：显控输入任务意图/模式 → 服务器控制面生成周期性波位安排表 → 按阵面拆分为三份执行表 → 通过与上行共用的 10G 全双工链路发往各前端 FPGA → FPGA 按 `current / next / queued` 语义缓存 → 在统一时间边界生效。显控不应直接长期手工逐波束生成执行表，而应负责任务/模式输入、状态展示和人工干预；真正的波位表生成与调度逻辑应位于服务器控制面。系统要求三阵面严格同步，授时基于北斗 GNSS，降级链路为 NTP 与本地晶振 holdover。原始数据需要在接收端尽早旁路保存，采用 NVMe 一级缓存并后台异步转存至数据盘。显控应展示表级运行摘要与告警结论，详细 ACK、校验、超时与降级过程应下沉至日志系统。当前协议字段细节尚未全部冻结，但前后端接口边界、资源分区、显控职责、调度粒度与控制流/数据流主脉络已较为清晰。

## 20. 这份文档对后续 AI 协作的价值

后续任何 AI 对话，只要基于本文档，都应优先遵循以下前提：

1. 不把显控误解为“逐波束手工控制端”
2. 不把前端误解为只传元数据而不控制硬件
3. 不把服务器误解为纯转发节点
4. 明确服务器是“控制调度 + 数据处理”双角色
5. 明确三阵面是独立执行、统一同步
6. 明确上下行当前倾向共用同一 10G 全双工链路
7. 明确前端/后端之间的协议边界应保持松耦合
8. 明确当前阶段重点是系统级主脉络，而非字段逐字节穷举

## 21. 附：当前最值得继续回答的 6 个实现冻结问题

在当前宏观结论已基本形成后，以下问题更值得继续往“可实现、可联调、可运维”方向细化：

1. 1 秒大周期内，CPI、Beam、PRT 三者的精确定义、编号规则和推进关系如何落字节级/实现级规范？
2. Global Schedule ID、全局表版本号、FaceSchedule ID、生效时间戳应如何编码，并如何支持回放、追踪与回滚？
3. `current / next / queued` 的缓存深度、切换条件、装载时序与超时续跑窗口应如何设定？
4. RxAck、ApplyAck、错误码、遥测字段应如何统一定义，才能同时服务联调、运行监控与故障定位？
5. 当某一阵面 Apply 失败而其余阵面成功时，系统整体应如何降级、告警、回滚或进入安全态？
6. 显控摘要状态与日志明细之间应如何分层映射，既保证可观测性，又避免界面过载？

## 22. 一句话总结（可直接复用）

> 当前系统应理解为：**显控下发任务意图，服务器按 1 秒全局周期统一调度并为三个阵面生成独立执行表，前端 FPGA 在统一时间基准下接表执行并回传状态；服务器同时承担三阵面原始数据接收与主处理链任务，并通过数据面/管理面分离保障实时性。**
