## 1. 术语与状态标记

### 1.1 状态标记

- **[已明确]**：当前对话中已被明确说明、可作为当前上下文基线使用
- **[暂定]**：已有明显倾向，但仍可能在后续设计中调整
- **[待决策]**：当前尚未冻结，需要后续专门讨论

### 1.2 术语说明

- **阵面（Face）**：雷达的单个固定阵面；当前系统共 3 个阵面
- **前端**：靠近阵面侧的硬件执行部分，至少包含 ADC、FPGA，以及被 FPGA 控制的 TR/时序/收发参数执行链路
- **FPGA**：前端侧负责控制表接收/缓存/生效、原始数据组包并上送服务器的核心可编程逻辑
- **服务器**：后端计算平台，同时承担**数据处理节点**与**控制调度节点**两个角色
- **显控（HMI）**：人机交互/显示控制端，当前设计为服务器本机上的独立软件进程，未来可外接显示终端或对接数据中心
- **CPI / PRT / Beam**：作为协议与调度中的基础时间/执行单元使用，当前项目对它们已有约定，但部分递增/生效规则尚未全部冻结
- **调度执行表（Schedule）**：服务器下发给前端阵面/FPGA 的周期性控制执行表，用于搜索/跟踪/精跟主链
- **管理/应急命令**：服务器面向前端下发的非周期性命令，用于启停、复位、安全态、测试模式与链路探测等管理动作
- **软件秒边界事件**：由前端 FPGA 分发并上送服务器控制面的 1PPS / 边界通知语义，用于周期推进、生效边界管理与授时状态回显；它不是高精度硬件授时本体，也不是服务器硬中断
- **参数快照**：由接收模块前移挂载、跨数据流与控制流透传的权威上下文锚点
- **控制建议输入对象**：数据处理模块输出给控制面的候选控制输入，不等于最终执行表
- **RxAck**：接收确认
- **ApplyAck**：生效/执行确认

## 3. 系统总体拓扑（宏观）

### 3.1 物理系统形态 [已明确]

当前雷达系统为：

- **3 个固定阵面**
- 每个阵面覆盖约 **120° 扫描范围**
- 三个阵面分别独立产生原始雷达回波采样数据
- 每个阵面的前端都包含 ADC 采样与 FPGA 逻辑
- FPGA 负责将前端采样数据组装为上行 UDP 数据包并发往服务器
- 服务器通过 10G 光口分别接收 3 个阵面的数据

### 3.2 服务器网口拓扑 [已明确]

服务器具备一块：

- **4 口 10Gbps 全双工网卡**
- 单口速率上限：**10Gbps**
- 其中：
  - **1 号口 ↔ 1 号阵面**
  - **2 号口 ↔ 2 号阵面**
  - **3 号口 ↔ 3 号阵面**
  - **4 号口预留**

工程现场命名基线（已有资料）：

- `receiver0` → 1 号阵面
- `receiver1` → 2 号阵面
- `receiver2` → 3 号阵面
- `receiver3` → 预留

### 3.3 当前上行主链路 [已明确]

单个阵面的上行主链路为：

**前端 ADC 采样 → FPGA 组包 → UDP/IP → 服务器 10G 网卡 → 接收模块 → 信号处理模块 → 数据处理模块 → 网关/USP → 显控**

三个阵面同理，三路并行进入服务器。

### 3.4 当前下行主链路 [暂定趋于明确]

当前倾向的下行主链路为：

**显控/任务输入 → 服务器控制面 → 候选安排生成 / 调度定版 / 下行发送 → 与对应阵面共用同一 10G 全双工链路 → FPGA 装载到 `next` → 在约定时间边界切换为 `current`**

说明：

- 当前倾向于 **上下行共用同一 10G 光口**
- 这是可行的，因为网口为**全双工**
- 下行控制不额外要求独立物理口
- 逻辑上依靠地址/逻辑 ID / 协议头进行区分

## 4. 前后端接口边界（极重要）

### 4.1 服务器视角的接口边界 [已明确]

作为后端开发者，当前不要求详细关心前端内部更深层的实现结构。
对后端而言，前端的**可见接口边界**正式冻结为 5 类：

#### A. 上行原始数据接口（前端 → 服务器）

- 前端 FPGA 负责将采样后的原始雷达数据组装为**自定义协议的 UDP 数据包**
- 服务器面向的输入单元是：**一个个 UDP 数据包**
- 数据面以这些原始包为接收入口，并进一步重组为**完整 CPI 数据单元**

#### B. 下行调度执行表接口（服务器 → 前端）

- 服务器周期性生成并提前下发**调度执行表**
- 调度执行表服务搜索/跟踪/精跟主链
- 前端对外可见状态语义冻结为 `current / next`
- 调度执行表会进一步作用到：
  - TR 组件
  - 时序控制
  - 发射参数
  - 接收参数
  - 波形/脉冲相关参数

#### C. 管理/应急命令接口（服务器 → 前端）

- 用于启停、复位、安全态、测试模式、链路探测等管理动作
- 不承担搜索/跟踪主链的周期性排表职责

#### D. ACK / 遥测 / 状态接口（前端 → 服务器）

- 用于回传 RxAck、ApplyAck、心跳、遥测、错误与运行状态
- 聚合视图统一由服务器控制面产出

#### E. 时基边界事件接口（前端 → 服务器）

- 前端上送 **1PPS / 边界通知包**
- 后端将其定义为**软件秒边界事件**
- 它用于提供统一边界语义、周期推进依据与授时状态回显
- 它**不是**高精度硬件授时本体，也**不是**服务器硬中断

### 4.2 松耦合原则 [已明确]

前后端设计应尽量保持**松耦合**：

- 前端对后端暴露：
  - 上行原始 UDP 数据接口
  - 下行调度执行表接口
  - 管理/应急命令接口
  - ACK / 遥测 / 状态接口
  - 时基边界事件接口
- 后端无需感知前端全部内部模块细节
- 前端无需感知后端内部信号处理链路细节

该原则有利于：

- 协议边界清晰
- 联调职责清晰
- 前后端独立演进
- 替换/升级某一侧时降低耦合风险

## 5. 服务器硬件与资源分区基线

### 5.1 CPU/NUMA 资源规划 [已明确]

服务器为 **32 核 CPU**，并划分为两组用途：

- **Node 1 / CPU 16-31**：承担实时数据面
- **Node 0 / CPU 0-15**：承担非实时管理面

### 5.2 物理/逻辑归属 [已明确]

当前已知工程基线：

- 10G 数据网卡位于 **Node 1**
- GPU 计算卡位于 **Node 1**
- 所有实时数据处理相关链路尽量驻留在 **Node 1**
- 所有非实时链路尽量放在 **Node 0**

### 5.3 数据面 / 管理面分离 [已明确]

当前服务器采取明确的“双平面”思路：

#### 数据面（实时）
负责：

- 网卡收数
- 数据接收与基础校验
- 重组/组帧
- 信号处理
- 数据处理
- 实时主链路

#### 管理面（非实时）
负责：

- 调度
- 日志
- 监控
- 显控相关控制逻辑
- 旁路落盘的异步转存
- 其他非实时系统服务

该分离的目的在于：

- 确保实时链路不被管理任务污染
- 保持 NUMA 亲和性
- 降低调度抖动与系统底噪
- 给后续性能优化留下空间

## 6. 上行原始数据流（当前理解）

### 6.1 上行链路的本质 [已明确]

上行链路承载的是：

- **前端 ADC 采样之后的原始雷达数据**
- 这些数据由 FPGA 组装为自定义 UDP 数据包
- 服务器将其作为原始数据接收并进入后续处理链

### 6.2 FPGA 在上行中的角色 [已明确]

FPGA 在上行数据链路中的职责包括：

1. 接收 ADC 采样数据
2. 按双方约定的协议格式进行组包
3. 填充协议字段（如 CPI/PRT/通道/包计数等）
4. 将组包结果通过 UDP/IP 发送给服务器

因此：

- **上行原始数据包是 FPGA 发出的**
- 协议由前后端共同定义
- 服务器直接面对 FPGA 生成的数据包

### 6.3 当前宏观阶段对字段细节的态度 [已明确]

虽然已有若干字段草案与示例，但当前架构阶段：

- **不强求所有字段细节冻结**
- 只需确认：
  - UDP 是输入边界
  - 协议可扩展
  - 字段由双方约定
  - 服务器按“包 → 重组 → 处理”思路工作

### 7.1 波位控制表早期简化约定（资料项）

已有一版较早期/较简化的控制表背景：

- 包头：`0x55AAFF00`

- 其余字节先填 0

该版本可视作控制表载荷雏形，不应简单等同于完整成熟的控制协议。

### 7.2 上行 UDP 数据包早期简化约定（资料项）

已有一版较早期/较简化的上行数据报文背景：

- 单个 UDP 包长度：**5120 字节**
- 一个 PRT 对应 **16 个 UDP 包**
- 4 个通道，每个通道 4 包
- 包头：`0x55AAFF03`
- 包内带：
  - CPI 计数
  - PRT 计数
  - 通道计数
  - 包计数
  - IQ 点数
  - 包尾
- 数据区有效长度：**5104 字节**
- IQ 按 `I/Q` 顺序组织

## 8. 下行波位控制流（当前理解）

### 8.1 下行控制的本质 [已明确]

下行链路承载两类南向对象：

- **调度执行表**：服务器生成的周期性执行对象，服务搜索/跟踪/精跟主链
- **管理/应急命令**：用于启停、复位、安全态、测试模式、链路探测等非周期性管理动作

其中调度执行表不是“随手发一个控制命令”，而是**按周期提前下发、在边界生效的一张执行表**。

### 8.2 当前设计倾向 [已明确]

当前较明确的设计倾向如下：

1. **下行控制与上行数据共用同一 10G 全双工链路**
2. **由服务器周期性生成并提前下发表**
3. 三个阵面各自接收与自身相关的执行内容
4. 三个阵面物理执行独立，但时间上严格同步

时间、边界与缓冲语义以后文第 11 节为准。

### 8.3 服务器与三阵面的表关系 [当前已形成清晰倾向]

虽然三阵面的执行内容不同，但更合理的服务器内部组织方式是：

- 服务器内部先维护一份**统一的全局调度视图/总计划**
- 然后根据阵面差异，拆分/生成：
  - Face1 执行表
  - Face2 执行表
  - Face3 执行表

因此建议认知为：

> **服务器内部统一调度，对外按阵面独立下发表。**

这比“完全无关联地独立生成三份表”更利于保持全局一致性。

### 8.3.1 全局周期号与阵面执行表的关系 [建议冻结]

冻结结论如下：

- 三个阵面共享同一个 **GlobalEpoch**
- 三个阵面共享同一个**软件秒边界语义**
- 三个阵面共享同一个**全局表版本号**
- 每个阵面保留各自独立的 **FaceSchedule**

统一理解为：

> **逻辑上一套全局计划，物理上三份阵面执行表。**

### 8.4 下行表的发送方式 [暂定]

当前倾向：

- 阵面专属表走**单播/定向发送**
- 不建议把“三份不同内容的阵面表”依赖广播实现
- 广播/组播更适合：
  - 全局时间通知
  - 统一停机/安全态通知
  - 心跳/发现类消息
  - 共性控制消息

### 8.5 ACK 需求 [已明确]

当前明确需要 ACK。
比较合理的语义分层为：

#### 1）RxAck（接收确认）
表示：

- 报文已到达
- 基础校验通过
- 已进入前端缓存/next 表缓冲

#### 2）ApplyAck（生效确认）
表示：

- 某张表已在约定时间边界正式生效
- 可附带表版本号/生效时间/错误码

### 8.6 控制协议底层承载 [待决策]

当前用户倾向过几种可能：

- TCP
- 更高级的基于 UDP 的 UCIP（表述为思路方向）
- 自定义协议内自带 ACK 字段

当前尚未冻结底层选择。
但**无论最终底层用 TCP 还是 UDP 类方案，ACK 语义本身都建议保留。**

### 8.7 控制表的连续续跑与容错策略 [建议冻结]

当前建议将前端控制表缓冲与生效语义明确为：

- **当前生效表：`current`**
- **待生效表：`next`**
- **服务器侧可选预备表语义：`queued`**

其中：

- `queued` 是**服务器侧预备表语义**
- 它不要求 FPGA 必须实现真三缓冲
- 后端需要冻结的是**可见状态语义**，而不是 FPGA 内部实现细节

在此基础上，建议允许如下容错行为：

- 若新表未按时到达，允许 `current` 在**限定时间窗**内继续运行多个周期
- 系统继续维持发射/接收主流程，避免因一次控制抖动导致整体掉空
- 同时上报“控制更新超时”或“陈旧表续跑”类告警
- 若超出续跑上限，再进入降级或安全态

并明确：

- **RxAck** 对应：`next` 已装载成功
- **ApplyAck** 对应：`current` 已切换生效

更合理的处理顺序建议为：

> **先保旧表 → 再告警 → 再超时降级 → 最后进入安全态。**

该策略更符合雷达系统对连续运行稳定性的要求，也比“单次控制超时立即停摆”更稳健。

## 10. 显控在整个控制流中的角色（当前已理顺）

### 10.1 显控不应承担的职责 [已明确]

显控**不应**成为：

- 逐波束手工操作员
- 每秒直接拼三份执行表的人工工具
- 面向前端执行层的低级控制器

### 10.2 显控更合理的职责 [已明确]

显控更适合承担：

1. **任务与模式输入**
   - 搜索 / 跟踪 / 测试等运行模式
2. **任务级参数输入**
   - 搜索扇区
   - 目标列表
   - 优先级
   - 工作约束
3. **状态展示**
   - 执行状态
   - 阵面状态
   - 当前模式
   - 表版本/ACK/错误
   - 处理结果/态势信息
4. **人工干预**
   - 启停
   - 模式切换
   - 安全停机
   - 参数切换

### 10.3 服务器控制面的角色 [已明确]

服务器控制面才是：

- 调度器
- 模式状态机承载者
- 执行表生成器
- 阵面表拆分器
- 前后端控制协议的终止点
- ACK / 状态汇聚点

### 10.4 一个简洁的职责定义（推荐冻结）

> **显控统一面向服务器控制面输入任务意图、状态展示与人工干预；服务器控制面负责将任务意图转换为调度执行表和管理/应急命令，并提前下发至各阵面；前端阵面永远只对服务器控制面负责。**

### 10.5 显控下发的最小对象 [建议冻结]

显控侧的最小输入对象不应是逐波束执行表，而应是“任务意图组合”，建议包括：

- **工作模式**
- **任务模板**
- **目标列表**
- **跟踪/搜索策略参数**
- **资源约束参数**
- **人工干预命令**

显控不应持续直接下发：

- 三个阵面的逐秒波位表
- 逐波束 Az / El / 脉宽 / 频点等执行层参数

因此建议将以下表述作为固定结论：

> **显控下发任务意图，服务器控制面生成执行表与管理命令，前端按表执行并对服务器控制面回传状态。**

### 10.6 显控状态展示与日志下沉原则 [建议冻结]

显控侧建议展示“运行态摘要”，而将详细过程证据下沉到日志系统。

#### 显控应展示的最小状态集

建议至少包括：

- 当前全局周期号 / 表版本号
- 当前生效时间戳
- Face1 / Face2 / Face3 当前 Apply 状态
- `next` 表是否已装载
- 当前是否处于旧表续跑状态
- 上次下发是否接收成功
- 上次 Apply 是否成功
- 告警摘要
- 当前授时状态（GNSS / PTP / Holdover）

#### 日志系统保留的过程细节

建议至少保留：

- 表校验失败原因
- 非法字段定位
- 具体哪个 face 超时
- ACK 明细
- 表切换时刻
- 降级原因
- 失效计数
- 连续重试统计

也就是：

> **显控看“结论和态势”，日志看“过程和证据”。**

## 11. 时间同步与生效边界

### 11.1 严格同步要求 [已明确]

三个阵面要求严格同步。
这意味着：

- 不能把阵面控制视为完全互不相关的三套系统
- 必须存在统一时间基准
- 必须明确定义控制表“何时生效”

### 11.2 授时体系 [已明确]

当前时基体系按三层语义冻结为：

1. **首选根时基来源：北斗 GNSS**
2. **降级时间来源：PTP / 本地晶振 holdover**
3. **运行时直接触发源：FPGA 分发并上送的软件秒边界事件**

补充约束：

- 时间戳主要用于**记录、追踪、回放与对时**
- 时间戳**不直接触发生效**

### 11.3 调度周期与执行粒度 [暂定已基本收敛]

结合当前结构性约束、时基体系与整体调度思路，当前更合理的宏观冻结建议为：

- **大周期 / Schedule Epoch = 1 秒**
- **表内执行边界 = CPI**
- **波束推进粒度 = Beam / PRT**

也就是说：

- **秒边界**：决定切换哪一张表、进入哪一个全局周期
- **CPI 边界**：决定表内往前推进到哪个执行段
- **PRT / 包级**：承载数据生产、发送、接收与重组

该模型与当前数据平面结构性约束兼容，协议背景见第 7 节。

### 11.4 当前建议的认知框架

建议将以下表述作为当前阶段的统一理解：

> **大周期按 1 秒定义；控制面以软件秒边界事件作为运行时直接节拍输入；秒边界负责换表/换周期，CPI 边界负责表内推进，PRT/包级负责数据生产与接收重组。数据面不直接管理秒边界，而通过共享周期状态与参数快照消费边界语义。**

## 12. 控制流是否闭环：跟踪模式下的动态调整

### 12.1 用户的真实疑问 [已明确]

用户提出一个关键问题：

- 在目标跟踪场景中，是否需要根据前一个波位获取的雷达回波结果，临时调整后续波位安排？

### 12.2 当前合理答案（架构层面）

需要区分工作模式：

- **搜索模式**：默认采用**预生成固定扫描表周期轮转**，不启动逐周期闭环重规划
- **跟踪 / 精跟模式**：允许基于上一周期处理结果生成**下一周期候选安排**
- **混合模式**：跟踪波束优先于搜索补盲波束

统一约束为：

- 执行表按**秒级定版**
- 在**秒边界生效**
- **秒内按表推进**
- **秒内不做任意热更新**

### 12.3 控制闭环的合理路径

- 搜索模式：**固定扫描表轮转**
- 跟踪 / 精跟模式：**完整 CPI 数据单元（带参数快照） → 信号处理 / 数据处理 → 控制建议输入对象 → 候选安排 → 调度中心定版 → 下一周期执行表**

因此：

- 显控是上层任务输入者与观察者
- 服务器控制面才是自动闭环调度执行者

## 13. 服务器内部宏观模块划分（当前建议）

### 13.1 数据面模块（实时链）

建议服务器内部宏观数据面模块至少包括：

1. **接收模块（Receiver）**
   - 网卡收数
   - 基础协议解析
   - 基础合法性校验
   - 包级接收统计
   - 形成**完整 CPI 数据单元**
   - 前移挂载**参数快照**
2. **重组/组织模块**
   - 按约定对包进行组织
   - 建立 PRT / CPI 级上下文
3. **信号处理模块（Signal Processing）**
   - 主算法链
   - GPU 加速部分在此阶段承载
4. **数据处理模块（Data Processing）**
   - 检测/估计/后处理/目标级输出
   - 输出**控制建议输入对象**
5. **网关/USP 模块**
   - 输出到显控或外部系统
6. **原始数据旁路落盘分支**
   - 在 Node1 从接收端尽早分流原始包镜像
   - 用于取证、回放与协议排障

补充约束：

- 数据面不直接管理秒边界
- 数据面通过共享周期状态与参数快照消费边界语义

### 13.2 控制面模块（非实时/较低实时）

建议服务器内部控制面模块至少包括：

1. **显控接口模块**
2. **PPS 接收线程 / 时基管理组件**
3. **模式状态机**
4. **任务管理模块**
5. **控制策略 / 候选安排生成模块**
6. **任务调度中心**
7. **阵面表拆分与下发模块**
8. **ACK / 遥测 / 状态汇聚模块**
9. **日志 / 监控 / 运维模块**

补充说明：

- **PPS 接收线程**负责接收软件秒边界事件、校验、去重与更新周期状态
- 它**不直接等于**权威周期状态机
- 权威周期状态由**时基管理组件**统一维护

### 13.3 角色总结

服务器当前同时扮演两个角色：

#### A. 数据处理节点
承担三阵面原始数据接收与主处理链

#### B. 控制调度节点
承担模式管理、表生成、表下发、状态汇聚

## 14. 原始数据录制策略

### 14.1 落盘位置与时机 [已明确]

当前明确倾向：

- **Node1 接收端旁路 tap**：一拿到 UDP 包就尽快分流**原始包镜像**
- **Node0 异步持久化服务**：负责 NVMe 一级缓冲与后台转存到 `/data`
- 采集 tap 与持久化服务不是同一模块，也不是同一线程职责

### 14.2 落盘层级 [已明确]

当前倾向的录制能力为：

1. **主录制对象**：原始包镜像流，用于取证、回放与协议排障
2. **可选增强录制对象**：重组后的 PRT / CPI 结构化数据，用于算法分析与离线复盘
3. **主基线**：必须保证原始包镜像旁路录制存在，且不得反压主链
4. 使用 **NVMe 作为一级缓存**
5. 后台异步转存到 `/data` 数据盘

约束如下：

- 原始录制不得反压实时主链
- 实时采集发生在数据面入口
- 持久化与转存发生在管理面

## 15. 显控部署与后续扩展需求

### 15.1 当前显控部署形态 [已明确]

当前设计下：

- 显控是**本机软件模块**
- 以**独立进程**方式存在
- 不与信号处理模块合并为一个进程

### 15.2 为什么当前仍保留网关/外部输出思路 [已明确]

虽然当前显控在服务器本机上运行，理论上本地 IPC 也可完成联通，但工程上仍保留网关/对外输出策略，是因为后续有扩展需求：

1. 调试时在本机临时查看
2. 外接外部电脑显示
3. 对接更大的数据中心/服务中心

因此当前更合理的理解是：

- **显控当前是本机独立进程**
- **系统未来具备本地/外接/中心化三种输出模式的扩展可能**
- 因此网关/USP 仍然是合理存在的结构

## 17. 当前建议的系统级认知（可直接供新 AI 对话使用）

下面这段可作为简洁背景说明直接复用：

> 当前雷达系统由 3 个固定阵面组成，每个阵面覆盖约 120°。前端 ADC 采样后的原始数据由前端 FPGA 按双方自定义协议组装为 UDP/IP 数据包，经各自对应的 10G 光口发送到服务器；上下行当前默认共用同一 10G 全双工物理链路。服务器具备 4 口 10G 全双工网卡，其中 3 口分别对应 3 个阵面，1 口预留。服务器为 32 核双 NUMA 架构，Node 1 负责实时数据面（网卡/GPU/接收/信号处理），Node 0 负责非实时管理面（调度、日志、监控、异步转存等）。根时基首选来自北斗 GNSS，降级时间来源为 PTP / 本地晶振 holdover；服务器运行时控制动作以 FPGA 分发并上送的**软件秒边界事件**作为直接节拍输入，时间戳主要用于记录、追踪、回放与对时，不直接触发生效。当前调度大周期按 1 秒定义；秒边界负责全局换表/换周期，CPI 边界负责表内推进，PRT/包级负责数据生产与接收重组。三个阵面共享同一个 GlobalEpoch、同一个全局表版本号与同一个软件秒边界语义，但各自持有独立的 FaceSchedule，形成“逻辑上一套全局计划、物理上三份阵面执行表”的组织方式。前后端正式接口清单冻结为 5 类：上行原始数据、下行调度执行表、管理/应急命令、ACK/遥测/状态、时基边界事件。搜索模式默认采用固定扫描表轮转，跟踪/精跟模式允许基于上一周期结果生成下一周期候选安排，但所有执行表均按“提前下发、边界生效、秒内按表推进”的原则运行。数据面不直接管理秒边界，而通过共享周期状态与参数快照消费边界语义；参数快照由接收模块前移挂载，作为跨数据流与控制流的权威上下文锚点。原始录制主基线为接收端原始包镜像旁路，Node1 负责采集 tap，Node0 负责异步录制与转存；可选增强录制对象为重组后的 PRT/CPI 结构化数据。显控统一面向服务器控制面输入任务意图与人工干预，不直接面向前端；显控应展示表级运行摘要与告警结论，详细 ACK、校验、超时与降级过程应下沉至日志系统。

## 20. 当前最值得继续回答的 6 个实现冻结问题

在当前宏观结论已基本形成后，以下问题更值得继续往“可实现、可联调、可运维”方向细化：

1. 1 秒大周期内，CPI、Beam、PRT 三者的精确定义、编号规则和推进关系如何落字节级/实现级规范？
2. GlobalEpoch、PpsSeq、TargetEffectiveEpoch、CurrentAppliedEpoch、TableVersion、FaceSchedule ID 之间应如何编码映射，并如何支持回放、追踪与回滚？
3. `current / next` 的装载提前量、覆盖规则、过期表丢弃策略与超时续跑窗口应如何设定？
4. RxAck、ApplyAck、错误码、遥测字段应如何统一定义，才能同时服务联调、运行监控与故障定位？
5. 当某一阵面 Apply 失败而其余阵面成功时，系统整体应如何降级、告警、回滚或进入安全态？
6. 显控摘要状态与日志明细之间应如何分层映射，既保证可观测性，又避免界面过载？
