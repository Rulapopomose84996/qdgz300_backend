# 第一部分：TrackData 对外一致性与快照语义

## 1. 一致性等级裁决（V1 冻结）

<a id="ssot-trackdata-snapshot"></a>

### 1.1 一致性定义与选择

**系统一致性模型**：TrackData 采用**帧级一致的全量快照（Frame-Consistent Snapshot）** 语义。

**冻结定义**：

- **帧（Frame）**：单一 `frame_seq` 对应的一次完整 TrackData 发布
- **全量快照**：一帧 TrackData 包含当前时刻**后端对外承诺的全部轨迹状态**（所有 CONFIRMED/COASTING 目标的完整信息）
- **帧级一致**：同一帧内所有 Track 的 `frame_seq`、`data_ts` 属于同一逻辑时刻，不混入不同版本或部分状态

**含义**：显控接收到更大 `frame_seq` 的新帧时，可**直接整帧替换显示状态**，无需对象级合并或增量处理。

### 1.2 允许与禁止项（可验收约束）

**MUST（必须遵守）**：

- **[TS-1.1] MUST**：每一帧 TrackData 必须携带 `frame_seq`（单调递增序号）与 `backend_instance_id`（实例标识），用于显控检测新鲜度与实例切换
- **[TS-1.2] MUST**：同一帧内所有 Track 必须共享相同的 `frame_seq`、`data_ts`、`system_quality_flags`；禁止帧内"半新半旧"现象（例如某些目标用旧状态，某些用新状态）
- **[TS-1.3] MUST**：若某目标在本帧被删除/不输出，必须显式标注 `track_status=DELETED` 并随一帧发送（或在元数据中清单该 track_id，见后续验证方式）；禁止直接"消失"导致显控无法区分"真的消失"与"网络丢包"

**SHOULD（强烈推荐）**：

- **[TS-2.1] SHOULD**：为整帧分配一个 `snapshot_id`（可选，等同 frame_seq 的补充标识，便于审计）

**FORBIDDEN（禁止）**：

- **[TS-3.1] FORBIDDEN**：禁止单一帧中存在"仅部分目标的快照"而未显式标注（若确实需要 partial snapshot，必须在 payload 中明确字段 `is_partial=true` 并携带 `coverage` 描述，详见后续 1.4 节）
- **[TS-3.2] FORBIDDEN**：禁止依赖 UDP 的"流式增量推送"（例如"T1 时刻发送目标 A、T2 时刻增量推送目标 B"）来组合形成完整状态；必须每帧自包含

### 1.3 latest-state-wins 整帧替换规则

**核心规则**：

显控在接收 TrackData 时，应遵循以下比较规则：

```txt
接收新帧 frame_N 时：
  if backend_instance_id(frame_N) != instance_cache:
      清空所有旧轨迹缓存，重新建立映射  // 后端重启

  if frame_seq(frame_N) > frame_seq(cached):
      替换整个显示状态为 frame_N 的全量快照
  else if frame_seq(frame_N) == frame_seq(cached):
      （可选）合并字段（如新增诊断字段），但轨迹集合与状态必须一致
  else:
      丢弃（这是旧帧，网络乱序）
```

**设计意图**：

- 通过**单调的 frame_seq** 与 **instance_id 版本控制**，显控无需复杂的"对象级 merge"逻辑
- 允许 UDP 丢包与乱序，最终收到最新帧后自动恢复一致状态
- 简化显控实现，降低多源融合的边界情况处理

### 1.4 降频与快照部分性的处理

**允许的降频行为**：

系统可以因为负荷、网络、或策略原因**降低发送频率**（例如从 50 Hz 降到 10 Hz），但**每一帧仍必须是全量快照**。

**冻结条款**：

- **[TS-4.1] MUST**：降频时仍发送全量快照；允许 N 个周期内仅发送 1 帧
- **[TS-4.2] MUST**：降频不改变 `frame_seq` 的单调性；若某个周期跳过，frame_seq 仍递增至下一个"应该发送的帧号"（可选：打上"跳过"标记便于离线分析）
- **[TS-4.3] MUST**：降频期间系统仍按内部节拍更新/外推轨迹，但**仅在计划发送的帧时刻**对外输出 TrackData（中间被跳过的周期不对外发帧）；外推与降频必须通过质量标志反映（例如 `OUTPUT_THROTTLED`、`COASTING`/`LOW_CONFIDENCE`）。

**MaxUdpPayloadBytes = 1200B 截断约束（与 FPGA 协议 V3.1 及第二篇 §12.2 对齐）**：

当单帧 TrackData 序列化后**超过 1200 字节**（MaxUdpPayloadBytes，V3.1 冻结值）时，发送端必须执行应用层截断。截断行为必须在 TrackData 帧级质量字段中反映：

- **[TS-4.4] MUST**：截断帧须携带 `is_truncated: bool = true` 标记
- **[TS-4.5] MUST**：截断帧须携带 `drop_reason: enum {OVER_MTU, PRIORITY_SHED, ...}` 标识截断原因
- **[TS-4.6] MUST**：截断帧须携带 `dropped_count: uint32` 表示本帧被丢弃/截断的航迹条目数
- **[TS-4.7] SHOULD**：截断策略应优先保留高优先级航迹（CONFIRMED / weapon-quality），优先丢弃低优先级航迹（TENTATIVE / COASTING）
- **[TS-4.8] MUST**：截断发生时，`system_quality_flags` 须置位 `OUTPUT_THROTTLED`（与降频标志复用同一位）；持续截断应触发 Degraded 告警

**TrackData 输出间隔监控（trackdata_gap_ms）**：

<a id="ssot-trackdata-gap-thresholds"></a>

- **定义**：距离上一帧 TrackData 的输出间隔（单位：毫秒），用于检测 TrackData 中断
- **阈值关键参数**：
  - `T1_trackdata_gap_ms`（Degraded 阈值，V1 冻结 500ms）：用于快速告警与进入降级状态
  - `T2_trackdata_gap_ms`（Fault 阈值，冻结 2s）：作为系统业务失败的最终判据，不可调
- **用途**：区别于第五篇指标中的 `track_missing_count`（帧间丢失跟踪数），二者名字不同、含义不同、阈值体系独立
- **冻结条款**：所有监控指标、告警、故障判定涉及"TrackData 输出间隔"时，统一使用 `trackdata_gap_ms` 命名，避免与"帧间丢失目标数"混淆

**部分快照（Partial Snapshot）的约束**：

若系统架构演化到**必须支持 partial snapshot**（例如流量/显存受限，无法每次输出所有目标），则：

- **[TS-5.1] FORBIDDEN（V1）**：V1 禁止 partial snapshot
- **[TS-5.2] SHOULD（V2+）**：若升级为 V2 支持 partial，必须：
  - 携带 `is_partial=true` 显式标记
  - 携带 `coverage` 字段描述"包含哪些目标范围"（例如按 ID 范围、空间区域、质量阈值筛选）
  - 显控规则：partial 帧**不能直接替换**，需要与缓存的其他部分帧合并，或显式告知用户"当前为部分态势"
  - 增加验证复杂度与错误风险，故 V1 不实现

### 1.5 验收要点与 FAT/SAT 用例

**验收用例 1.1：帧级一致性**

- **前置**：系统稳定运行，TrackData 正常输出
- **动作**：检查任意一帧的全部 Track，验证它们是否拥有相同的 `frame_seq`、`data_ts`、`system_quality_flags`
- **期望**：所有 Track 属于同一逻辑时刻；无"混杂"现象
- **验证方式**：离线分析录制日志，统计帧内字段一致性；100% 应为一致

**验收用例 1.2：整帧替换**

- **前置**：显控缓存帧 N 的状态（例如 3 条轨迹：T1、T2、T3）
- **动作**：模拟：帧 N+1 删除 T3（标注 `DELETED`），新增 T4；网络抖动导致帧 N 重复到达
- **期望**：
  - 显控收到帧 N+1 后，显示状态 → {T1、T2、T4}（T3 消失）
  - 若乱序收到帧 N 重复，直接丢弃（因为 frame_seq ≤ cached）
  - 最终状态一致，无"幽灵目标"或"缺失目标"
- **验证方式**：注入网络延迟/丢包，观察显控状态序列是否符合预期

**验收用例 1.3：降频不破坏一致性**

- **前置**：系统以 50 Hz 输出，现降频到 10 Hz
- **动作**：记录 frame_seq；每帧检查全量快照
- **期望**：
  - frame_seq 单调递增（可能 0→5→10 等跳跃）
  - 每一帧仍是全量快照
  - 显控整帧替换仍正常工作
- **验证方式**：对比帧数与频率，验证 frame_seq 跳跃规律；确保每帧都是完整的

## 2. TrackData 时间体系与时间戳口径（与 A1/A2 对齐）

### 2.1 三时间戳体系（冻结）

<a id="ssot-time-triple"></a>

TrackData 必须携带**三类时间戳**，各司其职，不得混淆：

| 时间戳 | 字段名 | 定义 | 来源 | 用途 |
|---|---|---|---|---|
| **事件时间** | `data_ts` | 雷达采样事件发生的物理时刻（世界时间） | 前端携带或后端推断（权威时间体系） | 显控展示时间轴、回放对齐、融合同步 |
| **接入时间** | `ingest_ts` | 后端系统确认接入完整处理单元的时刻 | 后端接收/重组线程打点 | 诊断网络/接入延迟、端到端性能拆解 |
| **交付时间** | `output_ts` | TrackData 在网关组包完成或发送时的时刻 | 网关线程打点 | 计算 E2E 延迟、评估系统实时性、监控发送侧抖动 |

**时间戳一致性约束**：

- **[TS-6.1] MUST**：`data_ts < ingest_ts < output_ts`（单调递增，用于异常检测）
- **[TS-6.2] MUST**：所有时间戳必须可追溯到同一权威时间根（北斗/PTP 体系），不得使用本地散落的时钟
- **[TS-6.3] MUST**：时间戳精度冻结为**纳秒（ns）级**；允许不同部件有微秒偏差，但必须可溯源与校正

> **与 FPGA 协议 V3.1 的精度衔接说明**：V3.1 冻结 DACS 侧时间戳为 **Unix Epoch 毫秒（ms）**格式（64-bit unsigned）。本系统内部统一采用 ns 表示是为了保留更高精度空间。桥接规则：
> - 后端从 DACS 接收到的 ms 级时间戳，转换为 ns 表示时**低 6 位补零**（`ts_ns = ts_ms × 1_000_000`），并在元数据中标注 `time_resolution_ns = 1_000_000`（即实际有效精度为 ms 级）。
> - 北斗/PTP 直接授时的后端系统时钟可提供 µs ~ ns 级精度，此时 `time_resolution_ns` 可为更小值。
> - `data_ts`（事件时间）若来源于 DACS 侧，其有效精度受限于 V3.1 的 ms 分辨率；`ingest_ts` 和 `output_ts` 由后端本地打点，精度取决于本地时钟。
> - **禁止**因 V3.1 仅提供 ms 精度就将系统时间戳降级为 ms；ns 是统一表示层，具体精度由 `time_resolution_ns` 字段标识。

### 2.2 时间源与质量标记（与 A1 对齐）

<a id="ssot-time-quality"></a>

**权威时间源层级**（冻结）：

```txt
一级（最优）：北斗授时（亚微秒精度）
  ↓ 故障转移
二级（备选）：PTP 硬件时钟（微秒精度）
  ↓ 故障转移
三级（临时）：NTP 自同步（ms 级精度，仅原型验证）
```

**时间质量状态**（必须输出，见后续 3.2）：

系统必须输出 `time_quality` 字段（与第二篇 SEV-1/2 判定绑定）：

| 状态 | 定义 | 后端行为 | TrackData 输出 | 对应 FSM |
|---|---|---|---|---|
| **LOCK** | 时间源已同步，误差 < 阈值 | 正常发送 | 正常输出 | Running |
| **HOLDOVER** | 同步质量下降或守时中 | 告警但续运 | 输出 + 标注 time_quality=HOLDOVER | Running/可进 Degraded |
| **FREERUN** | 时间源丢失，自由运行 | 告警 + 可能降级 | 若超 holdover 窗口→停止；否则输出 + time_quality=FREERUN | Degraded（暂有效）/ Fault（超期） |

> **枚举统一说明**：对齐 V3.1 TimeQuality 原始语义（Q0=LOCK/Q1=HOLDOVER/Q2=FREERUN）。旧枚举映射：OK→LOCK, WARN→HOLDOVER, LOST→FREERUN。

**冻结条款**：

- **[TS-7.1] MUST**：每条 TrackData 必须携带 `time_quality ∈ {LOCK, HOLDOVER, FREERUN}`
- **[TS-7.2] MUST**：当 `time_quality=FREERUN` 且已超过 holdover 允许窗口（详见下节）时，后端**MUST 停止 TrackData 输出**或标注为不可信
- **[TS-7.3] MUST**：时间源同步状态每 100ms 检测一次，触发 SEV-1（FREERUN 超过 T2_trackdata_gap_ms=2s）或 SEV-2（进入 Degraded）

**与 FPGA 协议 V3.1 时间术语的显式映射（冻结）**：

V3.1 冻结了 DACS 侧两个时间相关字段（见 Heartbeat 48B 报文 §5）：`TimeCoordSource`（时间协调源）和 `TimeQuality`（时间质量等级）。本系统必须实现以下映射：

| V3.1 TimeCoordSource | V3.1 值 | 对应本文时间源层级 | `clock_domain` 字段值 |
|---|---|---|---|
| 北斗/GNSS | 0 | 一级（最优，亚微秒） | `BEIDOU` |
| PTP | 1 | 二级（备选，微秒） | `PTP` |
| 本地晶振 | 2 | 三级（临时） | `LOCAL_OSC` |

| V3.1 TimeQuality | V3.1 值 | 对应本文 time_quality | 对应对内诊断级别 | 备注 |
|---|---|---|---|---|
| Q0 (Locked) | 0 | LOCK | level=3 / LOCK | 已同步，正常运行 |
| Q1 (Holdover) | 1 | HOLDOVER | level=2 / SYNC_WARN 或 level=1 / HOLDOVER | 守时中，根据持续时间进一步细分 |
| Q2 (Freerun) | 2 | FREERUN | level=0 / FREE_RUN | 自由运行，持续超阈值触发 SEV-1 |

**`clock_domain` 字段要求**：
- **[TS-7.4] MUST**：每条 TrackData 必须携带 `clock_domain` 字段，标识该帧数据的时钟域来源（映射自 V3.1 Heartbeat 中 `TimeCoordSource`）。
- **[TS-7.5] MUST**：三面阵各自独立上报时间状态；若不同面阵 `clock_domain` 不一致（例如一面阵为 BEIDOU，另一面阵已降级为 LOCAL_OSC），后端须在融合输出中标注 `mixed_clock_domain=true`，供显控做可信度判断。

**对内诊断与对外协议的映射表（重要）**：

系统内部使用更细粒度的 `authoritative_time_state` 或 `time_state_level(0~3)` 进行诊断与故障注入，但对 TrackData（对外协议）统一映射为 `time_quality ∈ {LOCK, HOLDOVER, FREERUN}` 三态。此映射是第五篇注入验收的唯一引用源：

| 对内诊断级别 | 含义 | 对外 time_quality | SEV 级别 | TrackData 输出 |
|---|---|---|---|---|
| level=3 / LOCK | 时间已锁定，精度最高 | LOCK | — | 正常输出 |
| level=2 / SYNC_WARN | 同步质量下降，接近失步 | HOLDOVER | 3 | 输出+标注HOLDOVER，可进Degraded |
| level=1 / HOLDOVER | 守时窗口内，时间可信但精度下降 | FREERUN | 2 | 输出+标注FREERUN，进Degraded |
| level=0 / FREE_RUN / INVALID | 无有效时间源，自由运行或失效 | FREERUN | 1 | 若超holdover窗口→停止输出、Fault |

### 2.3 守时与时间失真的容限（holdover 策略）

<a id="ssot-time-holdover"></a>

**定义**：当权威时间源（北斗/PTP）不可用时，系统可进入**守时（holdover）模式**，依赖本地振荡器继续走时。

**约束与策略**：

- **[TS-8.1] MUST**：守时窗口上限冻结为 **T_holdover_max**（建议 5~10 分钟，可在性能基线阶段确定）
- **[TS-8.2] MUST**：在 holdover 内，系统标注 `time_quality=LOST`，但仍发送 TrackData（用于保持最小可用性）
- **[TS-8.3] MUST**：超过 holdover 窗口后，**必须停止 TrackData 输出**或进入 Fault，防止因时间失真导致虚假态势
- **[TS-8.4] MUST**：时间戳单调性破坏（timestamp_monotonicity_violation_count > 0，即连续两帧 data_ts 回退）**立刻触发 SEV-1**，停止输出，不得继续发送可信数据

**可验收口径**：

- 记录 `time_quality` 状态转移日志与 holdover 持续时间
- 在 LOST 后第 N 分钟（N = T_holdover_max）验证输出停止
- 离线分析：检查所有 TrackData 的 data_ts 单调性，无回退

## 3. TrackData 质量字段最小集与系统状态映射

### 3.1 Snapshot（帧级）质量字段定义（MUST）

每帧 TrackData 必须携带以下帧级字段，用于表达"这一帧整体的系统健康度"：

#### 3.1.1 time_quality（时间质量）

- **类型**：枚举 `{OK, WARN, LOST}`
- **定义**：权威时间源的同步状态
- **来源**：时间子系统
- **检测周期**：100 ms
- **用途**：显控必须可见；回放/融合可按此过滤不可信数据段

#### 3.1.2 system_quality_flags（系统质量位集）

- **类型**：bitmask（例如 uint32）
- **定义**：表达系统当前是否因故障/降级而有可用性下降
- **必须位集**（V1 最小集）：

| 位名 | 含义 | 触发条件 | 清除条件 |
|---|---|---|---|
| `DEGRADED_MODE` | 系统处于 Degraded 状态 | FSM 转移至 Degraded | FSM 回到 Running |
| `INPUT_SENSOR_MISSING` | 至少一个传感器/阵面缺失 | 某阵面连接断开或故障 | 所有阵面恢复正常 |
| `OUTPUT_THROTTLED` | 输出因资源压力而降频 | 主动降频 或 TrackData 发送速率 < 标称值 | 频率恢复至标称 |
| `DATA_DROPPED_HIGH` | 数据丢弃率显著升高 | `drop_rate >= 10%`（TBD，需基线确认） | `drop_rate < 5%` 且持续 5s 以上 |
| `TIME_NOT_LOCKED` | 时间源不同步或 LOST | `time_quality != OK` | `time_quality = OK` 且稳定 |

**冻结条款**：

- **[TS-9.1] MUST**：system_quality_flags 必须在每一帧同步更新，反映当时的系统状态
- **[TS-9.2] FORBIDDEN**：禁止系统处于 Degraded 但未置 `DEGRADED_MODE` 位；禁止置位 `DEGRADED_MODE` 但系统仍为 Running（必须一致）

#### 3.1.3 coverage_mask / sensor_health_mask（传感器覆盖状态）

- **类型**：bitmask（每一位对应一个阵面或传感器）
- **定义**：按阵面/通道标注当前健康度
- **编码建议**（示例，可根据实际阵面数调整）：

```txt
byte 0: Array_0 状态（2 bits）
  00 = OK（正常，完整覆盖）
  01 = DEGRADED（部分功能受损）
  10 = LOST（完全不可用）

byte 1: Array_1 状态（2 bits）
byte 2: Array_2 状态（2 bits）
...
```

- **用途**：显控能直观看到哪个阵面故障；便于诊断与维护
- **冻结条款**：
  - **[TS-10.1] MUST**：若某阵面完全故障，对应位必须为 LOST；允许显控根据此拒绝渲染该阵面的数据
  - **[TS-10.2] SHOULD**：所有可用阵面的 Track 仍应正常输出，仅标注所属覆盖范围的质量标志

### 3.2 Track（目标级）质量字段定义（MUST）

每条轨迹必须携带以下字段，表达"这条轨迹本身的置信度与状态"：

#### 3.2.1 track_flags（轨迹状态位集）

- **类型**：bitmask（例如 uint16）
- **必须位集**（V1 最小集）：

| 位名 | 含义 | 语义 |
|---|---|---|
| `TENTATIVE` | 初生，未确认 | 刚检测到，未连续跟踪，可信度低 |
| `CONFIRMED` | 确认跟踪 | 连续稳定，可信度高 |
| `COASTING` | 短时丢失，外推 | 超 Ndata 帧无新检测，采用运动模型外推位置 |
| `LOW_CONFIDENCE` | 低可信 | 残差大或融合不确定性高 |

**冻结条款**：

- **[TS-11.1] MUST**：任意时刻 track_flags 至少包含一个"主状态位"（TENTATIVE / CONFIRMED / COASTING），表示轨迹的生命周期阶段
- **[TS-11.2] MUST**：TENTATIVE 与 CONFIRMED 互斥；不得同时置位
- **[TS-11.3] SHOULD**：`LOW_CONFIDENCE` 可与 CONFIRMED 或 COASTING 并存

#### 3.2.2 confidence（可信度数值，SHOULD）

- **类型**：浮点 [0, 1] 或整数 [0, 100]（冻结口径）
- **定义**：轨迹状态的数值置信度（综合 SNR、残差、融合权重）
- **用途**：UI 可用于渐变显示（例如颜色/线宽随置信度变化）；上层融合可作为权重
- **冻结条款**：
  - **[TS-12.1] SHOULD**：V1 强烈建议携带；若为空/缺失，视为保留字段，后续版本必冻
  - **[TS-12.2] MUST**：若 `LOW_CONFIDENCE` 置位，confidence 应 < 阈值（建议 0.5）；若 confidence 接近 1，禁止同时置 `LOW_CONFIDENCE`

#### 3.2.3 age / last_update_gap（轨迹年龄与更新间隔，SHOULD）

- **类型**：整数，单位 ms 或帧数
- **定义**：
  - `age`：轨迹从创建到当前的持续时间
  - `last_update_gap`：上一次有新检测数据到现在的间隔
- **用途**：显控解释"为什么这条轨迹在外推"；验证"降频/丢包时的行为"
- **冻结条款**：
  - **[TS-13.1] SHOULD**：V1 建议至少携带其中一个，便于显控与诊断
  - **[TS-13.2] MUST**：若 COASTING 置位，`last_update_gap` 应 > 0；若 CONFIRMED 置位，`last_update_gap` 应接近 0（或在上一帧间隔内）

### 3.3 系统故障事件 → TrackData 字段映射表（核心冻结）

这个表是第二篇**故障分级与降级策略**与第三篇**质量字段**的对齐桥梁。系统任何故障必须通过这张表的映射反映到 TrackData 中：

| 系统事件/状态 | SEV | FSM 状态 | 对应 TrackData 字段变化 | 是否停止输出 | 说明 |
|---|---|---|---|---|---|
| **正常运行** | — | Running | time_quality=OK、system_quality_flags=0、coverage_mask=ALL_OK | 否 | 基线状态 |
| **时间 WARN**（同步质量下降） | 3 | Running | time_quality=WARN、置 TIME_NOT_LOCKED | 否 | 告警但续输 |
| **时间 LOST**（<holdover 窗口） | 2 | Degraded | time_quality=LOST、置 TIME_NOT_LOCKED + DEGRADED_MODE | 否 | 守时期间续输 |
| **时间 LOST**（>holdover 窗口） | 1 | Fault | —— | **是** | **停止输出** |
| **时间戳单调性破坏**（连续≥3次） | 1 | Fault | —— | **是** | **停止输出** |
| **单阵面掉线** | 2 | Degraded | 置 INPUT_SENSOR_MISSING、更新 coverage_mask 该阵面为 LOST、置 DEGRADED_MODE | 否 | 其他阵面输出 |
| **多阵面掉线**（无覆盖） | 1 | Fault | —— | **是** | **停止输出** |
| **Plots 丢弃率高**（≥10%） | 2 | Degraded | 置 DATA_DROPPED_HIGH + DEGRADED_MODE | 否 | 降频处理 |
| **TrackData 中断**（T1_trackdata_gap_ms） | 2 | Degraded | 置 OUTPUT_THROTTLED + DEGRADED_MODE；最后一帧缓存 | 否 | 待恢复 |
| **TrackData 中断**（T2_trackdata_gap_ms≥2s） | 1 | Fault | —— | **是** | **停止输出** |
| **TrackData 发布队列持续满/覆盖** | 2 | Degraded | 置 OUTPUT_THROTTLED + DEGRADED_MODE（必要时置 DATA_DROPPED_HIGH） | 否 | 允许跳帧，仅保留最新快照，禁止反向阻塞数据面 |
| **GPU 队列深度≥90%** | 2 | Degraded | 置 OUTPUT_THROTTLED + DEGRADED_MODE（可能丢 CPI） | 否 | CPI 限流 |
| **GPU 单 CPI 超时/compute 超预算** | 2 | Degraded | 置 OUTPUT_THROTTLED + DEGRADED_MODE（必要时置 DATA_DROPPED_HIGH） | 否 | 超时结果丢弃/不参与更新；相关目标可能进入 COASTING/LOW_CONFIDENCE |
| **内存池高水位**（≥90%） | 2 | Degraded | 置 DATA_DROPPED_HIGH + DEGRADED_MODE | 否 | 丢弃低优先 |
| **DACS TimeQuality Q1（Holdover）** | 3→2 | Running→Degraded | time_quality=WARN、置 TIME_NOT_LOCKED；持续则进 Degraded + DEGRADED_MODE | 否 | 映射自 V3.1 心跳 TimeQuality=1；守时期间续输 |
| **DACS TimeQuality Q2（Freerun）持续≥1s** | 1 | Fault | —— | **是** | 映射自 V3.1 心跳 TimeQuality=2；**停止输出** |
| **DACS Execution Snapshot 不一致**（APPLIED_LATE / OVERRUN） | 2 | Degraded | 置 DEGRADED_MODE；受影响面阵 coverage_mask 标为 DEGRADED | 否 | 控制下发与实际执行参数不一致，需人工排查 |
| **DACS 心跳丢失**（连续≥3 个周期） | 2 | Degraded | 置 INPUT_SENSOR_MISSING、更新 coverage_mask 该阵面为 LOST、置 DEGRADED_MODE | 否 | 与"单阵面掉线"行为对齐；持续超 T2→Fault |
| **TrackData 截断**（单帧超 1200B） | 3 | Running | 置 OUTPUT_THROTTLED；帧字段 `is_truncated=true`、`dropped_count>0` | 否 | 应用层截断，持续触发可进 Degraded |

**冻结条款**：

- **[TS-14.1] MUST**：上表的映射关系为系统必冻固结；任何未在表中明确的故障必须在 FAT/SAT 前补齐映射
- **[TS-14.2] MUST**：所有 SEV-1 故障（时间 INVALID、无覆盖、TrackData 完全中断 ≥2s）**必须停止 TrackData 输出**；禁止以"继续输出过期数据"掩盖可信性问题
- **[TS-14.3] MUST**：SEV-2 故障进入 Degraded，TrackData 仍输出但必须置位对应 quality 标志；显控或上层可基于此进行降级渲染或告警

### 3.4 质量字段验证规则（用于 FAT/SAT）

| 验证项 | 检查方式 | 期望结果 |
|---|---|---|
| **时间质量一致性** | 离线分析：`time_quality=LOST` 出现时，同一帧的 `TIME_NOT_LOCKED` 必置 | 100% 一致，无漏标 |
| **故障事件完整反映** | 注入单阵面故障，验证 `coverage_mask` 与 `INPUT_SENSOR_MISSING` 变化 | 立刻反映，无延迟 |
| **COASTING 标注** | 制造漏检，验证轨迹转为 COASTING 且 `last_update_gap` 递增 | 正确标注与计数 |
| **降频反映** | 降频输出，验证 `OUTPUT_THROTTLED` 置位、`frame_seq` 跳跃但 `data_ts` 单调 | 位置与序列正确 |
| **Degraded 映射** | 触发多个 SEV-2 故障，验证 `DEGRADED_MODE` 置位与 `system_state=Degraded` 同步 | 完全对齐，无滞后 |

# 第二部分：Track 生命周期与 ID/唯一性契约

## 4. Track ID 设计与唯一性保证（与 B2 对齐）

### 4.1 双层 ID 与全局唯一性（冻结）

Track 的身份认证采用**双层 ID 设计**：

| ID 层 | 字段名 | 类型 | 唯一性范围 | 用途 |
|---|---|---|---|---|
| **实例层** | `backend_instance_id` | UUID / 128-bit 随机 | 全球唯一 | 标识后端系统启动实例，用于检测重启 |
| **轨迹层** | `track_id` | uint32 / uint64 | 单实例内永不复用 | 在当前实例内唯一标识轨迹 |
| **全局层** | `track_uid` | 推荐 128-bit | 全球唯一 | 显控对象键，跨实例/会话保持唯一性 |

**补充裁决（P0-1，冻结）**：
- **MUST**：对外 TrackData 中的 `track_id/track_uid` 表示**全局航迹身份**，由 M03（DataProc，全局 TrackManager 子职责）统一分配与维护。
- **FORBIDDEN**：任何阵面内临时 ID/局部航迹对象不得直接映射为对外 `track_id`，不得绕过 M03 输出。

### 4.2 ID 生成与维护规则（冻结）

**backend_instance_id**：

- **[B2-1.1] MUST**：每次后端系统启动，生成新的 `backend_instance_id`（可使用 UUID4 或系统内生成的 128-bit 随机数）
- **[B2-1.2] MUST**：该 ID 包含在每条 TrackData 中，用于显控检测后端重启
- **[B2-1.3] MUST**：重启前清理所有 track_id 分配状态；重启后 track_id 从 1 重新计算

**track_id**：

- **[B2-2.1] MUST**：在轨迹创建时分配，单调递增，永不复用（即使轨迹删除）
- **[B2-2.2] MUST**：运行实例内 track_id 在 TENTATIVE 状态创建后立刻分配，一旦分配**终生不变**，直到轨迹 DELETED
- **[B2-2.3] MUST**：禁止在同一实例内复用已删除轨迹的 track_id；原因是显控需要区分"真的删除"与"轨迹重生"

**track_uid（推荐设计）**：

- **[B2-3.1] MUST**：`track_uid = hash128(backend_instance_id || track_id)` 或结构化拼接 `(instance_prefix | track_id_suffix)`
- **[B2-3.2] MUST**：显控使用 `track_uid` 作为对象键（map key）来维护轨迹缓存，避免跨重启的 ID 碰撞
- **[B2-3.3] MUST**：显控侦测 `backend_instance_id` 变化，立刻清空所有旧 epoch 的 track_uid 映射，进入"新实例"模式

### 4.3 轨迹生命周期状态机（与 B2 对齐，冻结）

系统定义最小状态集合（可使用等价命名）：

```txt
创建
  │
  ▼
TENTATIVE ──── （连续N帧有量测）→ CONFIRMED
  │                                  │
  │                                  ▼
  │  ┌─────────────────────────────→ COASTING ← (短时漏检)
  │  │                                 │
  │  │ (重现 / 重新关联)                 │
  │  │                                  ▼
  │  └──────────────────────── COASTING + hold ── (超 T_coast/N_miss)
  │                                      │
  └──────────────────────────────────────▼
                                      LOST ── (对外不再输出)
                                        │
                                        ▼
                                     DELETED (释放 track_id)
```

**状态转移规则**（冻结）：

| 源状态 | 目标状态 | 触发条件 | 动作 |
|---|---|---|---|
| TENTATIVE | TENTATIVE | 未满足转移条件 | 保持，仅年龄递增 |
| TENTATIVE | CONFIRMED | 连续 M 帧内有量测 | `track_id` 确认，开始对外输出 |
| CONFIRMED | CONFIRMED | 连续有新量测 | 更新位置/速度/状态 |
| CONFIRMED | COASTING | 漏检 1 帧（无新量测） | 启用运动模型外推；标 `COASTING` |
| COASTING | CONFIRMED | 重现（重新关联到同一轨迹） | 恢复更新，清除 COASTING 标志（**ID 保持不变**） |
| COASTING | LOST | 超 `T_coast` 或 `N_miss` 帧 | 轨迹无法再外推，进入 LOST |
| LOST | CONFIRMED 或新 TENTATIVE | 见后续 4.4 节"丢失与重现" | **若重现，按规则创建新 track_id 还是复用旧的** |
| * | DELETED | 主动清理或超时失效 | 释放资源，对外不再输出 |

**冻结条款**：

- **[B2-4.1] MUST**：track_id 在 TENTATIVE 创建时分配，一旦分配**在该轨迹有生之年**永不复用
- **[B2-4.2] MUST**：COASTING 状态下若重现（短时漏检后检测恢复），**原 track_id 保持不变**，继续输出同一轨迹
- **[B2-4.3] MUST**：转移至 LOST 后，若再次检测到可能的"同一物体"，**必须新建 track_id**（作为新轨迹），不得复用

### 4.4 丢失与重现窗口定义（与 B2 对齐）

**定义两个关键参数**（冻结语义，数值 TBD 需性能基线）：

| 参数 | 定义 | 推荐值 | 用途 |
|---|---|---|---|
| `T_coast` | COASTING 状态允许的最长外推时间 | TBD（建议 200~500 ms） | 超过此时间进入 LOST |
| `N_miss` | COASTING 状态允许的最长连续漏检帧数 | TBD（建议 5~10 帧 @10Hz） | 超过此帧数进入 LOST |

**重现规则**（冻结）：

- **[B2-5.1] MUST**：在 COASTING 内（未超 `T_coast`/`N_miss`）重新检测到同一轨迹的量测，**保持原 track_id**，转回 CONFIRMED
- **[B2-5.2] MUST**：超过 `T_coast`/`N_miss` 进入 LOST，若再次出现，**必须新建 track_id**
- **[B2-5.3] SHOULD**：新建轨迹时，填写 `predecessor_uid` 字段指向原轨迹的 `track_uid`，用于显控/回放解释"这条轨迹可能是哪个旧轨迹演化而来"（可选但强烈建议便于 UI 表达）

#### 4.4.1 跨阵面交接（Handover）冻结规则（P0-1）

三阵面覆盖“空间无重叠”并不等价于“目标身份无需连续”。目标穿越阵面边界时，系统必须保证显控侧看到的轨迹身份与时间连续。

**冻结条款**：

- **[B2-5.4] MUST**：跨阵面边界导致的量测短暂缺失，必须优先使用 COASTING 窗口（`T_coast`/`N_miss`）维持全局航迹对象存活，避免在边界处产生“消失→新建”的对外断裂。
- **[B2-5.5] MUST**：当相邻阵面在 COASTING 窗口内重新提供可关联量测时，必须按“重现/重新关联”处理：**保持原 track_id/track_uid 不变**，并转回 CONFIRMED。
- **[B2-5.6] FORBIDDEN**：在 COASTING 窗口内，若新量测可与某全局 COASTING 航迹关联，禁止为该量测新建对外 track_id。
- **[B2-5.7] SHOULD**：交接不确定（存在多候选或残差异常）时，允许置位 `LOW_CONFIDENCE`，但不得输出两条互相冲突的“并存重复航迹”来掩盖身份裁决缺失。

**验证要点**：

- **验收用例 4.1：短时重现保 ID**
  - 前置：轨迹 T1 处于 CONFIRMED
  - 动作：漏检 100ms（<T_coast），再检测恢复
  - 期望：仍为 T1 的 track_id；未产生新 ID；无"轨迹消失又出现"现象
  - 验证：track_id 与 track_uid 保持不变

- **验收用例 4.2：长时丢失新 ID**
  - 前置：轨迹 T1 漏检 5s（>T_coast），进入 LOST 并对外停止输出
  - 动作：再次检测到同一物体（空间位置连续）
  - 期望：分配新 track_id，作为新轨迹 T1'；`predecessor_uid=T1.track_uid`
  - 验证：track_id 递增；predecessor_uid 非空；显控可识别"来源关系"

### 4.5 合并（Merge）与拆分（Split）规则（与 B2 对齐）

**合并规则**：

当多个轨迹融合为一个输出轨迹时：

- **[B2-6.1] MUST**：选择一个 survivor，继承其 `track_id/track_uid`
- **[B2-6.2] SHOULD**：survivor 选择优先级（按顺序）：
  1. 质量更高（confidence/SNR）
  2. 年龄更长（age 更大）
  3. track_id 更小（稳定性偏好）
- **[B2-6.3] MUST**：被合并轨迹在本帧后**对外停止输出**（转为 DELETED）
- **[B2-6.4] MUST**：survivor 填写 `merged_from_uids[]` 列表，记录所有被合并轨迹的 track_uid

**拆分规则**：

当一个轨迹拆分成多个输出轨迹时：

- **[B2-7.1] MUST**：主分支（质量最高或与历史最一致）继承原 `track_id/track_uid`
- **[B2-7.2] MUST**：其他分支创建新 `track_id/track_uid`
- **[B2-7.3] MUST**：新分支填写 `parent_uid = 原轨迹的 track_uid`，用于显控解释分裂关系

**冻结条款**：

- **[B2-8.1] MUST**：merge/split 事件必须在 `merged_from_uids[]` 或 `parent_uid` 字段显式记录；禁止隐式发生导致显控无法解释
- **[B2-8.2] MUST**：被合并轨迹的 track_id 不复用；新建分支的 track_id 继续单调递增

**验证要点**：

- **验收用例 4.3：merge 与 split**
  - 模拟两条接近的轨迹靠近融合，验证 survivor 的选择规则与 merged_from_uids 的正确性
  - 模拟一条轨迹分裂，验证主分支 ID 保留、分支 ID 递增、parent_uid 非空

## 5. 重启 Epoch 与跨重启契约

**定义**：系统重启产生新的 `backend_instance_id`，标记一个新的"epoch"。跨 epoch 不承诺 ID 连续性。

**冻结条款**：

- **[B2-9.1] MUST**：重启前后的 `backend_instance_id` 变化是显控侦测重启的可靠信号；显控应立刻清空旧 epoch 的轨迹映射与缓存
- **[B2-9.2] MUST**：重启后新轨迹的 track_id 从 1 重新开始；禁止跨重启连续编号
- **[B2-9.3] MUST**：显控对"重启前轨迹 T1 与重启后轨迹 T1"不得默认关联；若需要跨重启关联，必须由上层应用逻辑（例如通过空间或时间匹配）显式建立

# 第三部分：配置分层与生效边界

## 6. 配置参数分级（Static / Dynamic，与 C1 对齐）

### 6.1 分级定义与准入状态（冻结）

**配置层级优先级规则（SSOT 对齐 00_顶层架构.md §2.8）**：

```
优先级（高→低）：
1. FROZEN 参数（绝对不可改，架构红线）
2. Static 参数（仅 Init/Standby 可改）
3. Dynamic 参数（运行期可改，受 effective_boundary 约束）
```

> **来源**：配置分级规则统一定义于 [00_顶层架构.md](00_顶层架构.md) §2.8，本节为实现约束展开。

**Static 参数**：

- **定义**：改变它会引入重新分配、重建资源或破坏热路径确定性
- **准入状态**：仅能在 `Init` / `Standby` 被装载或修改
- **限制**：系统处于 `Running` / `Degraded` / `Fault` 时，**禁止通过控制面修改 Static 参数**
- **生效方式**：需要经过"受控重启"路径：`Shutdown → Init → Standby → Running`

**冻结条款**：

- **[C1-1.1] MUST**：Static 参数改变必须经过完整 Init 初始化链路
- **[C1-1.2] FORBIDDEN**：任何 Static 改参在运行期采用"热改"或"下一帧生效"，视为架构违规

**Dynamic 参数**：

- **定义**：改动不引入重新分配/重建；有明确、可验证的生效边界
- **准入状态**：可在任意系统状态下通过控制面修改
- **生效方式**：受 `effective_boundary` 约束（见下节 6.2）

**冻结条款**：

- **[C1-2.1] MUST**：每个 Dynamic 参数必须显式标注 `effective_boundary`，否则按 Static 处理
- **[C1-2.2] MUST**：Dynamic 改参仅通过 ControlRequest 进行，遵循控制面可靠性与幂等框架

### 6.2 Static 参数最小范围（冻结）

以下参数**必冻**为 Static（运行期不可改）：

| 参数类别 | 典型例子 | 冻结原因 |
|---|---|---|
| **内存池预分配规模** | pinned_host_pool_size、device_buffer_size、track_pool_capacity | 改了需要重新分配，影响热路径性能与内存一致性 |
| **有界队列容量** | 各 SPSC ringbuffer 容量、MPSC 控制队列容量 | 改了需要重新初始化数据结构，影响流控 |
| **GPU 并行度** | streams_per_array、max_inflight_per_stream | 改了需要重新创建 CUDA stream，影响内核调度 |
| **CPU 绑定与 NUMA 亲和** | core_affinity、numa_node_select | 改了需要线程迁移，影响内存延迟与确定性 |
| **热路径队列水位阈值** | HWM、FWM（High/Full Watermark） | V1 采用固定策略，不动态自适应 |

**冻结条款**：

- **[C1-3.1] MUST**：上述参数的完整清单应在性能基线阶段补齐与量化，但**必须在 FAT 前冻结所有值**

### 6.3 Dynamic 参数与生效边界（effective_boundary，冻结）

<a id="ssot-effective-boundary"></a>

系统定义**三类生效边界**，以避免"半帧/半 CPI"的中间态：

#### 6.3.1 NEXT_CPI（推荐默认，用于热路径参数）

**语义**：改参不影响当前正在处理的 CPI，在**下一 CPI 开始前一次性切换**

**适用范围**：
- 接收/拼包策略开关
- GPU 调度轻量阈值（不涉及重建资源池）
- 某些降频/抽样策略
- 信号处理通道级参数

**实现要求**：

- **[C1-4.1] MUST**：标注为 `NEXT_CPI` 的参数须实现"待生效版本（pending）→ CPI 边界原子切换"
- **[C1-4.2] FORBIDDEN**：禁止在 CPI 处理中途逐字段生效，会产生不可复现的输出差异

**验证要点**：

- **验收用例 6.1：NEXT_CPI 参数生效**
  - 前置：记录当前 CPI_seq
  - 动作：发送参数修改请求（effective_boundary=NEXT_CPI）
  - 期望：ACK 返回 `effective_at=CPI_seq+1`；当前 CPI 输出不受影响；下一 CPI 起行为变化
  - 验证：逐 CPI 检查输出一致性，无"半 CPI"现象

#### 6.3.2 NEXT_FRAME（用于 TrackData 侧，与帧级一致性绑定）

**语义**：改参在 `frame_seq` 边界整帧生效

**适用范围**：
- TrackData 输出节流/降频策略
- 轨迹字段开关（例如是否输出某个可选字段）
- 质量标志阈值（例如 confidence 显示阈值）

**实现要求**：

- **[C1-5.1] MUST**：标注为 `NEXT_FRAME` 的参数须保证"帧级一致"；整帧切换，无帧内混杂
- **[C1-5.2] FORBIDDEN**：禁止某些目标用新参数、某些目标用旧参数；必须帧内一致

**验证要点**：

- **验收用例 6.2：NEXT_FRAME 参数生效**
  - 前置：记录当前 frame_seq=N
  - 动作：修改 TrackData 降频策略参数
  - 期望：frame_seq=N 保持旧参数；frame_seq=N+1 起新参数；无"帧内混杂"现象
  - 验证：逐帧检查，确保整帧切换

#### 6.3.3 IMMEDIATE（仅限非热路径/观测类）

**语义**：立即生效，不等待 CPI/Frame 边界

**适用范围**：
- 日志级别
- 指标采样周期
- 诊断开关（黑匣子触发）

**限制**：

- **[C1-6.1] MUST**：`IMMEDIATE` 仅允许不改变热路径分支/资源使用的参数
- **[C1-6.2] FORBIDDEN**：任何会影响热路径确定性的参数禁止标注为 IMMEDIATE

### 6.4 回滚与可复现性记录（与验收绑定）

> **V1 策略说明**：根据 [00_顶层架构.md](00_顶层架构.md) §2.8 决策，**V1 不支持运行期自动回滚**。回滚需通过 `Shutdown → Init` 重新加载目标 config_revision。V2+ 可扩展在线回滚能力。

**回滚规则**：

- **[C1-7.1] MUST**：每次成功改参产生新的 `config_revision`（单调递增，实例内唯一），在 ACK 中返回
- **[C1-7.2] MUST**：后端支持"回滚到指定 config_revision"的控制命令；回滚的 effective_boundary 必须与参数一致
- **[C1-7.3] SHOULD**：支持自动回滚（若改参导致关键指标越界），此为可选策略，V1 可先冻结接口，不强制自动化

**可复现性记录**：

- **[C1-8.1] MUST**：每条 ParamUpdate 必须被审计记录：
  ```
  {
    command_id,
    issuer,
    old_value,
    new_value,
    effective_boundary,
    effective_at (CPI_seq / frame_seq),
    config_revision,
    timestamp
  }
  ```
- **[C1-8.2] MUST**：审计日志必须与运行清单（run manifest，见后续 7 节）关联，保证"同一份数据能还原当时的配置状态"

### 6.5 冻结条款汇总

- **[C1-9.1] MUST**：Static 参数改变仅在 Init/Standby 期间，系统进入 Running/Degraded/Fault 后禁止修改；需要经过受控重启生效
- **[C1-9.2] MUST**：Dynamic 参数必须显式标注 effective_boundary ∈ {NEXT_CPI, NEXT_FRAME, IMMEDIATE}
- **[C1-9.3] FORBIDDEN**：禁止在 Fault 状态下修改功能性参数（仅允许诊断/取证类参数）
- **[C1-9.4] MUST**：所有参数修改必须返回新的 config_revision，且可回滚追溯
