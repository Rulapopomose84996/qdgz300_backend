## 第三部分：故障处理与系统恢复

### 9. 故障分级与降级策略（A1- 核心裁决）

<a id="ssot-sev-def"></a>

系统故障并非非黑即白，而是采用**三级分级模型**，与第一篇的 " 可恢复故障 " 概念保持一致。

#### 9.1 故障分类体系

##### SEV-1：可信性故障（业务致命）

**定义**：输出可信性被破坏或无法保证，必须立刻停止对外输出或进入 Fault 状态。

**可操作判定最小集（冻结）**：以下任一条件触发 SEV-1：

| 判定条件 | 检测信号 | 检测周期 | 阈值字段 | 触发动作 |
|---|---|---|---|---|
| **权威时间源不可用** | `authoritative_time_state ∈ {LOST, SYNC_LOST}` | 每 100ms | 持续 ≥ 1s | 进入 Fault；停止 TrackData 输出 |
| **时间戳失真** | `timestamp_monotonicity_violation_count > 0` | 每帧 | - | 单次失反即记录；连续 ≥ 3 次则进入 Fault |
| **数据完整性检验失败** | `crc_fail_rate` 连续 N 帧 | 每帧统计 | 失败率 ≥ threshold（建议 5%） | 触发 Fault；详细记录失败帧供诊断 |
| **关键状态一致性破坏** | State Machine 约束违反（例如时间戳回退、Track ID 重复、跨阵面身份裁决失效导致对外出现重复并存/无解释跳变） | 运行期实时 | 任何一次 | 立刻进入 Fault |

**说明**：
- 这些条件必须在监控模块中实现，监控模块不依赖算法细节即可判定
- 所有相关计数与标志必须实时可观测（见后续 19. 验收准则）
- 阈值字段需在性能基线与兼容性测试中确定，但**必须冻结所有字段与检测周期**

**典型场景**：
- 权威时间源（GPS/PTP）掉线，系统切换到守时模式（NTP/本地时钟），需明确判定何时 " 无法守时 " 而触发 SEV-1
- 数据通道出现 bit flip 或校验异常，连续 N 帧失败表明系统链路故障，应停止信任数据

**处理原则**：
- **MUST**：此类故障触发 Fault 状态或停止对外输出
- **MUST**：禁止通过 " 继续输出过期 TrackData" 来掩盖可信性问题
- **MUST**：必须明确告警并记录根因用于事后诊断

**与 TrackData 质量字段的映射（重要）**：
- `authoritative_time_state ∈ {LOST, SYNC_LOST}` 对应 TrackData 的 `time_quality=LOST`
- 若持续超 holdover 窗口仍无恢复，触发 SEV-1 且**停止 TrackData 输出**
- 时间戳单调性破坏立刻触发 SEV-1，必须停止输出以防虚假态势
- 此映射是第三篇"故障事件→TrackData字段映射表"的反向参考，确保判定链路闭环可追溯

**与 FPGA 协议 V3.1 时间质量等级的衔接**：

V3.1 冻结了 DACS 侧的时间质量分级：`Q0=Locked`（已同步）、`Q1=Holdover`（守时）、`Q2=Freerun`（自由运行），以及时间协调源 `TimeCoordSource`：0=北斗/GNSS、1=PTP、2=本地晶振。后端须将 DACS 上报的时间质量**桥接**至本系统的时间状态判定：

| DACS TimeQuality | 对应后端 authoritative_time_state | 对应 TrackData time_quality | 行为 |
|---|---|---|---|
| **Q0 (Locked)** | OK | OK | 正常运行 |
| **Q1 (Holdover)** | SYNC_WARN | WARN | 告警 + 启动 holdover 计时；超 holdover 窗口升级为 Q2 处理 |
| **Q2 (Freerun)** | SYNC_LOST | LOST → 若持续 ≥ 1s 触发 SEV-1 | 若 V3.1 心跳连续报告 Q2 超 holdover_timeout → 进入 Fault |

**要求**：
- 后端须从 V3.1 Heartbeat（48B）中提取 `TimeCoordSource` 与 `TimeQuality` 字段并映射到上述表格。
- 三面阵各自上报独立时间质量；**任一面阵** Q2 持续超阈值即触发 SEV-1（保守策略）。

##### SEV-2：可用性故障（降级输出）

**定义**：系统可用性下降但仍可输出 " 可信的部分态势感知 "。

**典型场景**：
- 单阵面接收掉线（另两个阵面仍可输出有效航迹）
- Plots 丢弃率持续升高（但通过降频或合并仍能维持 Track 输出）
- 处理延迟 P99 超过阈值（影响交互响应但不破坏输出正确性）
- 内存池高水位或队列持续溢出

**处理原则**：
- **MUST**：进入 Degraded 运行态，并启动 " 降级动作集合 "
- **SHOULD**：降级动作只限：丢弃低价值数据、降频输出、latest-state-wins 合并更新
- **MUST**：继续输出 TrackData，允许降频但保持状态连续性
- **MUST**：向上报告 Degraded 事件与降级原因

##### SEV-3：瞬时抖动（告警但无需改变运行态）

**定义**：短暂性的性能抖动，允许自然恢复而无需强制降级。

**典型场景**：
- 网络数据包瞬时丢弃
- 处理队列短期溢出但很快消化
- 单帧的 Plots 数量正常变化
- 磁盘写入延迟瞬时升高

**处理原则**：
- 允许短期忍耐（通过缓冲/队列吸收）
- 记录性能指标用于离线分析
- 仅当持续触发 SEV-2 条件时才进入 Degraded

#### 9.2 故障响应三段论

**第一响应（毫秒级 - 瞬时恢复）**：
- 不阻塞任何热路径
- 按优先级丢弃/覆盖旧数据，让系统保持 " 最新态势 "
  - Raw：drop-oldest；Plots：drop-lowest-snr 优先（依据第一篇丢数据优先级与 10.2 冻结条款）
  - TrackData：latest-state-wins（合并更新、允许降频）
- 计数并记录丢弃事件

**第二响应（秒级 - 进入 Degraded）**：
- 进入 `Degraded` 运行状态
- 启动降级动作集合（降频、减少计算量、提高丢弃阈值）
- 向外部系统通报 StateChange 事件
- 触发条件示例：
  - 持续超 T1_water 或 T1_trackdata_gap_ms 的队列溢出或高水位
  - 单阵面故障导致覆盖空域下降但仍可输出
  - 处理延迟 P99 持续超过告警阈值

**第三响应（持续 - 进入 Fault）**：
- 当触及 " 业务最低可用标准 " 或 " 可信性红线 " 时进入 `Fault`
- 触发条件示例：
  - ≥ T2_trackdata_gap_ms 无 TrackData 输出（彻底失去态势）
  - 多个关键子系统级联故障
  - 可信性验证失败（SEV-1）
- 进入 Fault 后：停止对外输出，启动故障告警流程

#### 9.3 故障响应阈值统一表（冻结）

<a id="ssot-sev-thresholds"></a>

为避免 "Degraded 还是 Fault" 的歧义，以下阈值表统一所有故障检测的转移条件：

| 指标 | SEV-3 告警阈值 | SEV-2 Degraded 阈值 | SEV-1 Fault 阈值 |
|---|---|---|---|
| **TrackData 输出中断** | N/A | T1_trackdata_gap_ms（V1 冻结默认 500ms） | T2_trackdata_gap_ms（冻结 2s） |
| **队列溢出频率** | 偶发（< 1 次/min） | 持续（≥ 1 次/10s） | 极频繁（≥ 1 次/s）+ 丢弃显著 |
| **内存池高水位** | ≥ 80% | ≥ 90% 持续 T1_water | TrackPool/ControlPool 接近或达 100% |
| **时间源同步状态** | SYNC_WARN（接近失步） | SYNC_LOST（≥ 1s 无同步） | SYNC_LOST + 时间戳失真（SEV-1） |
| **处理延迟 P99** | 工程目标 + 50%（告警） | 工程目标 + 100%（降级） | 超 X ms 导致 TrackData 不可用 |

**说明**：
- **T1_trackdata_gap_ms（Degraded 阈值，V1 冻结默认 500ms）**：
  - 用于快速告警与进入降级状态，允许系统在短暂中断时立刻响应；
  - 理由：500ms 足够检测到单次队列溢出或 GPU backlog，同时不会因正常抖动（< 100ms）误判；
  - 此值显著小于 T2（2s），为后续恢复/人工干预留出时间窗口。
- **T2_trackdata_gap_ms（Fault 阈值，冻结 2s）**：不可调，作为系统业务失败的最终判据。
- 其他阈值字段（TBD）需在性能基线测试中确定，但**必须在正式测试前冻结所有字段**

#### 9.4 故障转移状态矩阵（冻结）

这个矩阵定义了系统在各种事件触发时的确定性转移行为，避免联调期反复争论：

**说明**：system_state 对外冻结为 5 态：`Init / Standby / Running / Degraded / Fault`。状态定义与转移规则见 [分块设计文档/顶层设计_1.md](分块设计文档/顶层设计_1.md#ssot-fsm-5state)。`Shutdown` 不属于 system_state，而属于 **lifecycle 事件**（进程生命周期控制），由关闭命令驱动，不上报为对外状态（原文档中的 `Normal` 状态在此统一更名为 `Running`，语义不变）。

| 当前状态 | 触发事件 | 判定条件 | 转移目标 | 对外行为 |
|---|---|---|---|---|
| **Init** | 启动完成 | 所有子模块就绪 + 心跳 OK | Standby | 等待启动命令 |
| **Standby** | 启动命令（Start Command） | 显控连接 + Hello + 协议版本兼容（前置就绪检查） | Running | 准备接收控制 |
| **Standby** | 内部故障（SEV-1） | 时间源丢失 / 校验失败 | Fault | 保留上次快照，停止新输出 |
| **Running** | TrackData 持续输出 | 更新率 ≥ 1Hz | Running | 正常态势输出 |
| **Running** | 队列溢出 OR 内存高水位 | 持续超 T1_water 或 T1_trackdata_gap_ms | Degraded | 告警 + 启动降级 |
| **Running** | SEV-1 事件 | 时间源丢失 / 数据校验失败 | Fault | 停止对外输出 |
| **Degraded** | 恢复正常 | 队列恢复 + 无新告警 ≥ T_recovery | Running | 恢复正常输出 |
| **Degraded** | TrackData 中断 | 连续 ≥ T2_trackdata_gap_ms | Fault | 停止输出，启动告警 |
| **Degraded** | SEV-1 事件 | 可信性破坏 | Fault | 停止输出 |
| **Fault** | 手动恢复命令 | 收到 RECOVER 控制请求 | Init | 需要重启后方可恢复 |
| **Fault** | 自动重启（v2+ 可选） | TBD | Init | TBD（v1.0 暂不支持） |
| **Running** | DACS TimeQuality=Q1 | 任一面阵心跳报告 Q1（Holdover） | Degraded | 告警 + 标记受影响面阵 time_quality=WARN |
| **Running** | DACS TimeQuality=Q2 持续≥1s | 任一面阵心跳报告 Q2（Freerun）持续超阈值 | Fault | 触发 SEV-1，停止 TrackData 输出 |
| **Running** | DACS 心跳丢失 | 连续 ≥3 个心跳周期（V3.1 默认 1s/次）未收到 Heartbeat | Degraded | 告警；若持续超 T2 → Fault |
| **Degraded** | DACS Q1→Q0 恢复 | 所有面阵 TimeQuality 恢复 Q0 且持续 ≥ T_recovery | Running | 恢复正常输出 |
| **Degraded** | DACS Q2 升级 | 任一面阵 Q2 持续≥1s | Fault | 触发 SEV-1 |

#### 9.5 TCP 断链保护与自动重连（冻结）

**LinkDown 的定位（Ticket-03 澄清）**：
- `LinkDown` **不属于系统 FSM 状态**（system_state 仍为 Running/Degraded/Fault 等）；
- 它是**连接子状态** `conn_state ∈ {Up, Down}`，用于标记 TCP 控制链路是否可用；
- TCP 断链时 `conn_state=Down`，触发断链保护策略（见下表），但 `system_state` 不因断链而自动降级。

当 TCP 控制链路断开时（`conn_state=Down`），系统按当前 `system_state` 执行保护行为：

| 当前 system_state | 检测到断链（心跳失败 N 次） | 立刻行为 | 缓存策略 | 重连恢复（conn_state → Up） |
|---|---|---|---|---|
| **Running** | `conn_state=Down` | 告警但**继续本地处理** | 缓存最新 TrackData（latest-wins）到本地快照 | 重连 → 发送 StateSnapshot → 显控同步 |
| **Degraded** | `conn_state=Down` | 告警但**继续本地处理** | 缓存最新 TrackData（latest-wins）到本地快照 | 重连 → 发送 StateSnapshot → 显控同步 |
| **Fault** | `conn_state=Down` | 告警 + **停止 TrackData 输出**（或保留最后快照） | 不再更新 TrackData | 重连 → 发送 StateSnapshot → 需要人工/自动恢复 |

**说明**：
- 断链（conn_state=Down）不自动触发 system_state 降级或故障转移（除非已处于 Fault）；
- Running/Degraded 断链时，后端**继续生成 TrackData** 并缓存于本地（latest-wins），以抵抗网络抖动；
- **断链期间 UDP 发送策略**（Ticket-05 细化）：
  - 数据处理链继续生成 TrackData；
  - 网关侧保留最新一帧快照 `snapshot_cache` 用于：(1) 重连后立即下发；(2) 本地诊断/录制；
  - UDP 发送行为：若无 active peer（TCP 未连接），可选择停止 sendto 或继续向最后已知地址发送（实现可选，但必须保留快照）；
  - **背压裁决（P0-2，冻结）**：无论是否断链，只要发送侧变慢（socket 阻塞/网口拥塞/下游不可达），都**不得**反向阻塞数据面热路径；网关必须采用“仅保留最新快照”的有界策略（新快照覆盖旧快照），允许跳过中间帧，并在输出质量字段中反映（第三篇 `OUTPUT_THROTTLED`，持续触发则进入 Degraded）。
- UDP TrackData 基于 `instance_id` 和 `frame_seq` 的自洽设计，使得断链重连后显控能自动检测并处理（无需显式重连流程即可恢复）。

#### 9.6 冻结条款

- **[A-1.1] MUST**：建立 "≥ T2_trackdata_gap_ms（冻结 2s）无 TrackData 输出" 为 SEV-1 Fault 判据；同时建立 " 超 T1_trackdata_gap_ms（冻结 500ms）无输出" 为 SEV-2 Degraded 判据；后者不得用 " 继续输出 " 掩盖，必须进入 Fault 或停止对外输出。
- **[A-1.2] MUST**：`Degraded` 状态的定义以 " 仍能输出可信的部分态势感知 " 为判据，而非 " 所有性能指标达标 "。
- **[A-1.3] SHOULD**：V1 的降级动作仅允许：丢弃低优先级数据、降频、latest-state-wins 合并更新，**禁止**复杂的自愈编排（后续版本再升级）。
- **[A-1.4] SHOULD**：建立 "≥ 2s 无 TrackData 输出 " 作为 Degraded→Fault 的统一转换阈值。

### 10. 内存池耗尽与优先级丢弃（A3- 内存管理）

内存池是系统的关键资源，耗尽处理直接影响实时性与可用性。

#### 10.1 内存池分类与角色

系统分为多个独立内存池，按优先级排列：

| 池名 | 用途 | 所有者 | 容量约束 | 耗尽行为 |
|---|---|---|---|---|
| **RawPool** | 原始采样数据块（IQ） | Receiver | 预分配 | drop-oldest Raw |
| **PlotPool** | 检测点迹 | SignalProc | 预分配 | drop-lowest-snr（优先）；fallback：drop-oldest |
| **TrackPool** | 航迹对象 | DataProc | 预分配 | 严格监控，触发告警 |
| **ControlPool** | 控制事件 | Orchestrator | 预分配 | 严格监控，触发告警 |

#### 10.2 耗尽响应机制

**RawPool 耗尽**：
- 意味着接收速率 > 处理速率（吞吐不足或下游阻塞）
- 立刻采用 drop-oldest 策略，丢弃最老的未处理 Raw Block
- 递增 `raw_pool_drop_count` 并立刻上报
- 继续接收，让系统保持 " 最新采样 "

**PlotPool 耗尽**：
- 意味着检测速率 > 融合速率
- 优先采用 drop-lowest-snr 策略（基于 SNR 或置信度），优先丢弃低质量 Plots
- 若无法获取 SNR/置信度信息，则退化为 drop-oldest（Fallback 机制）
- 递增 `plot_pool_drop_count` 并记录丢弃策略（SNR-based vs fallback）
- 保证 Track 有足够输入继续生成

**TrackPool / ControlPool 耗尽**：
- 此类耗尽表明业务关键资源已到极限
- **MUST 触发 Degraded 或 Fault**，并启动告警通知机制
- 禁止丢弃 TrackData 或 ControlEvent
- 需要显式的人工干预或系统重启恢复

#### 10.3 冻结条款

- **[A-3.1] MUST**：任何内存池耗尽**不得反向阻塞**热路径（尤其 Receiver）；只能通过丢弃低优先级数据来恢复。
- **[A-3.2] MUST**：RawPool 耗尽 → 立即 drop-oldest Raw；同时递增丢弃计数并触发降级告警。
- **[A-3.3] MUST**：PlotPool 耗尽优先采用 drop-lowest-snr 策略；若无法计算价值则退化为 drop-oldest；但必须保证 Track 仍有足够输入生成。
- **[A-3.4] MUST**：TrackPool / ControlPool 耗尽 → 触发 Degraded 或 Fault，启动告警流程，禁止丢弃业务必达数据。
- **[A-3.5] SHOULD**：所有池的耗尽事件必须计数记录用于诊断；丢弃计数与高水位告警需要实时可观测。

### 11. 录制策略与热路径隔离（A2- 非阻塞录制）

Raw IQ 数据录制是系统可复现性的保证，但录制链路不得阻塞核心实时处理。

#### 11.1 录制的业务定位

- **属性**：调试/可复现性支持，不是实时链路交付必达
- **优先级**：最低，资源不足时完全允许牺牲录制完整性
- **定位**：旁路 best-effort，独立队列、独立线程、独立 CPU 亲和

#### 11.2 录制拥塞处理

**拥塞来源分类**：
- 磁盘带宽瞬时不足/写放大/RAID 抖动
- Recorder 线程消费能力不足（绑核/NUMA 访存/缓冲不足）
- 存储空间策略（循环覆写达上限）

**拥塞时的响应**：
- **MUST**：采用 `drop-oldest` 或循环覆写（ring buffer on disk），**不得阻塞 Receiver**
- **MUST**：录制丢弃必须计数并上报诊断信号，但不视为故障
- 允许存储空间满后切换文件或循环覆写

#### 11.3 冻结条款

- **[A-2.1] MUST**：Recorder 与主链路完全解耦（独立 SPSC、独立线程、独立 CPU 亲和），避免共享锁与共享资源竞争。
- **[A-2.2] MUST**：录制队列满时采用 `drop-oldest` 或循环覆写（ring buffer on disk），**绝不能阻塞 Receiver**。
- **[A-2.3] MUST**：录制丢弃必须计数上报诊断，用于离线分析；但**录制完整性不是失效判据**。
- **[A-2.4] FORBIDDEN**：任何情况下不得以 " 保证录制完整性 " 为借口阻塞或降低实时数据处理速率。

---

#### 11.4 控制事件的可靠性与死锁防护（新增，对应 Ticket-06）

控制事件必须确保 " 绝对必达 " 与 " 幂等执行 "，但实现中需要避免优先级反转和死锁风险。

**关键原则**：

1. **发送线程与 ACK 线程解耦**
   - 发送线程**绝不得**持有任何会被接收侧/ACK 侧依赖的锁或资源
   - ACK 必须在独立线程或独立上下文中返回（不得等待发送线程释放资源）
   - 若发送阻塞（队列满），阻塞的线程**必须非热路径线程**（例如仅 Orchestrator 控制线程允许阻塞）

2. **热路径线程禁止阻塞等 ACK**
   - Receiver / SignalProc / DataProc 等热路径线程**禁止**同步等待 ControlEvent 的 ACK
   - 控制事件的下发应采用异步模式（发送后立刻返回）或在 Orchestrator 控制线程中完成（Orchestrator 允许较短的阻塞以保证可靠性）

3. **ACK 必须明确的通知机制**
   - 接收侧处理完控制事件后，**必须**通过独立的 ACK 队列或事件通知发送线程
   - ACK 队列满时**绝不阻塞**接收侧；必须 drop-oldest ACK 并计数（虽然不建议，但比阻塞热路径更安全）

4. **幂等缓存的超时清理**
   - 基于 `command_id` 的幂等缓存需要定期清理（避免内存泄漏）
   - 超时策略：**ACK 返回后 + N 秒**（建议 60s）后可清理该 `command_id` 的缓存

5. **控制面不可死锁工程约束（Ticket-04 补充）**
   - **MUST**：等待 ACK/NACK 或模块完成确认必须设置超时（建议 5s～30s 可配置）；超时后进入 `UNKNOWN/TIMEOUT` 状态并触发告警，不得无限阻塞；
   - **MUST**：控制线程等待期间不得持有会阻塞数据面热路径的锁或资源；
   - **MUST**：每条 ControlRequest 必须携带唯一 `command_id`（UUID），后端实现幂等缓存与去重窗口（建议保留 60s）；
   - **可验证性**：注入"模块不回 ACK/卡死"场景，验证系统在超时时间后产生 TIMEOUT 响应与确定的故障处理动作（例如进入 Degraded 或 Fault），不会无限阻塞。
MUST**：每条 ControlRequest 必须携带唯一 `command_id`；后端实现幂等缓存与去重窗口（建议 60s）。
- **[A-6.5] MUST（Ticket-04 补充）**：等待 ACK/NACK 或模块完成必须设置超时（建议 5s～30s 可配置）；超时后自动标记 `UNKNOWN/TIMEOUT` 并触发告警/降级判定，不得无限阻塞。
- **[A-6.6] MUST（Ticket-04 补充）**：控制线程等待期间不得持有会阻塞数据面的锁或资源
**冻结条款（Ticket-06 相关）**：

- **[A-6.1] MUST**：发送线程不得持有任何会被接收/ACK 侧依赖的锁；ACK 必须在独立上下文返回。
- **[A-6.2] MUST**：热路径线程（Receiver/SignalProc/DataProc）禁止同步等待 ControlEvent ACK；仅 Orchestrator 控制线程允许短暂阻塞以保证控制可靠性。
- **[A-6.3] MUST**：ACK 队列满时不阻塞接收侧，采用 drop-oldest ACK 并计数记录。
- **[A-6.4] SHOULD**：实现基于 `command_id` 的幂等缓存；超时（建议 60s）后清理以避免内存泄漏。

## 第四部分：外部接口与系统边界

### 12. 后端对外交付口径（B- 系统外部契约）

后端系统对外交付的接口是用户可依赖的契约，需要明确边界、角色与可靠性保证。

#### 12.1 系统角色与职责隔离

**后端系统**：
- 完全独立、无 GUI 依赖
- 状态权威（唯一决策者）
- 在无外部连接时可独立运行

**显控终端**：
- 仅作为外部控制参与者
- 显控发起请求，后端做决策
- 不直接写系统状态

**关键冻结**：单控制者 + 多观察者模型，V1 禁止多控制者（后续版本再支持）。

#### 12.2 通道职责与分离（TCP 控制 / UDP 态势）

系统采用**双通道分离**架构，职责明确：

##### 控制面通道（TCP）

**职责**：握手、同步状态、心跳、控制请求响应、告警通知

**可靠性要求**：
- 有序投递
- 可靠到达
- 明确的流量控制与重传
- 连接管理与心跳检活

**必须支持的操作**：
- 握手与协商（版本、功能）
- StateSnapshot 同步
- ControlRequest / ControlResponse 幂等闭环
- 系统告警与事件通知

##### 数据面通道（UDP）

**职责**：TrackData（航迹）传输

**特性**：
- 允许丢包乱序（网络异常下的可接受损耗）
- 高吞吐量低延迟
- 无连接（无会话维护开销）
- 禁止在 UDP 上发送任何控制语义

**MaxUdpPayloadBytes = 1200B 约束（与 FPGA 协议 V3.1 §数据面对齐）**：
- 每个 UDP 数据报 Payload **不得超过 1200 字节**（V3.1 冻结值），以匹配前端 FPGA 与后端的对称约束。
- 当单帧 TrackData 序列化后超过 1200B 时，发送端必须执行**应用层截断**（truncation），携带截断元数据字段：
  - `is_truncated: bool` — 该帧是否被截断
  - `drop_reason: enum {OVER_MTU, PRIORITY_SHED, ...}` — 截断原因
  - `dropped_count: uint32` — 本周期被丢弃/截断的航迹条目数
- 截断策略：优先保留高优先级航迹（如 confirmed / weapon-quality），低优先级航迹（tentative / coasted）优先丢弃。
- **clock_domain 字段**：每条 TrackData 必须携带 `clock_domain` 字段标识数据时钟域来源（与 V3.1 TimeCoordSource 对齐），供显控判断时间可信度。

**TrackData 自洽设计**：
- 采用**全量快照（Snapshot）**语义，每帧独立完整
- 必须携带 `frame_seq`（用于丢包检测）、`backend_instance_id`（用于异常重连检测）、`data_timestamp`（用于新鲜度判定）
- 允许客户端通过序号检测丢包并自主决定是否重连重同步

#### 12.3 重连与状态重建（业界标准实践）

**后端实例标识**：
- 每次启动生成新的 `backend_instance_id`（UUID）
- 包含在每条 TrackData 与 StateSnapshot 中
- 显控检测到变化 → 清除本地缓存并进入重新同步流程

**握手流程**（与 05_通信协议设计规范 §4.3 对齐）：
- 显控发起 TCP 连接
- 显控发送 Hello（含协议版本、功能集）
- 后端响应 Welcome（含版本、instance_id、协议版本兼容性检查结果）
- 后端推送 StateSnapshot（系统全量状态）
- 显控完整接收并应用 StateSnapshot 后方可下发业务控制

#### 12.4 控制闭环与幂等性

**必达语义**：
- 每条 ControlRequest 必须携带 `command_id`（UUID）
- 后端必须实现基于 `command_id` 的幂等缓存
- 每条可解析请求必须返回 ACK/NACK，**禁止静默失败**

**响应定义**：
- ACK：命令已接受并开始执行
- NACK + reason_code：命令被拒（原因：不支持、参数错误、系统状态不允许等）
- 超时：显控若 N ms 内未收到响应，自动标记为 UNKNOWN/TIMEOUT，禁止靠 " 观察 TrackData 变化 " 来猜结果

**幂等桥接层：HMI command_id ↔ DACS 幂等键（与 FPGA 协议 V3.1 对齐）**：

显控→后端使用 `command_id`（UUID）作为幂等标识；后端→DACS（FPGA）使用 V3.1 冻结的三元组幂等键 `(SourceID, ControlEpoch, SequenceNumber)`。两者语义等价但格式不同，后端**必须实现幂等桥接层**：

| 方向 | 幂等标识 | 格式 | 管理者 |
|---|---|---|---|
| 显控 → 后端 | `command_id` | UUID (128-bit) | 显控生成，后端缓存 |
| 后端 → DACS | `(SourceID, ControlEpoch, SequenceNumber)` | V3.1 冻结三元组 | 后端生成，DACS 校验 |

**桥接规则**：
- 后端收到显控 `command_id` 后，映射为 DACS 对应的 `(SourceID, ControlEpoch, SequenceNumber)`，缓存双向对应关系。
- DACS Ack 到达后，后端通过映射表翻译回 `command_id` 并向显控返回 ControlResponse。
- **三面阵 Ack 聚合**：一条 HMI 命令可能同时下发至三块 DACS。后端须等待所有目标面阵 Ack/Nack 到齐后（或超时 `RTO_MS=2500ms × MAX_RETRY=3`），聚合为一个统一的 ControlResponse 返回显控。
- **超时处理**：任一面阵超时未响应，该面阵对应子结果标记为 TIMEOUT；聚合后返回部分成功/部分超时的响应。
- **ControlEpoch 对齐**：后端每次重启或主动重置时递增 ControlEpoch，DACS 侧可据此识别旧 Epoch 命令（stale command）并拒绝。

#### 12.5 协议版本与演进

**双层版本模型**：

| 版本层 | 名称 | 含义 | 不兼容时 |
|---|---|---|---|
| **线协议** | WireVersion | TCP Framing 格式（Magic + Version + MsgType + Length） | 立即断链 |
| **语义** | ProtoVersion | 消息体定义、字段语义、行为约定 | 明确拒绝 + supported_range |

**兼容规则**：
- WireVersion 不兼容 → 立即断链（framing 错位会把后续消息全部解析成垃圾）
- ProtoVersion 不兼容 → 返回 PROTO_VERSION_MISMATCH + supported_range
- V1 策略：**V1 只能与 V1 通**；major 版本号不一致，拒绝连接

**Protobuf 演进规则**：
- 允许新增字段（向前兼容）
- 禁止删除/复用 tag（向后兼容）
- 语义变更必须升 major 版本并评审

#### 12.6 " 同机不特化 " 与可验收性

**核心原则**：协议实现不得依赖 localhost 特性，禁止共享内存/文件直读/直接函数调用旁路。

**工程含义**：
- 所有数据交换通过 TCP/UDP 进行
- 所有 endpoint 与阈值必须配置化
- 必须提供 loopback/LAN 配置模板
- 联调验收必须包含网络异常注入（丢包/乱序/延迟/断链）

#### 12.7 冻结条款

- **[B-1.1] MUST**：后端系统无 GUI 依赖；对外唯一交付为网络接口（控制 TCP + 态势 UDP）。
- **[B-1.2] MUST**：后端在无显控连接时可独立运行；显控缺失不得阻断 TrackData 生成与输出。
- **[B-1.3] MUST**：Single State-of-Truth 原则；显控只产生请求，状态裁决权仅在后端 Orchestrator。
- **[B-1.4] MUST**：单控制者 + 多观察者模型；V1 禁止多控制者（后续版本再支持）。
- **[B-2.1] MUST**：TCP 仅承载控制面（握手/快照/心跳/控制请求响应/告警）；禁止在 TCP 上传输 TrackData。
- **[B-2.2] MUST**：UDP 仅承载 TrackData；显控禁止在 UDP 上发送任何控制/确认/订阅请求。
- **[B-2.3] MUST**：TrackData 语义为全量快照，必须携带 `frame_seq`、`backend_instance_id`、`data_timestamp` 等字段用于丢包检测与新鲜度判定。
- **[B-3.1] MUST**：TCP Framing 采用显式格式（Magic + Version + MsgType + Length），不得依赖 read 边界；长度超限立即断开。
- **[B-3.2] MUST**：握手完成后后端必须发送 StateSnapshot；显控未收到并应用快照前禁止下发业务控制。
- **[B-3.3] MUST**：后端每次启动生成新的 `backend_instance_id`；显控检测到变化必须清缓存并进入重新同步流程。
- **[B-4.1] MUST**：每条 ControlRequest 必须携带唯一 `command_id`；后端必须实现幂等缓存；对每条可解析请求必须返回 ACK/NACK。
- **[B-4.2] MUST**：控制结果仅以 ControlResponse 为判定依据；超时必须显式标记 UNKNOWN/TIMEOUT，禁止靠观察 TrackData 或系统状态 " 猜结果 "。
- **[B-5.1] MUST**：采用双层版本模型（WireVersion 线协议 + ProtoVersion 语义）；WireVersion 不兼容立即断链；ProtoVersion 不兼容返回 supported_range。
- **[B-5.2] MUST**：Protobuf 演进仅允许新增字段；禁止删除/复用 tag；语义变更必须升 major 版本。
- **[B-6.1] MUST**：协议实现不得依赖 localhost 特性；禁止共享内存/文件直读/函数调用旁路。
- **[B-6.2] MUST**：联调验收包含 TCP 断链/抖动与 UDP 丢包/乱序注入。
- **[B-2.6] MUST**：UDP TrackData 单报文 Payload ≤ 1200B（MaxUdpPayloadBytes，与 FPGA 协议 V3.1 冻结值对齐）。超限帧必须执行应用层截断并标注 `is_truncated`、`drop_reason`、`dropped_count`。
- **[B-2.7] MUST**：每条 TrackData 必须携带 `clock_domain` 字段，标识数据时钟域来源（映射自 V3.1 TimeCoordSource）。
- **[B-2.8] SHOULD**：截断策略应优先保留高优先级航迹（confirmed/weapon-quality），低优先级航迹（tentative/coasted）优先丢弃。

## 第五部分：GPU 与实时处理路线

### 13. GPU 集成与提交者模型（C-GPU 架构）

GPU 是信号处理的核心，其组织形态直接决定系统的实时性与可维护性。

#### 13.1 GPU 提交者模型（集中式）

**架构选择**：采用**单一 GPU 调度线程（GPU Dispatcher）**统一提交所有 GPU 工作。

**组织形态**：

```txt
┌─────────────────────────────────────────┐
│  各阵面处理链路（Receiver / Pre-Proc）   │
│  └→ 完整 CPI（或处理块）推到输入队列     │
└────────────┬────────────────────────────┘
             │ SPSC Queue
             ▼
┌─────────────────────────────────────────┐
│   GPU Dispatcher（唯一 GPU 提交线程）    │
│   • 统一调度：streams、in-flight、背压    │
│   • 监控：GPU 队列、显存、pinned 内存    │
│   • 降级与丢弃：资源不足时控制流量        │
└────────────┬────────────────────────────┘
             │ CUDA Streams
             ▼
┌─────────────────────────────────────────┐
│   GPU Device（并行计算）                  │
│   • FFT / CFAR / MTD / 其他信号处理      │
│   • 双缓冲 (ping-pong) 流水线             │
└────────────┬────────────────────────────┘
             │ CUDA Events
             ▼
┌─────────────────────────────────────────┐
│   CPU 融合（Track / DataProc）           │
│   • 结果回收与处理                        │
└─────────────────────────────────────────┘
```

**优势**：
- 统一计量：GPU 队列深度、in-flight 数、显存使用、pinned 内存使用
- 明确的背压点：队列满时执行 drop-oldest CPI 策略
- 易于验收与回归：集中的调度决策便于复现与监控

#### 13.2 Streams 与 In-Flight 管理

**Streams 分配（固定，V1 禁止动态）**：

| 模式 | 数量 | 说明 |
|---|---|---|
| **每阵面 1 stream** | 3（假设三阵面） | 推荐起步值，简单稳健 |
| **每阵面 2 streams** | 6 | 高级模式：copy + compute 分离，仅在确保可驾驭时启用 |

**In-Flight 上限（与显存预算配合）**：

- **每 stream 最多 2 个 in-flight CPI**（对应 ping-pong / 双缓冲）
- 含义：提交第 N+1 个 CPI 的 H2D 时，第 N 个 CPI 可能在 compute 或 D2H 阶段（充分重叠）
- 上限确保：显存与 pinned 缓冲上界可控，延迟可预测

**流水线覆盖**：

```txt
时刻 t=0: CPI_0 → H2D
时刻 t=x: CPI_0 compute | CPI_1 H2D（重叠）
时刻 t=2x: CPI_0 D2H | CPI_1 compute | CPI_2 H2D（三段重叠）
```

#### 13.3 GPU↔CPU 背压与丢弃

**背压主丢弃点**：进入 GPU 前（CPI 输入队列）

**策略**：
- 队列满 → **drop-oldest CPI**（不阻塞生产者）
- 立刻记录：`gpu_queue_drop_cpi_total`、`gpu_queue_depth`
- 收集指标：`gpu_latency_p95/p99`、`gpu_stage_latency`（H2D/compute/D2H）

**次级降级**（V1 预留开关，暂不强制复杂逻辑）：
- 若 GPU backlog 持续超过阈值，允许跳过低价值 stage（例如生成 Plots/可选中间产物）
- 必须确保 Track 所需的最小检测链路仍可用

#### 13.3.1 跨阵面公平性与降级阈值（新增，对应 Ticket-07）

**问题**：三阵面并行输入 GPU 时，某阵面高负载可能长期挤占 GPU，导致其他阵面服务降低。

**公平性策略**（冻结）：

1. **提交调度规则**：
   - GPU Dispatcher 采用**简单轮询**（round-robin）逐阵面提交：Array0 → Array1 → Array2 → Array0 …
   - 或采用**加权轮询**，各阵面权重相等（v1.0 暂不做动态权重）

2. **Per-Array Backlog 限制**（可选，v1.0 可 TBD）：
   - 每阵面 in-flight CPI 数上限：例如 `max_inflight_per_array = 4`
   - 当某阵面 backlog 达上限时，该阵面暂停提交，继续服务其他阵面
   - 避免单阵面囤积导致其他阵面等待过长

3. **Degraded 阈值映射**：

| 触发条件 | 指标字段 | 阈值 | Degraded 行为 |
|---|---|---|---|
| GPU 队列高水位 | `gpu_queue_depth_percent` | ≥ 90% | 启动降级，提高 CPI drop 率或跳过低价值 stage |
| 单阵面 backlog | `array_backlog_depth[i]` | ≥ T_array_max（TBD） | 该阵面暂停提交，继续服务其他 |
| GPU 处理延迟升高 | `gpu_end2end_latency_p99` | ≥ 工程目标 + 100% | 进入 Degraded，启动软抢占（拆分 kernel）或跳过 stage |
| CPI 丢弃率 | `gpu_queue_drop_rate` | ≥ threshold（建议 10%） | Degraded 告警，可能需要限制输入帧率 |

**说明**：
- `T_array_max` 是每阵面允许的最大 in-flight CPI 数，需在性能基线中确定
- " 暂停提交 " 是非阻塞的（Dispatcher 继续处理其他阵面，只是跳过当前阵面）
- 这套策略确保在单阵面高负载时，其他阵面仍能获得最低服务份额

#### 13.4 内存与缓冲管理

**Host 侧 Pinned 内存池**：

- **所有权**：Node1（GPU 最近的 NUMA 节点）
- **用途**：H2D/D2H 传输缓冲
- **管理**：从池中租用，使用后归还池（RAII）
- **大小**：与 in-flight 上限 + stream 数量配合计算（预分配，不动态扩展）

**Device 侧缓冲预分配**：

- 为每 stream 预分配固定大小的 device 缓冲
- 与 in-flight 上限相匹配（避免运行时 cudaMalloc 抖动）
- 示例：3 streams × 2 in-flight × CPI_size = device 缓冲总量

**数据结构生命周期**：

- CPI / RawBlock 必须支持**智能指针 + 自定义 deleter**
- deleter 负责将块归还内存池（RAII 保证）
- 禁止手动 free / delete 在热路径

#### 13.5 完成通知与结果回收

**推荐方案**：轮询 CUDA events（非回调线程）

```cpp
// GPU Dispatcher 主循环
for (auto& stream : streams) {
    if (cudaEventQuery(event[stream]) == cudaSuccess) {
        // 该 in-flight CPI 已完成
        push_result_to_fusion_queue(stream);
        cudaEventRecord(event[stream], stream);  // 重用 event
    }
}
submit_next_cpi_if_space();  // 若有空位，提交下一个
```

**优点**：
- 非阻塞轮询，与忙轮询数据面风格一致
- 简单可控，避免回调线程模型的复杂性与调试难度

#### 13.6 错误与超时处理

**GPU 错误**（kernel failure / memcpy failure）：
- 立刻检测（通过 CUDA error checking wrapper）
- 上报并进入 Degraded 或 Fault（取决于是否还能产出可信 Track）
- 记录错误码与调用栈用于诊断

**GPU backlog 超时**：
- 不将 " 继续处理旧 CPI" 视为正确行为
- 应采用 drop-oldest 策略，保证输出 " 最新态势 "
- 监控 `gpu_max_latency` 与 `gpu_latency_p99`，超阈值时进入告警/降级

**GPU 单 CPI 超时/长耗时（P0-3，冻结裁决）**：
- **MUST**：任何单个 CPI 的 GPU 处理必须存在**可验收的时间上界**（超时阈值在性能基线阶段确定，并在 FAT 前冻结为 `T_gpu_cpi_deadline_ms` 或等价参数）。
- **MUST**：当单 CPI 处理超出该上界时，必须将该 CPI 标记为 TIMEOUT 并**丢弃其结果**（不参与后续更新），释放 in-flight 槽位继续处理后续 CPI；不得让单次任务无限期占用流水线。
- **MUST**：超时必须计数与上报，并按第二篇 9.3 的阈值体系触发 Degraded；若进一步导致 `trackdata_gap_ms` 触及 T1/T2，则按既定规则进入 Degraded/Fault。
- **FORBIDDEN**：热路径线程（Receiver/SignalProc/DataProc）不得同步等待某个 GPU 结果“直到完成”；不得以阻塞等待来换取“完整处理旧数据”。

#### 13.7 冻结条款

- **[C-1.1] MUST**：GPU 工作提交由单一 GPU Dispatcher 线程统一负责；其他线程禁止直接提交 CUDA kernel。
- **[C-1.2] MUST**：Dispatcher 需要统一管理：stream 生命周期、in-flight 上限、背压策略、显存与 pinned 内存预算。
- **[C-2.1] MUST**：Streams 数量固定分配；V1 禁止运行时动态创建/销毁 stream；默认每阵面 1 stream。
- **[C-2.2] MUST**：每 stream in-flight 上限固定为 2；超过上限不得阻塞生产者，采用 drop-oldest CPI。
- **[C-3.1] MUST**：主背压点在 " 进入 GPU 前（CPI 队列）"；队列满时 drop-oldest CPI 并计数记录。
- **[C-3.2] SHOULD**：V1 将 " 跳过低价值 stage" 的降级作为可配置开关预留，但不作为强依赖能力。
- **[C-3.3] MUST（新增，Ticket-07）**：采用轮询调度逐阵面提交 CPI，避免单阵面长期囤积；监控 `gpu_queue_depth_percent`、`array_backlog_depth[i]`、`gpu_drop_rate` 与 Degraded 阈值绑定。
- **[C-3.4] SHOULD（新增，Ticket-07）**：预留 Per-Array Backlog 上限（例如 `max_inflight_per_array`），确保各阵面获得最低服务份额；此值需在性能基线中确定。
- **[C-4.1] MUST**：Host pinned 内存池必须 NUMA 感知（优先 Node1）；Device 缓冲预分配；运行期热路径禁止 malloc/cudaMalloc。
- **[C-4.2] MUST**：所有 CPI / RawBlock 的生命周期必须可回收（智能指针 + 自定义 deleter 归还池）。
- **[C-5.1] SHOULD**：完成通知采用轮询 CUDA events 而非回调线程；实现简洁且易调试。
- **[C-6.1] MUST**：GPU 错误与超时必须记录并上报；backlog 超时不得保留旧数据，应采用 drop-oldest 保证新鲜度。
- **[C-6.2] MUST（P0-3）**：单 CPI GPU 处理必须有可验收的超时上界；超时结果必须丢弃并释放 in-flight，防止流水线被单次长耗时卡死。
- **[C-6.3] FORBIDDEN（P0-3）**：禁止任何“无限期等待 GPU 完成”的路径进入热路径；禁止用阻塞等待替代丢弃/降级。

## 第六部分：工程落地与实现准则

### 14. 数据面处理准则（E- 实时保证）

#### 14.1 收包模型与轮询优先

**决策**：Data Plane（Receiver）采用 Busy-Poll / Polling 模型

**理由**：
- GB/s 级数据流 + 微秒级低时延诉求下，轮询相比事件驱动提供**一致的、确定的**低延迟
- 避免中断处理与唤醒延迟带来的抖动
- 参考项目（RadarFlow）中 Receiver 使用 `recvfrom` 在用户态紧凑轮询

**实现约束**：

```cpp
// Receiver 主循环示意
while (running) {
    ssize_t len = recvfrom(sock, buffer, size, MSG_DONTWAIT, ...);
    if (len > 0) {
        process_fragment(buffer, len);
    }
    // 不 sleep，继续轮询
}
```

#### 14.1.1 FPGA 协议 V3.1 接收适配层（冻结，与一致性耦合分析报告对齐）

Receiver 必须实现严格遵守 FPGA 协议 V3.1 的**协议适配层（Protocol Adapter + Reassembly Engine）**，确保接收端行为与已定版协议完全一致。以下为核心适配要求：

**应用层分片重组引擎（冻结）**：

- **[RECV-V31-1] MUST**：重组超时冻结为 **`T_reasm = 100ms`**（从收到该重组键的第一个分片起计）；**禁止**使用更短超时（例如 10ms）提前丢弃不完整帧。
  - 原设计中"T_reassembly=10ms 超时丢弃不完整 CPI"与 V3.1 冻结条款存在**硬不一致**，必须以 V3.1 为准整改。
- **[RECV-V31-2] MUST**：重组键必须按 V3.1 定义：`(ControlEpoch, SourceID, FrameCounter, BeamID, CpiCount, PulseIndex, ChannelMask, DataType)` 严格分桶，禁止对重组键做简化或省略字段。
- **[RECV-V31-3] MUST**：当 `T_reasm` 到期仍未收齐所有分片时，必须按 `FragIndex` 顺序输出一帧，**缺失分片按分片补零**（每个缺失 FragIndex 补入等长全 0 字节），并标记 `IncompleteFrame=1`、累加缺片计数。**禁止**直接丢弃不完整帧。
- **[RECV-V31-4] MUST**：迟到分片（重组超时输出后到达的同重组键分片）一律丢弃并计数。
- **[RECV-V31-5] MUST**：重组资源限制冻结：`MAX_REASM_BYTES_PER_KEY = 16 MiB`、`MAX_TOTALFRAGS = 1024`；超出则丢帧并计数。

**尾包 Execution Snapshot 提取（冻结）**：

- **[RECV-V31-6] MUST**：V3.1 冻结尾包（`FragIndex == TotalFrags - 1`）携带 40B Execution Snapshot（实际执行参数快照：方位角、俯仰角、工作频率、增益、带宽、脉宽、积累点数、姿态、经纬高等）。接收层**必须提取并传递 Snapshot**，供 Orchestrator 进行"控制下发 vs 前端实际生效"的闭环一致性验证。
- **[RECV-V31-7] FORBIDDEN**：禁止接收层仅取 RAW 数据而忽略/丢弃 Execution Snapshot；否则将失去"控制是否按期生效"的证据链，使 APPLIED_LATE、OVERRUN 等问题无法定位。

**心跳优先队列（与 V3.1 QoS 对齐）**：

- **[RECV-V31-8] MUST**：Data（0x03）与 Heartbeat（0x04）复用 UDP/9999 时，接收端必须按 `PacketType` 分类：Heartbeat 进入**高优先级队列并优先处理**；Data 进入普通队列。
- **[RECV-V31-9] MUST**：V3.1 冻结 Heartbeat DSCP=CS6、Data DSCP=CS0。若网络侧清除 DSCP，仍须通过软件侧 PacketType 分类保底，确保数据洪峰下心跳不被饿死。
- **[RECV-V31-10] FORBIDDEN**：禁止接收端在数据重组与心跳解析共享同一拥塞点；否则数据面洪峰将导致心跳饿死→误判前端失联→错误降级/复位的级联故障。

#### 14.2 等待策略与禁区

**禁止在热路径（Data Plane）使用被动等待机制**：
- `sleep()` / `usleep()`
- `condition_variable::wait()`
- `sem_wait()`
- 任何会导致线程阻塞/让出 CPU 的同步原语

**例外**：非热路径（Control/Management Plane）可适度使用事件驱动（低频指令，实时性要求宽松）。

#### 14.3 冻结条款

- **[E-1] MUST**：Data Plane 收包模型采用 Busy-Poll / Polling；不得使用被动等待（sleep/condvar）。
- **[E-2] FORBIDDEN**：热路径（Receiver / SignalProc / DataProc）禁止使用 sleep / condition_variable / sem_wait 等被动等待；任何阻塞操作会立刻破坏实时性。
- **[E-3] SHOULD**：Control/Management Plane 可采用事件驱动（条件变量、事件对象等），以降低 CPU 占用。
- **[E-4] MUST**：分片重组超时冻结为 `T_reasm = 100ms`（与 FPGA 协议 V3.1 §4.1 对齐）；禁止使用 10ms 或其他更短超时。
- **[E-5] MUST**：接收层必须对 V3.1 尾包携带的 40B Execution Snapshot 进行提取与转发，不得丢弃。
- **[E-6] MUST**：Heartbeat（PacketType=0x04）与 Data（PacketType=0x03）须实现队列分离或优先级分类，保障数据洪峰下心跳不饿死。

### 15. NUMA 与内存亲和（C3-NUMA 意识）

#### 15.1 内存池 NUMA 分离

**架构**：至少分离 Node0 / Node1 两个池

| 节点 | 用途 | 相邻硬件 | 绑定线程 |
|---|---|---|---|
| **Node0** | 控制/管理相关 | CPU0-15、网口/其他 | M04（网关）与控制接入/运维线程（若独立） |
| **Node1** | 实时数据处理 | CPU16-31、NIC（聚合）、GPU | M01/M02/M03（接收、信号处理、融合）+ Orchestrator（FSM 核心） |

**内存池分配原则**：
- Data Plane 线程**必须仅使用 Node1 本地池**，杜绝跨 NUMA 访存
- Control/Management Plane 默认使用 Node0；仅允许少量跨域访问用于低频诊断/运维，不得进入热路径关键循环
- GPU 传输缓冲（pinned memory）优先 Node1，最近的 NUMA 邻近性

#### 15.2 线程 CPU 绑定

**绑定策略**：
- M01/M02/M03（Data Plane）：静态绑定至 CPU 16-31（Node1）
- M04（网关）：绑定至 CPU 0-15（Node0）
- 禁止线程浮动（系统默认调度可能导致频繁跨 NUMA 迁移）

#### 15.3 并行队列与缓存行隔离

**相邻 stage 队列**：优先使用 SPSC（Single Producer Single Consumer）

**理由**：
- 生产者与消费者各自独占，无锁竞争
- 配合 cacheline 隔离避免 false sharing（伪共享）

**cacheline 隔离示例**：

```cpp
struct SPSC_Queue {
    alignas(64) std::atomic<uint64_t> head;  // 生产者独占
    alignas(64) std::atomic<uint64_t> tail;  // 消费者独占
    std::array<T, capacity> buffer;          // 数据缓冲
};
```

**禁止场景**：
- 避免 MPMC（Multi Producer Multi Consumer）在热路径出现（会引入锁竞争）
- 若多生产者/多消费者，需要在融合点汇聚（单点无锁合并）

#### 15.4 冻结条款

- **[C3-1] MUST**：内存池采用 NUMA-aware 分池（至少 Node0/Node1 分离）；Data Plane 必须仅使用 Node1 本地池。
- **[C3-2] MUST**：Data Plane 线程（M01/M02/M03）静态绑定至 CPU 16-31（Node1）；禁止线程浮动。
- **[C3-3] SHOULD**：GPU pinned memory 缓冲优先分配在 Node1，最小化 NUMA 延迟。
- **[C3-4] MUST**：相邻 stage 队列优先采用 SPSC；融合阶段处理多生产者汇聚。
- **[C3-5] SHOULD**：共享变量与队列头尾指针进行 cacheline 隔离（alignas(64)），避免 false sharing。

### 16. 运行期内存管理（C3- 内存零分配）

#### 16.1 禁止运行期动态分配

**核心准则**：启动期静态预分配所有对象，运行期严格禁止 malloc/new/cudaMalloc。

**执行方式**：
- 启动期：根据最坏情况容量预分配所有缓冲与对象池
- 运行期：仅从池中租赁对象，使用后归还
- 出现超容量需求时：不动态扩展，而是拒绝/丢弃新请求

#### 16.2 对象生命周期管理（RAII）

**设计模式**：

```cpp
// RawBlock 拥有权管理
using RawBlockPtr = std::shared_ptr<RawBlock>;

RawBlockPtr block = raw_pool.acquire();  // 租赁
// ... 使用 block ...
block.reset();  // 或自动析构时，deleter 调用 pool.release()
```

**deleter 定义**：

```cpp
class RawBlockDeleter {
    RawMemoryPool* pool;
    void operator()(RawBlock* block) const {
        pool->release(block);  // 归还池
    }
};
```

#### 16.3 零拷贝域与指针传递

**设计原则**：RawBlock / CPI 在流水线中**传递指针，不复制数据**

**受禁操作**：
- 禁止在模块间复制 RawBlock（浪费带宽）
- 禁止拆分 RawBlock（打破单位的完整性与生命周期管理）
- 允许重新解释（在同一块内存上进行不同的数据视图，但所有权与回收不变）

#### 16.4 冻结条款

- **[C3-2] MUST**：动态分配禁止出现在实时循环/热路径；必须使用静态预分配 + 内存池的 O(1) 模式。
- **[C3-3] MUST**：RawBlock / CPI 在流水线中传递指针（零拷贝）；禁止复制/拆分；所有权通过智能指针与自定义 deleter 管理。
- **[C3-4] MUST**：对象回收必须是自动化的（RAII）；禁止手动 free/delete 在热路径。
- **[C3-5] FORBIDDEN**：运行期出现 malloc/new/cudaMalloc；超容量需求应拒绝/丢弃，不扩展。

### 17. 背压与队列管理（C3/D-4 流控）

#### 17.1 非阻塞队列设计

**目标**：队列满时**不阻塞生产者**，采用 " 丢弃/覆盖旧数据 " 策略。

**实现模式**：

```cpp
// 环形缓冲，写满时覆盖最老数据
template<typename T>
class RingQueue {
    std::array<T, capacity> buffer;
    uint64_t write_idx = 0;
    uint64_t read_idx = 0;

    bool try_push(const T& value) {
        if ((write_idx - read_idx) >= capacity) {
            // 队列满，覆盖最老数据
            read_idx++;
            drop_count++;
        }
        buffer[write_idx % capacity] = value;
        write_idx++;
        return true;  // 永远不阻塞
    }
};
```

#### 17.2 丢弃计数与告警

**必须记录的指标**：
- `queue_drop_total`：累计丢弃数
- `queue_depth_watermark`：队列高水位
- `queue_overflow_count`：溢出次数

**告警逻辑**：
- 单次溢出 → SEV-3 告警（记录但暂不降级）
- 持续溢出（≥ N ms 内 M 次）→ SEV-2 告警（进入 Degraded）
- 关键队列（TrackPool）满 → SEV-1 告警（进入 Fault）

#### 17.3 冻结条款

- **[D-4.1] MUST**：队列满与背压采用非阻塞策略；不得阻塞生产者等待消费。
- **[D-4.2] SHOULD**：队列满时采用 " 丢弃/覆盖最旧数据 "（drop-oldest）而非阻塞等待。
- **[D-4.3] MUST**：所有丢弃操作必须计数记录；丢弃计数与队列高水位用于监控告警。
- **[D-4.4] MUST**：关键队列（TrackData / ControlEvent）满时触发 SEV-1 告警；非关键队列（Raw/Plots）满时触发 SEV-2 告警。

### 18. 网络传输与可靠性（F- 通道保证）

#### 18.1 UDP TrackData 传输的最小可靠性集

**若选择 UDP 作为 TrackData 通道**，必须实施以下保证机制：

**序号机制**：
- 每条 TrackData 必须携带 `frame_seq`（递增序号）
- 显控可通过序号检测丢包
- 丢包不重传，仅作为诊断信号

**完整性校验**：
- CRC32 或 SHA256（视负荷容限决定）
- 校验失败的 frame 直接丢弃（不应用到显示）

**丢包监控**：
- 统计连续序号间隔，识别丢包与乱序
- 上报到监控系统（不触发业务降级，仅诊断）

**重传策略**（可选，V1 推荐不做）：
- 若实现，需要双向通道（显控 → 后端的 ACK/NACK）
- 复杂度较高，V1 建议先用 " 丢包告警 " 而非重传

#### 18.2 TCP 控制面的可靠性

**已在 B 部分详述**，此处补充传输层细节：

**Framing**：

```txt
┌─────────┬─────────┬──────────┬────────────┬───────────┐
│ Magic   │ Version │ MsgType  │ PayloadLen │ Payload   │
│ 4 bytes │ 2 bytes │ 2 bytes  │ 4 bytes    │ Variable  │
└─────────┴─────────┴──────────┴────────────┴───────────┘
```

**断链检测**（与 05_通信协议设计规范 §8.3 对齐）：
- 心跳周期：默认 1000ms
- 心跳超时阈值：3000ms（连续 3 次心跳未响应）
- 超时后判定为链路断开，触发重连

#### 18.3 冻结条款

- **[F-1] MUST（仅当选 UDP）**：TrackData 必须携带 `frame_seq`（序号）与 CRC（校验码）。
- **[F-2] MUST（仅当选 UDP）**：显控侦测丢包并上报诊断；失败帧直接丢弃。
- **[F-3] SHOULD（仅当选 UDP）**：V1 不实现重传，仅做丢包监控；后续版本可升级。
- **[F-4] MUST**：TCP 控制面采用显式 framing（Magic + Version + MsgType + Length）；长度超限立即断开。
- **[F-5] MUST**：心跳检活周期配置化；失败 N 次后判定断链，触发重连与状态重同步。

## 第七部分：综合验收与测试要点

### 19. 关键验收准则

#### 19.1 单系统可用性验收（与第 9 节阈值表对齐）

| 验收项 | 判定标准 | 对应阈值 | 备注 |
|---|---|---|---|
| **TrackData 连续性** | 输出中断 ≥ T2_trackdata_gap_ms(2s) | SEV-1 Fault | 必达指标，失败立刻停止输出 |
| **TrackData 更新率** | 输出中断 ≥ T1_trackdata_gap_ms(500ms) | SEV-2 Degraded | 降频输出；超 2s 进入 Fault |
| **队列溢出频率** | ≥ 1 次/10s 持续 | SEV-2 Degraded | 告警并启动降级 |
| **内存池高水位** | ≥ 90% 持续 | SEV-2 Degraded | 触发降级策略 |
| **Plots 丢弃率** | ≥ 10% | SEV-2 Degraded | 或启动优化/降级 |
| **GPU 队列深度** | `gpu_queue_depth_percent` ≥ 90% | SEV-2 Degraded | 启动 CPI drop 或跳过低价值 stage |
| **处理延迟 P99** | ≥ 工程目标 +100% | SEV-2 Degraded | 可能触发软抢占或降级 |
| **时间源同步** | `SYNC_LOST` ≥ 1s | SEV-2 Degraded | 告警；继续输出已缓存态势 |
| **时间戳失真** | 连续 ≥ 3 次失反 | SEV-1 Fault | 停止输出；记录诊断信息 |

#### 19.2 网络可靠性验收

**必须包含的异常注入测试**：

```txt
场景 1: UDP 丢包（1%/5%/10%）
  验证：TrackData 序号完整性检测、丢包告警上报

场景 2: UDP 乱序
  验证：显控能通过 frame_seq + instance_id 判定新鲜度

场景 3: TCP 断链 + 快速重连
  验证：状态重同步正确，TrackData 不中断

场景 4: TCP 缓冲写满
  验证：背压策略不阻塞数据面

场景 5: 显控宕机，后端独立运行
  验证：无显控连接时 TrackData 仍正常生成
```

#### 19.3 故障恢复验收

**必须验证的场景**：

```txt
场景 1: Receiver 掉线 30s 后恢复
  验证：进入 Degraded，恢复后回到 Running

场景 2: GPU backlog 超过阈值
  验证：触发 CPI drop-oldest，延迟恢复正常

场景 3: 单阵面高负载，其他阵面公平性
  验证：注入阵面 A 高帧率（2x），监控阵面 B/C 的 latency/backlog
  目标：B/C 虽然延迟升高但仍能继续处理（不被饿死）

场景 4: 内存池耗尽
  验证：丢弃优先级遵循预定规则，不阻塞热路径

场景 5: 时间源故障
  验证：进入 Fault，停止对外输出，有明确告警

场景 6: 连续故障导致 TrackData 长时间缺失（≥ T2_trackdata_gap_ms）
  验证：系统进入 Fault，显控收到告警并自动化处理
```

#### 19.4 冻结条款

- **[验收 -1] MUST**：所有功能测试必须包含网络异常注入（丢包/乱序/延迟/断链）。
- **[验收 -2] MUST**：故障恢复测试必须覆盖 SEV-1/SEV-2/SEV-3 三个严重度。
- **[验收 -3] MUST**：无显控连接情况下的独立运行必须通过验证。
- **[验收 -4] SHOULD**：建立 " 故障场景库 " 并定期执行，以验证系统的可恢复性与自愈能力。

## 附录 A：术语与定义

| 术语 | 定义 |
|---|---|
| **Degraded** | 系统可用性下降，但仍能输出部分有效态势，允许继续执行降级策略 |
| **Fault** | 系统可信性或可用性到达不可接受的底线，停止对外业务输出 |
| **TrackData** | 稳定跟踪目标的状态快照（ID/时间戳/位置/速度/置信度等），业务必达 |
| **Plots** | 检测点迹，中间产物，允许丢弃 |
| **CPI** | Complete Processing Instance，一个完整处理单元（包含多帧采样数据） |
| **RawBlock** | 原始采样数据块（IQ 数据），允许完全丢弃 |
| **drop-oldest** | 丢弃队列中最老的数据，优先保持最新态势 |
| **SPSC** | Single Producer Single Consumer 队列，无锁高效 |
| **Pinned Memory** | 页锁定主机内存，用于 GPU 传输，支持异步 DMA |
| **StateSnapshot** | 系统全量状态快照，用于断链后的重同步 |
| **Command ID** | 控制命令的唯一标识，用于幂等性与追溯 |
| **instance_id** | 后端实例的唯一标识，每次启动生成新值，用于检测重启 |
