# 00_TopLevel_Archure.mditect

## Part 1. System Scope & Objectives

### 1.1 System Identity & Mission

本系统为 **中国铁塔低成本两维相控阵雷达（2025 定制版）** 中的 **信号处理与态势生成后端系统（QDGZ300）**，其核心使命是：

> **在严格的实时约束下，持续输出可信、可操作的目标航迹态势（TrackData），用于低空无人机、地面人员及车辆的探测与监视。**

系统通过对雷达前端（QDGZ100）采集的高速原始数据进行实时信号处理、数据处理与融合，并经数据网关向显控系统下发结果，构成完整的后端处理链路。

---

### 1.2 Primary Outputs and Business Completion Criteria

#### 1.2.1 **Business-Critical Output (MUST)**

系统**唯一的业务必达输出**定义为：

**TrackData（目标航迹数据）**

每一条被稳定跟踪的目标航迹 **必须** 提供以下最小信息集合：

- Target ID
- 高精度时间戳（强一致时间）
- 空间位置（Position）
- 运动速度（Velocity）
- 置信度（Confidence）

在正常负载条件下，系统应满足以下工程目标（非 SLA）：

- **航迹更新率 ≥ 1 Hz**

> **业务完成的唯一判据**：
> 系统能够持续输出可信、实时、可操作的 TrackData。

以下情形 **不视为业务成功**：

- 仅输出未关联的点迹（Plots）；
- 输出大量虚假或不稳定航迹；
- 航迹输出存在严重延迟，已失去操作价值。

---

#### 1.2.2 **Non-Critical / Deprioritized Outputs (NOT MUST)**

- **Detection / Plots（点迹）**
    - 属于中间产物，**不作为业务必达输出**；
    - 在资源紧张或局部故障情况下，允许丢弃、降精度或跳帧处理；
    - 仅需满足生成稳定 TrackData 的最低输入需求。
- **Logs / Telemetry（日志与遥测）**
    - 用于运维、监控与问题定位；
    - 允许降频、采样或丢弃。
- **本地存档分析数据**
    - 主要用于调试与离线复现；
    - 不影响在线业务成败判定。

---

### 1.3 System Consumers

系统输出面向多类消费者，按优先级划分如下：

1. **Primary Consumers**
    - 显控终端（人类值守操作员）
    - 显控平台 / 上级系统
2. **Secondary Consumers**
    - 系统监控与日志平台（运维态势）
3. **Debug / Engineering Consumers**
    - 本地存档与离线分析工具链

---

### 1.4 Deployment Scope (V1)

- **V1 版本即按产品交付形态运行**：
    - 单台后端服务器（QDGZ300）
    - **同时接入三阵面雷达数据**
    - 用于完成必要的数据融合与整体行为验证

> 单阵面处理仅作为算法或模块级调试手段存在，不代表系统交付形态。

---

### 1.5 Explicit Non-Goals (V1)

在 V1 阶段，以下内容 **明确不在系统目标范围内**：

- 复杂目标识别与分类（仅采用最简算法）；
- 端侧 AI 训练（未来可在本地训练，端侧仅加载模型）；
- 跨平台兼容（硬件平台被明确限定）；
- 云端分布式处理或弹性伸缩。

V1 的目标聚焦于：

- **获得稳定、可信的点迹与航迹结果**；
- 满足 **50 目标搜索 + 1 目标精确跟踪** 的系统能力目标。

## Part 2. Constraints & Assumptions

### 2.1 Hardware & Platform Constraints

系统运行在**严格限定的硬件与软件平台**之上：

- **CPU**：飞腾 S5000C（ARM64 / aarch64，32 核）
- **NUMA 架构**：
    - Node 0：CPU 0–15（管理/非实时平面）
    - Node 1：CPU 16–31（实时计算平面）
- **GPU**：Iluvatar MR-V100
    - CoreX SDK 4.3.8
    - CUDA 10.2 API 兼容
- **操作系统**：麒麟 Linux Advanced Server V10（Kernel 4.19.90）
- **数据输入**：10GbE 光纤（原始数据率约 1.25 GB/s）
- **存储层级**：
    - NVMe SSD：高速缓冲与录制
    - HDD RAID10：长期存储

以上约束为系统设计的**不可变前提**。

---

### 2.2 Runtime Modes (Frozen)

系统必须支持以下全局运行模式集合：

- `Init`
- `Standby`
- `Running`
- `Degraded`
- `Fault`

> **注**：`Shutdown` 不属于 `system_state` 值域，而是 **lifecycle 事件**（关闭命令触发的优雅退出流程）。

该状态集合在 V1 即被冻结，后续版本仅允许扩展其内部语义，不得随意删除或合并。

---

### 2.3 Time Model & Synchronization Assumptions

系统采用 **双时间体系模型**：

1. **强一致高精度时间（Authoritative Time）**

    - 用于雷达信号处理、航迹生成与跨阵面时间连续性；
    - **唯一权威来源**：外部高精度时间设备（北斗/GPS/PTP）；
    - 当前原型阶段暂使用 NTP，但**架构上不承认其为有效权威时间源**；
    - 外部授时失效时，系统将进入守时模式，并预留算法对齐能力（TBD）。

2. **近似时间 / 挂钟时间**

    - 用于日志、监控、运维展示；
    - 不参与雷达处理与数据融合逻辑。

即使三阵面覆盖空域无重叠，系统仍 **强制要求全局时间一致性**，以避免目标跨阵面边界时出现速度失真、方向跳变或因果倒置。

---

### 2.4 Performance Targets (Engineering Targets)

当前阶段所有性能指标均为 **工程目标（SHOULD meet）**，不作为系统失效判据：

- **Gaze / Track 模式**：端到端更新延迟 ≤ 60 ms
- **Frame 更新周期**：≤ 6 s

允许短时间抖动，后续将通过原型验证逐步量化与收敛。

---

### 2.5 Real-Time Priority Policy

系统明确采用以下设计哲学：

> **保证实时性优先于完整性，允许丢弃过期或低价值数据。**

在系统过载或背压情况下，数据丢弃优先级（从最可丢到最不可丢）冻结如下：

1. Raw IQ（原始采样数据）
2. Logs / Telemetry
3. Detection / Plots（点迹）
4. **TrackData（语义必达：持续发布最新快照，可降更新率）**
5. **Control Events（绝对不可丢，独立高可靠通道）**

---

### 2.6 Raw IQ Recording Constraint (V1 MUST)

- **Raw IQ 数据录制能力在 V1 中为必选能力**；
- 用于算法验证、问题复现与对比分析；
- 录制应优先使用 NVMe 等高速介质，并采用异步写入；
- **录制链路不得阻塞或拖慢核心实时处理路径**；
- 若存储资源不足，允许牺牲录制完整性，但不得影响 TrackData 输出。

---

### 2.7 Failure Definition & Initial Availability Criteria (V1)

在 V1 阶段，系统采用以下**初始可用性判据**（用于监控与评估，非 SLA）：

- **分级判定方案（冻结）**：
  - `T1 = 500ms`：连续 ≥ 500ms 无任何 TrackData 输出，视为进入 `Degraded`
  - `T2 = 2s`：连续 ≥ 2s 无任何 TrackData 输出，视为进入 `Fault`
- TrackData 延迟 P99 超过预设阈值（初始建议 2–5 s）触发告警；
- 持续队列溢出或数据丢弃计数异常增长触发降级处理。

上述阈值中 T1/T2 已冻结，其他为 **TBD 项**，需在原型运行与实测中逐步校准。

---

### 2.8 Evolution Assumptions

- V1 以 **简洁性与功能可用性** 为优先；
- 自动降级、自愈与无人值守能力为后续版本目标；
- 架构必须为上述能力 **预留接口与扩展点**，但 V1 不强制实现。

---

### 2.9 外部接口一致性冻结（P0）

本系统的“三方一致性闭环”以 **已定版协议为唯一事实源**。顶层架构层面冻结以下约束，用于消除联调阶段的“硬不一致”：

#### 2.9.1 SPS↔DACS（射频前端 FPGA）— 协议 V3.1 一致性（MUST）

- **[DACS-CTRL-1] MUST**：控制平面运行在 UDP 上，必须实现 **CRC32C 校验 + ACK 回执闭环 + 超时重传**。
- **[DACS-CTRL-2] MUST**：必须实现 **幂等桥接与去重边界**，并以 DACS 侧幂等键（`SourceID + ControlEpoch + SequenceNumber`）为准。
- **[DACS-DATA-1] MUST**：数据平面采用 **应用层分片/重组**，后端接收端必须按 V3.1 的 `FragIndex/TotalFrags/TailFragPayloadBytes` 规则解析。
- **[DACS-DATA-2] MUST**：重组超时冻结为 **`T_reasm = 100ms`**；超时后必须 **按缺失分片补零输出** 并标记 `IncompleteFrame`；迟到分片必须丢弃并计数。
    - 推论：本架构的“实时优先、允许丢弃低价值数据”只适用于 **重组完成后** 的后续流水线阶段；不得在接收端以更短超时（例如 10ms）提前丢弃“不完整但按协议仍需输出”的 CPI。
- **[DACS-DATA-3] MUST**：必须提取并保留尾包携带的 **Execution Snapshot**，作为“控制是否按期生效”的一致性证据链输入。
- **[DACS-TIME-1] MUST**：必须识别 DACS 的时间来源/质量（如 Q0/Q1/Q2 等），并在系统状态机与对外快照中体现“定时生效可用/不可用”。

#### 2.9.2 SPS↔HMI（显控）— 协议 V1 一致性（MUST）

- **[HMI-CTRL-1] MUST**：TCP 控制面必须在握手后下发 `StateSnapshot`；显控在快照应用前禁止业务控制（否则后端应拒绝并返回可解释错误）。
- **[HMI-CTRL-2] MUST**：控制请求以 `command_id` 为幂等键（TTL ≥ 10min），后端必须返回一致响应并实现断链重连后的安全恢复。
- **[HMI-DATA-1] MUST**：UDP 数据面 `TrackData` 必须满足 **单 datagram、应用层不分片**。
- **[HMI-DATA-2] MUST**：`TrackData` 发送负载上限冻结为 **`MaxUdpPayloadBytes = 1200B`**；超限时必须执行选择性输出/截断，并显式标记（例如 `is_truncated/drop_reason/dropped_count`）。
- **[HMI-DATA-3] MUST**：`TrackData` 必须携带并保持语义一致：`backend_instance_id`、`frame_seq` 单调递增、`clock_domain`、`data_timestamp(ns)`（新鲜度主判据）。

#### 2.9.3 时间域与状态域桥接（MUST）

- **[BRIDGE-1] MUST**：后端必须提供从 DACS 时间质量/来源 → HMI `clock_domain` 的稳定映射，并在 `StateSnapshot/TrackData` 中保持一致。
- **[BRIDGE-2] MUST**：HMI `command_id` ↔ DACS（`ControlEpoch, SequenceNumber`）必须建立可审计映射，并将多事务 ACK 聚合为单一可解释 `ControlResponse`。

# 3. 系统架构风格（Architecture Style）

## 3.1 架构范型与核心选择

本系统采用 **实时数据处理架构（Real-Time Data Processing Architecture）** 的范型，其核心设计包括三个相互强化的维度：

### 3.1.1 处理平面分离（Data Plane / Control Plane Separation）

系统明确采用 **双平面分离架构**，这是高吞吐、低延迟系统的通行设计模式：

````txt
┌─────────────────────────────────────────────────────────────┐
│  Control/Management Plane (Node0: CPU0-15)                  │
│  ├─ 系统编排与决策 (FSM, Commands)                          │
│  ├─ 非实时服务 (Logging, Monitoring, Config)               │
│  └─ 网络控制面 (TCP, State Sync)                            │
│                                                               │
│  ↕ (Async Event Bus, 低频同步通道)                          │
│                                                               │
│  Data Plane (Node1: CPU16-31)                                │
│  ├─ 接收 (Receiver M01)                                      │
│  ├─ 实时处理 (Signal/Data Proc M02/M03)                     │
│  └─ 数据发送 (Gateway M04)                                   │
│                                                               │
│  ↕ (UDP TrackData, 高速数据面)                              │
│                                                               │
│  显控系统 / 上级应用                                         │
└─────────────────────────────────────────────────────────────┘
````

**设计意图**：
- **隔离性**：实时数据处理的抖动与延迟不受控制面服务（磁盘 IO、网络延迟、动态分配）影响
- **独立性**：数据平面可在无控制面连接情况下独立运行并产出有效 TrackData
- **可扩展性**：控制平面的服务（监控、日志、配置）可独立增强而不影响核心处理

**约束推论**：

- 控制面可以承载“系统级授时服务/状态裁决/协议桥接”（例如 PTP/NTP 守护、控制闭环聚合），但**不得**将其引入到逐帧实时决策热路径中（避免抖动传播到数据面）。
- 数据平面**禁止阻塞式磁盘 IO**、禁止复杂 RPC/重连逻辑等高抖动操作；但为了满足 SPS↔DACS 的数据入口与心跳可观测性，数据面允许实现最小化、非阻塞的网络收包与协议解析（必要时可演进为旁路收包）。
- 跨平面参数修改必须通过版本化快照/消息投递完成，不得以共享可变状态直接“同步修改”。

---

### 3.1.2 流式并行处理（Streaming Parallelism Pattern）

系统采用 **流式管道并行** 而非批处理并行，这是适应实时约束与低延迟诉求的必然选择：

````mermaid
graph LR
    subgraph RCV_STAGE["Pipeline Stage 1: Receiver"]
        R1["Receiver_1"]
        R2["Receiver_2"]
        R3["Receiver_3"]
    end

    subgraph SIG_STAGE["Pipeline Stage 2: SignalProc"]
        GPU["GPU Dispatcher<br/>3×CUDA Streams"]
    end

    subgraph DAT_STAGE["Pipeline Stage 3: DataProc"]
        FUSE["Fusion Engine"]
    end

    subgraph GW_STAGE["Pipeline Stage 4: Gateway"]
        NET["Network Output"]
    end

    RCV_STAGE -->|RawCPI Stream| GPU
    GPU -->|Plots Stream| FUSE
    FUSE -->|TrackData Stream| NET
    NET -->|UDP/TCP| OUTPUT["显控系统"]

    style RCV_STAGE fill:#fff9c4
    style SIG_STAGE fill:#ffccbc
    style DAT_STAGE fill:#c8e6c9
    style GW_STAGE fill:#b3e5fc
    style OUTPUT fill:#f3e5f5
````

**管道特征**：
- **无缝连接**：各阶段间采用有界缓冲队列（SPSC RingBuffer），而非文件、消息队列等离散存储
- **持续流动**：不存在 " 批次 " 概念，数据按帧/CPI 粒度持续进入、处理、输出
- **背压处理**：当下游处理能力不足时，采用数据丢弃（drop-oldest）或合并（latest-state-wins）而非阻塞上游

**与批处理对比**：

| 维度 | 批处理 | 流式管道（本系统） |
|------|--------|-------------------|
| 延迟特征 | 累积延迟（高） | 单帧延迟（低） |
| 缓冲需求 | 大缓冲（等待批次） | 小缓冲（数帧） |
| 背压处理 | 等待或拒绝 | 丢弃或合并 |
| 资源利用 | 周期性峰值 | 平稳均匀 |

**流式管道的启发**：本系统在每个管道阶段间的转移点采用 **无锁环形队列（lock-free circular buffer）** 而非通用消息队列，这是吞吐与延迟的关键权衡。

**GPU 计算任务的超时管理**：

为避免单个 GPU 内核长时间执行导致整个数据流水线停滞，系统采用以下机制：

````mermaid
graph LR
    subgraph GPU_DISPATCH["GPU Dispatcher (Node1)"]
        SUBMIT["Task Submit"]
        MONITOR["Timeout Monitor"]
        ACTION["Remedial Action"]
    end

    SUBMIT -->|提交后记录时间戳| MONITOR

    MONITOR -->|轮询 GPU 事件| CHECK{"计算耗时<br/>超过阈值？"}

    CHECK -->|NO| DONE["Task Complete<br/>回收资源"]

    CHECK -->|YES<br/>≥ 1.5×帧周期| ACTION

    ACTION --> CHOICE{"后续动作"}

    CHOICE -->|算法支持分段| PREEMPT["中断当前任务<br/>标记帧无效"]
    CHOICE -->|无法中断| TIMEOUT["记录超时告警<br/>继续等待完成"]

    PREEMPT --> RELEASE["释放 GPU 用于下一帧"]
    TIMEOUT --> RELEASE

    RELEASE --> TRIGGER{"超时频繁?"}

    TRIGGER -->|YES| DEGRADE["触发系统降级<br/>启动 GPU 监控告警"]
    TRIGGER -->|NO| DONE

    style MONITOR fill:#fff9c4
    style ACTION fill:#ffccbc
    style DEGRADE fill:#ffcdd2
````

**架构约束**：
- **[GPU-1] MUST**：GPU Dispatcher 对每个提交的计算任务启动看门狗定时器
  - 阈值设为帧处理周期的 1.5 倍（例如 20ms 帧周期 → 30ms 超时）
  - 用途：快速检测异常长耗时的内核执行
- **[GPU-2] SHOULD**：算法实现支持分段计算时，系统应在超时时：
  - 发起 GPU 命令流取消或中断信号（如果硬件支持）
  - 标记当前帧计算结果无效
  - 立即释放 GPU 供后续帧使用
  - 该帧对应的 Plots 被丢弃（但不影响下一帧处理）
- **[GPU-3] MUST**：若某阵面的 GPU 计算连续 N 次（建议 N=3）超时，系统应：
  - 向 Orchestrator 上报 `GPU_COMPUTE_TIMEOUT` 告警
  - 触发系统进入 Degraded 状态
  - 可选降频该阵面数据入队或降低信号处理精度
- **[GPU-4] MUST**：无法中断的长耗时情况下，不应无限等待
  - 若超时 ≥ 2×帧周期，系统应主动进入故障恢复流程（见 3.4 节）
  - 而非让整个流水线陷入死锁

**设计背景**：
- NVIDIA GPU 硬件本身**不支持强制任务抢占**
  - 一旦内核启动，除非完成或驱动 Timeout Detection & Recovery (TDR) 触发，否则持续占用
  - 这是 GPU 实时性的固有限制
- 系统通过软件层面的监控与主动丢弃来规避这一问题
  - 即允许某帧的计算结果被放弃以保证后续帧的及时处理
  - 这与 " 优先保证实时性而非完整性 " 的设计哲学一致

**验收要点**：
- 在联调测试中构造极端场景（例如某帧信号特殊导致 GPU 计算异常耗时）
- 验证系统能及时检测并触发告警/降级而非长时间卡顿
- 确保 GPU 超时不会导致后续数据处理中断

### 3.1.3 NUMA 感知内存架构（NUMA-Aware Memory Architecture）

系统的内存分配与线程绑定均围绕 NUMA 拓扑进行，而非假设统一内存模型：

````mermaid
graph TB
    subgraph NUMA0["NUMA Node 0 (CPU0-15)"]
        direction TB
        POOL0["本地内存池<br/>ControlEvents<br/>Logs / Config"]
        THD0["控制面线程<br/>Orchestrator<br/>Logger / ConfigMgr"]
        POOL0 ---|本地访问| THD0
    end

    subgraph NUMA1["NUMA Node 1 (CPU16-31)"]
        direction TB
        POOL1["本地内存池<br/>RawData Buffers<br/>Track Pool<br/>GPU Pinned Mem"]
        THD1["数据面线程<br/>Receiver ×3<br/>SignalProc / DataProc"]
        POOL1 ---|本地访问| THD1
    end

    NUMA0 -->|~1000ns远程访问延迟| NUMA1

    style NUMA0 fill:#e3f2fd
    style NUMA1 fill:#fff8e1
````

**设计原则**：
- **本地优先**：所有频繁访问的数据结构必须从线程所在 NUMA 节点的本地内存池分配
- **跨节点最小化**：禁止数据面线程跨 NUMA 节点分配内存（可导致 5 倍延迟）
- **预分配冻结**：在启动期静态确定每个内存池的大小，运行期禁止动态扩展（规避碎片化与锁竞争）

**约束推论**：
- 热路径中禁止动态 malloc/new，所有对象必须从预分配池租赁
- 数据平面与控制平面的共享数据（如配置）需要通过原子操作或消息队列而非直接共享内存

## 3.2 数据流模式

### 3.2.1 拓扑级数据流（Topology-Level Data Flow）

系统的宏观数据流采用 **线性管道 + 融合点** 的组合：

````mermaid
    A2["Array_2 Raw Stream"] --> RCV2["Receiver_2"]

    GPU --> ARR2["Array_2 Plots"]
    GPU --> ARR3["Array_3 Plots"]

    ARR1 --> DP["DataProc<br/>航迹生成+融合"]
    ARR2 --> DP
    ARR3 --> DP

    DP --> TRK["TrackData Stream"]

    TRK --> GW["Gateway"]

    GW --> OUT["TCP/UDP Output"]

    style MQ fill:#fff9c4
    style DISP fill:#ffccbc
    style DP fill:#c8e6c9
    style TRK fill:#b3e5fc
    style OUT fill:#ffe0b2
````

**设计特点**：
- **三路独立接收**：避免 Receiver 内部复杂的协议处理竞争
- **单点融合**：所有三阵面的处理结果在 DataProc 中集中融合，确保全局一致性
- **轮询而非多生产者**：融合点采用轮询多个 SPSC 队列而非 MPMC，规避锁竞争

**跨阵面航迹融合与连续性保证**：

在三面阵无缝覆盖的部署中，目标在阵面边界处的转移需要特殊处理以保持航迹连续：

````mermaid
graph LR
    subgraph ARR1["Array_1<br/>120°扇区"]
        T1["Track_ID=1001"]
    end

    subgraph ARR2["Array_2<br/>120°扇区"]
        T2["Track_ID=?"]
    end

    ARR1 -->|Target 跨边界<br/>转移| ARR2

    T1 -->|关联匹配| T2

    subgraph FUSION["DataProc 融合引擎"]
        MATCH["跨阵面关联算法<br/>状态外推 + 门限匹配"]
        MERGE["统一 Track ID<br/>复用原有 1001"]
    end

    MATCH --> MERGE

    style FUSION fill:#c8e6c9
````

**架构约束**：
- **[FUSION-1] MUST**：DataProc 融合器需实现 **跨阵面航迹关联算法**
  - 当某阵面内航迹消失且相邻阵面出现新航迹时，基于目标位置/速度的外推预测进行匹配
  - 若匹配置信度超阈值（建议 ≥ 0.8），则判定为同一目标
- **[FUSION-2] MUST**：跨阵面关联成功时，新阵面的 Track ID 应复用原阵面的 Track ID
  - 确保 TrackData 输出中同一真实目标的 ID 始终保持一致
  - 避免 " 同一目标被视为多条航迹 " 的现象
- **[FUSION-3] SHOULD**：关联失败时（匹配度不足），系统可：
  - 保留原航迹的 COASTING（外推）状态直到确认消失
  - 在新阵面给该目标分配新 Track ID（记录其与前一阵面航迹的关联信息）
  - 待后续帧再次尝试匹配

**关键接口**：
- DataProc 的融合引擎需要暴露以下接口给 Orchestrator / 监控：
  - `cross_array_association_success_rate`：跨阵面关联成功率指标
  - `cross_array_track_id_collision_count`：Track ID 冲突计数（应始终为 0）

**验收要点**：
- 目标在三个阵面的边界处连续通过，航迹 ID 始终不变
- 在离线日志中验证目标状态的连贯性（无跳变或重复分配）

### 3.2.3 背压与流控策略（Backpressure Pattern）

系统不使用阻塞式流控（backpressure as blocking），而是 **丢弃优先级流控**：

````mermaid
graph TD
    subgraph CONDITION["生产速率 > 消费速率 时"]
        direction LR
        CHECK["检测背压"]
    end

    CHECK --> DECIDE{"根据数据类型<br/>决定丢弃策略"}

    DECIDE -->|Raw IQ| DROP1["drop-oldest<br/>中间产物，完全可丢"]
    DECIDE -->|Plots| DROP2["drop-oldest/SNR<br/>中间产物，部分可丢"]
    DECIDE -->|TrackData| DROP3["latest-state-wins<br/>业务必达，不丢<br/>但降更新率"]
    DECIDE -->|ControlEvents| BLOCK["阻塞生产者<br/>控制信息，绝对可靠"]

    DROP1 --> RESULT1["禁止阻塞 Receiver"]
    DROP2 --> RESULT2["禁止阻塞 SignalProc"]
    DROP3 --> RESULT3["禁止中断输出"]
    BLOCK --> RESULT4["允许阻塞<br/>低频操作"]

    style CONDITION fill:#fff9c4
    style DECIDE fill:#ffccbc
    style DROP1 fill:#ffcdd2
    style DROP2 fill:#ffcdd2
    style DROP3 fill:#c8e6c9
    style BLOCK fill:#bbdefb
````

**流控分层**：

| 数据类型 | 生产速率过高时 | 原因 | 约束 |
|----------|---------------|------|------|
| Raw IQ | drop-oldest | 中间产物，可完全丢 | 禁止阻塞 Receiver |
| Plots | drop-oldest/SNR | 中间产物，可部分丢 | 禁止阻塞 SignalProc |
| **TrackData** | latest-state-wins | 业务必达：尽力持续发布最新快照（语义不丢），允许降更新率 | 禁止因背压阻塞热路径 |
| ControlEvents | 阻塞生产者 | 控制信息，绝对可靠 | 允许阻塞（低频） |

**设计哲学**：宁愿丢弃 N 帧的点迹，保证最新的航迹快照持续发布；而非因缓冲溢出导致 TrackData 在后端侧排队延迟。

补充说明：TrackData 经 UDP 下发到显控时，链路本身允许丢包/乱序；“不可丢”指后端应持续生成并发布最新快照，而不是要求网络传输 0 丢包。

**TrackData 流控的特殊处理**：

与其他数据类型不同，TrackData 作为业务必达输出，采用以下三层流控策略：

````mermaid
graph TD
    subgraph TRK_FLOW["TrackData 发布队列状态"]
        QN["队列未满<br/>正常发布"]
        QW["队列接近满<br/>触发降频"]
        QF["队列已满"]
    end

    subgraph ACTION["对应动作"]
        AN["全速输出<br/>frame_seq 递增"]
        AW["降频输出<br/>OUTPUT_THROTTLED=true<br/>跳过低优先级帧"]
        AF["非阻塞投递失败<br/>触发降级告警"]
    end

    QN --> AN
    QW --> AW
    QF --> AF

    AF -->|持续| DEGRADE["system_state → Degraded<br/>启动降级流控"]
    DEGRADE -->|≥2s 无 TrackData| FAULT["system_state → Fault<br/>停止输出"]

    style QN fill:#c8e6c9
    style QW fill:#fff9c4
    style QF fill:#ffcdd2
````

**约束条款**：
- **[FLOW-1] MUST**：TrackData 发布队列需设置有限容量，基于系统可容忍的最大延迟计算
  - 建议大小：容纳 3–5 帧的 TrackData（与业务响应时间对齐）
- **[FLOW-2] MUST**：当队列深度 ≥ 80% 时，系统标记 `OUTPUT_THROTTLED=true` 并开始丢弃优先级最低的帧
  - 优先保留最新的完整航迹快照，不保留中间帧
- **[FLOW-3] MUST**：M03 线程尝试入队若超时（例如 100μs）未成功，应：
  - 不阻塞，立即返回失败状态
  - 向 Orchestrator 上报 `OUTPUT_QUEUE_FULL` 事件
  - Orchestrator 收到后触发系统进入 Degraded 状态
- **[FLOW-4] SHOULD**：在 Degraded 期间，若队列仍继续溢出，系统应采用 **latest-state-wins 合并策略**
  - 对同一 `frame_seq` 的重复更新进行覆盖
  - 而非盲目丢弃或阻塞

**验收要点**：
- 模拟显控离线 / 网络拥塞等背压场景，验证系统能优雅进入 Degraded 而不阻塞热路径
- 确保在这种情况下 TrackData 仍能断续输出（频率降低但不中断）

### 3.3.1 并发范式选择

### 3.3.2 线程绑定与调度

系统采用 **静态线程绑定**，每个逻辑处理单元映射到固定的 CPU 核：

````mermaid
graph LR
    subgraph NUMA_N1["NUMA Node 1 (CPU16-31) - 实时计算"]
        R1["Receiver_1<br/>CPU16"]
        R2["Receiver_2<br/>CPU17"]
        R3["Receiver_3<br/>CPU18"]
        SP["SignalProc<br/>CPU19-22"]
        DP["DataProc<br/>CPU23-24"]
        DISP["GPU Disp<br/>CPU25"]
        RSV["Reserved<br/>CPU26-31"]
    end

    subgraph NUMA_N0["NUMA Node 0 (CPU0-15) - 管理/控制"]
        ORH["Orchestrator<br/>CPU0"]
        LOG["Logger<br/>CPU1"]
        MON["Monitor<br/>CPU2"]
        GW["Gateway<br/>CPU3"]
        RSV0["Reserved<br/>CPU4-15"]
    end

    R1 -.->|SPSC Queue| SP
    R2 -.->|SPSC Queue| SP
    R3 -.->|SPSC Queue| SP
    SP -.->|SPSC Queue| DP
    DP -.->|UDP Frames| GW

    style NUMA_N1 fill:#fff8e1
    style NUMA_N0 fill:#e3f2fd
````

**特点**：
- ✅ 固定映射 → 缓存行为一致 → 延迟确定
- ✅ 无调度开销 → 上下文切换最小
- ⚠️ 需要显式绑定配置（不依赖 OS 调度)
- ⚠️ 不支持动态负载均衡（OK，因为系统是实时处理)

**资源预算与约束**：

当前配置基于以下假设：

| 模块 | 核心数 | 理由 | 风险 |
|------|--------|------|------|
| **Receiver ×3** | 3 核 | 三阵面独立接收，各占一个专属核 | ⚠️ 若任一路数据突增可能溢出；如接收线程上下文切换过于频繁会影响数据率 |
| **SignalProc** | 4 核 | 支持 GPU 前后处理的 CPU 侧并行 | 正常 |
| **DataProc** | 2 核 | 航迹生成与融合的串行处理 | 根据融合算法复杂度可能需要增加 |
| **GPU Dispatcher** | 1 核 | 统一调度三条 CUDA 流 | ⚠️ 单点，若分发逻辑复杂可能成为瓶颈 |
| **Reserved** | 6 核 | 应对负载波动与扩展 | 可作为缓冲 |

**关键考量**：
- **[RESOURCE-1] SHOULD**：根据实际测试数据调整资源分配
  - 若 Receiver 在满载（10GbE ×3 路高数据率）时出现队列溢出
  - 应增加 Receiver 核心数至 4 或更多
  - 从 Reserved 池中补充
- **[RESOURCE-2] MUST**：禁止过度超分
  - 不允许多条接收线程共用同一物理核心（无虚拟化超分）
  - 物理核映射应一一对应

**性能调优指标**：

在原型运行中应持续监控以下指标来评估资源是否充足：
- Receiver 队列深度：应在 1–3 帧范围内正常波动，不应频繁接近满队列
- CPU 占用率：各线程占用应在 70–90% 范围，留出应对突发的裕度
- 上下文切换频率：应尽量低，表示无过度竞争

若发现资源不足，应在下一个版本迭代中动态调整核心分配。

### 3.3.3 共享状态管理

系统中少数需要跨线程/跨平面的共享状态通过以下机制管理：

| 共享数据 | 保护机制 | 频率 | 位置 |
|----------|---------|------|------|
| 系统 FSM | 单点写（Orchestrator） | 低频 | 控制平面 |
| 配置参数 | 版本化快照 + RCU | 极低频 | 两平面共享 |
| 性能指标 | 原子计数器 | 高频 | 数据平面写，控制平面读 |
| TrackData 缓存 | Latest-wins + 时间戳 | 高频 | 网关输出 |

**设计原则**：
- 最小化共享（能不共享就不共享）
- 写入端尽可能单一
- 读取端无锁（通过原子操作或版本化快照）

## 3.4 故障恢复架构

### 3.4.1 分层故障处理（Layered Fault Handling）

系统采用 **三层故障响应**，而非单点式故障转移：

````mermaid
graph TD
    EVENT["故障事件"]

    EVENT --> L1["第一层<br/>本地缓冲与丢弃<br/>毫秒级"]

    L1 --> L1_ACT["队列溢出 → drop-oldest<br/>资源不足 → 缓存吸收<br/>短期压力 → 吸收"]

    L1_ACT -->|持续未缓解| L2["第二层<br/>进入 Degraded<br/>秒级"]

    L2 --> L2_ACT["持续溢出 → 告警<br/>TrackData 中断 < 2s → 警告<br/>启动降级动作"]

    L2_ACT -->|还未恢复| L3["第三层<br/>进入 Fault<br/>持续"]

    L3 --> L3_ACT["≥2s 无 TrackData → 不可恢复<br/>关键资源耗尽 → 停止输出<br/>需人工/自动重启"]

    style L1 fill:#fff9c4
    style L1_ACT fill:#ffecb3
    style L2 fill:#ffccbc
    style L2_ACT fill:#ffab91
    style L3 fill:#ffcdd2
    style L3_ACT fill:#ef9a9a
````

**启示**：分层响应避免了 " 中断即故障 " 的过度反应，同时保证了真正的业务中断会被及时检测。

### 3.4.2 恢复策略

系统明确区分两种恢复能力：

| 恢复类型 | 触发条件 | 动作 | 谁来执行 |
|----------|---------|------|----------|
| **自动降级恢复** | TrackData 中断 <2s | 清除积压、启动流控 | 数据平面（自主) |
| **有序重启恢复** | 系统进入 Fault | 完整 Init 序列 | 显控或人工（外驱) |

**注意**：系统 V1 不实现 " 自动重启 " 能力（该能力属于 v2+），仅预留接口。

## 3.6 架构风格小结

| 维度 | 选择 | 原因 |
|------|------|------|
| **处理范型** | 流式管道并行 | 适应实时、低延迟约束 |
| **平面分离** | 数据/控制平面 | 隔离实时处理与服务 |
| **内存模型** | NUMA 感知分池 | 规避跨节点延迟 |
| **并发方式** | 无锁/低锁 + 静态绑定 | 确定性与可预测性 |
| **流控策略** | 优先级丢弃 | 保证最新态势而非完整性 |
| **故障恢复** | 分层响应 + 有序重启 | 防止过度反应与级联故障 |
| **可观测性** | 分层度量 + 审计链 | 支持运维诊断与问题复现 |
