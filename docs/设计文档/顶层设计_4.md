## 1. 文档目的与适用范围

本篇为 " 工程约束总表 + 可观测性/可复现契约 + 资源与时延预算 + 安全边界声明 " 的统一入口。

**适用范围**：
- 工程落地与联调的唯一裁决来源
- FAT/SAT 验收依据
- AI 生成代码时的约束规范
- 系统状态判据（Running/Degraded/Fault）的工程绑定

**本篇不做**：算法推导、实现细节（除验收/打点定义必须外）。

## 2. 统一术语与 ID 模型

### 2.1 系统状态（冻结，不可新增）

| 状态 | 语义 |
|------|------|
| `Init` | 系统初始化中 |
| `Standby` | 就绪待命，可接受启动命令 |
| `Running` | 正常运行，实时输出 TrackData |
| `Degraded` | 降级运行，部分能力受限但仍可输出 |
| `Fault` | 故障，停止对外输出 |

**说明**：`Shutdown` 不属于 system_state 值域；系统关闭由 lifecycle 事件（如 "Exiting" / "Stopped" 或显式关闭命令）驱动。这保证了对外协议中 system_state 始终为 5 态，避免与第一篇、第二篇的状态转移表出现矛盾。
**转移规则与裁决权**：见 [分块设计文档/顶层设计_1.md](分块设计文档/顶层设计_1.md#ssot-fsm-transitions)。

### 2.2 连接状态

| 状态 | 语义 |
|------|------|
| `conn_state=Up` | 连接正常 |
| `conn_state=Down` | 连接断开/拒绝 |

### 2.3 时间戳语义（三时间戳体系，冻结）

三时间戳的定义、单调关系与精度约束统一见 [分块设计文档/顶层设计_3.md](分块设计文档/顶层设计_3.md#ssot-time-triple)。

### 2.4 统一 ID 模型

| ID 名称 | 语义 | 生成时机 | 生命周期 |
|---------|------|----------|----------|
| `run_id` | 一次运行实验的唯一标识 | 每次进入 `Running` 时生成 | 固定至退出 `Running` |
| `backend_instance_id` | 进程实例标识，检测重启 | 每次进程启动时生成 | 进程生命周期 |
| `trace_id` | 跨模块链路追踪 ID | 每个可追踪事件生成（可由后端生成/派生） | 事件级 |
| `command_id` | 控制面幂等去重业务命令 ID | 由客户端提供 | 命令级 |

**[ID-1] MUST**：`run_id` 作为所有输出（日志/录制/metrics）的顶层关联键。
**[ID-2] MUST**：`trace_id` / `command_id` 作为事件级关联键。
**[ID-3] MUST**：四类 ID 语义不可混用、不可替代。

### 2.5 关键哲学（贯穿全文）

- **热路径不阻塞**：热路径禁止 sleep/condvar/sem_wait 等被动等待
- **旁路 best-effort**：录制/日志/监控均为旁路异步，拥塞时可丢弃
- **drop 策略可观测但不视为故障**：除非触发可信性判据（如 `authoritative_time_state=INVALID` 或 `time_state_level=0`）

## 3. C 章汇总：运行清单与控制面安全边界

### 3.1 C2：运行清单（run manifest）

#### 3.1.1 定位

run manifest 是 " 审计账本 "，不是配置文件。必须满足：
1. **完整**：能复刻 " 跑的是什么 "
2. **一致**：通过 `run_id`/`trace_id` 与日志、Raw IQ、metrics 关联
3. **不可抵赖**：落盘原子，字段不可运行中悄悄改写

#### 3.1.2 条款

**[C2-1.1] MUST**：每次系统进入 `Running` 必须生成唯一 `run_id`，并在该次 `Running` 生命周期内保持不变。
*验收*：进入 Running 后检查 manifest 文件存在且 run_id 非空。

**[C2-1.2] MUST**：每次进程启动必须生成新的 `backend_instance_id`，并写入 run manifest。
*验收*：重启两次，验证每次 backend_instance_id 不同。

**[C2-1.3] MUST**：run manifest 必须落盘为单文件（推荐 JSON），路径包含 `run_id`；写入必须原子化（tmp→fsync→rename）。
*验收*：检查 `runs/<run_id>/manifest.json` 存在；断电测试不出现半写文件。

**[C2-1.4] FORBIDDEN**：运行中原地改写历史字段（例如覆盖原始 build/版本/配置快照）；只允许追加事件记录或写新 revision 文件。
*验收*：对比运行前后 manifest 的基础字段 hash，核心字段不变。

#### 3.1.4 与日志/追踪的关联规则

**[C2-3.1] MUST**：所有结构化日志与关键事件日志必须包含 `run_id`。
*验收*：grep 日志文件，所有条目均含 run_id 字段。

**[C2-3.2] MUST**：控制面事件必须同时记录 `command_id`（客户端提供）与 `trace_id`（链路追踪），并可在日志中互相检索。
*验收*：发送控制命令后，日志可同时查到 command_id 与 trace_id。

**[C2-3.3] SHOULD**：对跨模块链路（Receiver→SignalProc→DataProc→Gateway/Orchestrator）生成 trace span。
*验收*：trace 导出文件包含至少三段 span。

#### 3.1.5 故障/诊断闭环

**[C2-4.1] MUST**：run manifest 必须被纳入 " 最小诊断包 "，并在 `Fault` 时保证可取到最后一次已落盘版本。
*验收*：触发 Fault 后，诊断包内含完整 manifest。

**[C2-4.2] SHOULD**：Raw IQ/关键录制文件命名或元数据中包含 `run_id`，用于从数据反查 manifest。
*验收*：录制文件路径或元数据含 run_id。

### 3.2 C3：控制面安全边界

#### 3.2.1 安全边界声明（V1）

**[C3-0.1] MUST**：控制面属于高风险入口；无论是否启用加密/认证，都必须具备 " 最小安全闭环 "（C3-1）。
*验收*：C3-1 各条款全部通过。

**[C3-0.2] MUST**：V1 若不启用 TLS/mTLS，必须在顶设中明确声明：
- 控制面仅允许在**受控内网/隔离网络**使用
- 不承诺抵御中间人/窃听/伪造来源攻击
- 风险由部署侧（网络隔离/隧道）承担
- 超出边界的部署属于 **FORBIDDEN** 使用方式

*验收*：顶设文档包含上述声明。

**[C3-0.3] SHOULD**：推荐部署侧采用外部安全承载（VPN/WireGuard/IPsec/SSH Tunnel）作为 V1 的加密与认证手段。
*验收*：部署文档明确安全承载方案。

#### 3.2.2 V1 最小安全闭环

**[C3-1.1] MUST**：控制面监听地址必须可配置，默认不得监听 `0.0.0.0`；默认仅绑定管理网口或回环地址。
*验收*：默认启动后 netstat 检查监听地址非 0.0.0.0。

**[C3-1.2] MUST**：支持 `ip_allowlist`（白名单）。未命中白名单的连接必须拒绝（conn_state=Down）。
*验收*：非白名单 IP 连接被拒绝，审计日志记录拒绝事件。

**[C3-1.3] MUST**：限制资源消耗：最大并发连接数、连接建立速率、单连接命令速率（rate limit），并记录拒绝原因。
*验收*：高频连接/命令触发 rate limit，系统不中断。

**[C3-1.4] MUST**：对每条 ControlRequest 进行审计记录：`run_id / trace_id / command_id / src_ip:port / command_type / params_digest / result(ACK|NACK|TIMEOUT) / reason_code / ts`。
*验收*：任意控制命令可在审计日志中找到完整记录。

**[C3-1.5] SHOULD**：将 " 危险命令 "（如重启、清配置、强制模式切换等）纳入额外保护：至少需要 `dangerous_ops_enabled=true` 才允许执行；默认关闭。
*验收*：dangerous_ops_enabled=false 时，危险命令返回 NACK（reason_code=OP_NOT_ALLOWED）。

#### 3.2.3 安全模式与接口预留（面向 V2/V3）

**[C3-2.1] MUST**：预留统一安全配置入口：
- `security.mode = OFF | TRANSPORT_TLS | TRANSPORT_MTLS | EXTERNAL_TUNNEL`
- `security.require_authz = true/false`

*验收*：配置项存在且可被 manifest 记录。

**[C3-2.2] MUST**：当 `security.mode=TRANSPORT_TLS/MTLS` 时，必须支持以下配置项（V1 可暂不实现，但字段与行为冻结）：
- `tls.ca_path / tls.cert_path / tls.key_path`
- `tls.min_version`（例如 TLS1.2+）
- `tls.client_auth_required`（mTLS）

*验收*：配置 schema 包含上述字段。

**[C3-2.3] SHOULD**：支持证书轮换策略之一：
- A) `Standby` 状态加载新证书并切换
- B) 双证书窗口（old+new 并存）+ 定时切换
- C) 滚动重启

*验收*：文档声明采用的轮换策略。

#### 3.2.4 不做的风险与边界

**[C3-3.1] MUST**：若 V1 不启用 TLS/mTLS，则必须列出明确风险：窃听、伪造来源、中间人、命令注入；并写明缓解措施只能来自**部署隔离/外部隧道**。
*验收*：顶设文档包含风险声明与缓解措施。

**[C3-3.2] FORBIDDEN**：在不可信网络（公网/跨安全域未隔离网络）直接暴露控制面端口。
*验收*：部署 checklist 包含网络隔离检查项。

## 4. D 章汇总：可观测性最小集合与预算

### 4.1 总则

**[D-0.1] MUST**：定义并实现 Metrics/Logs/Tracing 的最小集合，用于：
1. `Running/Degraded/Fault` 判据自动化
2. 性能回归（p95/p99）
3. 现场取证与复现（与 run manifest 关联）

*验收*：D-1/D-2/D-3 各条款全部通过。

**[D-0.2] MUST**：所有观测输出必须包含 `run_id`；关键事件必须包含 `trace_id`。
*验收*：抽样检查日志/metrics/trace 均含 run_id。

**[D-0.3] FORBIDDEN**：在热路径同步落盘日志/同步上报网络指标；热路径只允许原子计数器/无锁 ring buffer 写入。
*验收*：热路径代码审查无同步 IO 调用。

**[D-0.4] MUST**：冻结观测开销上限（见 D-4），并在 CI/回归测试中验收。
*验收*：CI 包含观测开销回归测试。

### 4.3 Logs 最小事件

**[D-L1] MUST**：结构化 " 状态切换日志 "：每次 `system.state` 变化记录（含 from/to、reason、trace_id）。
*验收*：状态切换后日志可查到完整记录。

**[D-L2] MUST**：结构化 " 控制命令审计日志 "：含 `command_id、trace_id、command_type、params_digest、result、reason_code、latency_ms、src`。
*验收*：控制命令可在日志中找到完整审计记录。

**[D-L3] MUST**：结构化 "SEV 事件日志 "：含 sev、触发条件摘要（枚举化）、关联关键指标快照。
*验收*：SEV 事件触发后日志包含完整上下文。

**[D-L4-1] MUST**：结构化 "DACS 控制审计日志"：每条 DACS 控制事务必须记录 `command_id（映射源）、array_id、ControlEpoch、SequenceNumber、AckResult（APPLIED/APPLIED_LATE/NOT_APPLIED/OVERRUN/TIMEOUT）、latency_ms`。用于控制闭环根因追溯。
*验收*：任意 DACS 控制事务可在日志中查到完整审计记录。

**[D-L4-2] SHOULD**：结构化 "Execution Snapshot 不一致日志"：当 Snapshot 与下发参数不一致时，记录 `array_id、FrameCounter、预期参数摘要、实际 Snapshot 摘要、偏差原因（APPLIED_LATE/OVERRUN 等）`。
*验收*：Snapshot 不一致事件可在日志中查到。

**[D-L4] FORBIDDEN**：热路径按帧/按包 INFO 日志；热路径日志仅允许：
- ERROR/FATAL 类事件（且要限频）
- 旁路缓冲（ring buffer）写入，异步落盘

*验收*：代码审查热路径无 INFO 级别同步日志。

### 4.4 Tracing 最小集合与采样策略

**[D-T1] MUST**：控制命令链路必须有 trace（至少单 trace_id 贯穿）：Conn→Parse→Validate→Execute→Respond。
*验收*：控制命令可在 trace 系统中端到端追踪。

**[D-T2] SHOULD**：数据面链路提供 " 阶段耗时 "trace（低采样）：Receiver→Dispatcher→Gateway（至少三段）。
*验收*：trace 导出含数据面分段 span。

**[D-T3] MUST**：Tracing 必须支持采样率配置：

| 系统状态 | 采样率要求 |
|----------|-----------|
| `Running` | 默认 ≤ 1%（或按每秒固定上限） |
| `Degraded` | 可提升到 5%（上限仍受预算约束） |
| `Fault` | 触发一次性诊断窗口采样（最后 N 秒全量 span，旁路缓冲异步落盘） |

*验收*：配置项存在且各状态下采样率符合要求。

### 4.5 采样/聚合策略

**[D-S1] MUST**：metrics 采样周期分三档：

| 档位 | 频率 | 适用指标 |
|------|------|----------|
| 快速档 | 10–20 Hz | queue.depth、inflight、time_state、system_state、reasm.active_keys、heartbeat.last_seen_ms |
| 中速档 | 1 Hz | p95/p99 延迟汇总、drop 计数增量、控制面结果增量、reasm.incomplete_total 增量、gateway.truncated_total 增量、heartbeat.loss_total 增量 |
| 慢速档 | 0.1 Hz | 资源池利用率、环境摘要 |

*验收*：指标采集频率符合档位要求。

**[D-S2] MUST**：延迟类指标必须以滑动窗口聚合输出（1s 窗/10s 窗），同时提供 p50/p95/p99；不得输出原始样本流（除非 Fault 诊断窗口）。
*验收*：延迟指标输出为聚合分位数。

### 4.6 观测开销预算

**[D-B1] MUST**：观测 CPU 开销上限：在基准负载下，观测线程总 CPU 使用率 ≤ **1 个逻辑核**（或 ≤ 总 CPU 的 3%）。
*验收*：基准负载下测量观测线程 CPU 占用。

**[D-B2] MUST**：热路径额外开销上限：每帧/每 CPI 的观测写入不得引入任何阻塞；仅允许原子加法与无锁写 ring buffer。
*验收*：对比观测开/关，吞吐与 p99 延迟差值在预算内。

**[D-B3] MUST**：观测内存上限：旁路 ring buffer（日志 +trace）总容量固定（64–256MB 范围），不得无界增长。
*验收*：监控 ring buffer 内存占用，验证上界。

**[D-B4] MUST**：观测 IO 上限：`Running` 状态持续写盘速率 ≤ **X MB/s**（X 在平台基线测定后冻结）；`Degraded/Fault` 允许短时突发，但必须有封顶与回压策略。
*验收*：监控观测 IO 速率，验证上界。

## 5. E 章汇总：资源预算/隔离、录制容量模型、时延预算

### 5.1 E1：CPU/NUMA/IRQ/GPU 预算与隔离

#### 5.1.1 总原则

**[E1-0.1] MUST**：系统运行形态按 " 任务岛/管理岛 " 物理隔离：
- **Node1 (CPU16-31)**：热路径（数据接入/信号处理/GPU 调度/数据处理；热路径止于 TrackData 发布/提交，不含网关 IO）
- **Node0 (CPU0-15)**：管理面（控制面、网关发送/组包、日志/指标、录制与后台搬移、远程开发/显控转发）

*验收*：Running 时检查关键线程 CPU 亲和性（Node1/Node0 分区正确）。

**[E1-0.2] MUST**：`Running/Degraded/Fault` 状态下必须强制亲和性与 IRQ 策略；不满足则视为 " 未达部署基线 "，拒绝进入 `Running`（或进入 `Degraded` 并上报 reason=AFFINITY_NOT_APPLIED）。
*验收*：亲和性未配置时，系统拒绝进入 Running 或报告 Degraded。

#### 5.1.2 线程/核/NUMA 绑定策略

**热路径（Node1=CPU16-31）**

| 条款 | 组件 | 核分配建议 |
|------|------|-----------|
| **[E1-1.1] MUST** | Receiver（网络接收/解包/入队） | Node1 预留 **3 核**（每阵面接收线程 1 核；busy-poll 不得与其他热路径线程共享） |
| **[E1-1.2] MUST** | GPU Dispatcher | Node1 独占 1 核 |
| **[E1-1.3] MUST** | SignalProc（GPU 前后处理） | Node1 4–8 核（随阵面数缩放） |
| **[E1-1.4] MUST** | DataProc（融合/跟踪） | Node1 2–4 核 |
| **[E1-1.5] SHOULD** | Gateway 数据面发送 | Node0 1 核（允许 IO 抖动，但不得反向阻塞数据面；仅保留最新快照） |
| **[E1-1.6] MUST** | 热路径内存分配 | 默认绑定 Node1（numa preferred/bind） |

**管理与后台（Node0=CPU0-15）**

| 条款 | 组件 | 说明 |
|------|------|------|
| **[E1-1.7] MUST** | 控制面（TCP 控制通道） | 绑定 Node0 |
| **[E1-1.8] MUST** | 录制进程/线程 | 绑定 Node0，采用 NVMe 一级缓冲 + 异步搬移 |
| **[E1-1.9] MUST** | 日志/指标/追踪导出线程 | 绑定 Node0 |
| **[E1-1.10] SHOULD** | 远程开发与显控转发 | 绑定 Node0 |

*验收*：numastat/进程映射验证热路径内存主要落在 Node1。

#### 5.1.3 IRQ 绑定策略

**[E1-2.1] MUST**：10G NIC（Intel X710）IRQ 与接收处理核必须绑定 Node1（CPU16-31）。
*验收*：/proc/interrupts + smp_affinity 验证。

**[E1-2.2] MUST**：1G NIC（远程开发/显控转发）所有 IRQ 必须绑定 Node0（CPU0-15）。
*验收*：/proc/interrupts + smp_affinity 验证。

**[E1-2.3] SHOULD**：非关键 IRQ（磁盘、管理口、BMC 等）尽量从热路径核移出；必要时使用 irqbalance 黑名单。
*验收*：热路径核 IRQ 计数显著低于管理核。

#### 5.1.4 GPU 资源预算与约束

**[E1-3.1] MUST**：Device buffer pool 预分配并设置上限 `device_pool_max_bytes`；禁止运行期无界 cudaMalloc。
*验收*：GPU 显存分配失败触发可观测计数与既定策略。

**[E1-3.2] MUST**：Pinned host pool 预分配并设置上限 `pinned_pool_max_bytes`。
*验收*：pool 上限生效，不出现无界增长。

**[E1-3.3] MUST**：GPU 并发与排队受限：每阵面固定 stream；in-flight 上限=2；Dispatcher 单线程统一提交。
*验收*：inflight 指标不超过上限。

**[E1-3.4] SHOULD**：GPU 显存使用/分配失败计数上报并绑定到 `Degraded/Fault` 判据。
*验收*：分配失败触发 Degraded。

#### 5.1.5 资源超限动作

**CPU 超限**

**[E1-4.1] MUST**：热路径核集合（Node1）出现以下任一条件持续 ≥2s → 进入 `Degraded`：
- Dispatcher/Receiver 的 runqueue 长期拥塞（CPU util 接近 100% 且队列深度持续上升）
- GPU queue depth 持续上升且 drop 增长

*验收*：压测制造 CPU 拥塞，系统进入 Degraded。

**[E1-4.2] MUST**：`Degraded` 超限动作按优先级执行：
1. 降低观测（metrics/trace/log）频率或丢弃低优先级观测
2. 降低录制强度（降采/分片/暂挂后台搬移），确保不阻塞热路径
3. 触发数据面降频/跳帧（仍需保证 TrackData 状态连续可得）
4. 若触发可信性故障（如 `authoritative_time_state=INVALID` / `time_state_level=0`），按 SEV-1 进入 `Fault`

*验收*：Degraded 时各降载动作按顺序生效。

**显存/内存池超限**

**[E1-4.3] MUST**：任何资源池分配失败不得阻塞热路径等待；必须触发：
- 统计 `alloc_fail` 并上报
- 对应输入队列执行 drop 策略（例如 drop-oldest）
- 若导致状态连续可得不可满足，进入 `Degraded` 或 `Fault`

*验收*：分配失败不阻塞热路径，触发 drop 并上报。

**IO 超限**

**[E1-4.4] MUST**：录制链路不得反压热路径；当 HDD 写入瓶颈出现（≈400MB/s），必须：
- 先保 NVMe 缓冲
- 超出缓冲上限时丢弃录制数据（可记录缺口），不得影响 TrackData 输出

*验收*：录制 IO 饱和时，热路径不被阻塞。

### 5.2 E2：录制吞吐/容量模型

#### 5.2.1 定量输入（冻结）

#### 5.2.2 整形时延预算

**[E2-6.1] MUST**：前端整形为显式开关 `front_end_shaping_enabled`，必须在 run manifest 中记录。
*验收*：manifest 含 front_end_shaping_enabled 字段。

**[E2-6.2] MUST**：当 `front_end_shaping_enabled=true`，后端必须将整形引入的排队上界纳入 E2E 预算：
- `front_end_shaping_delay_max_s` = **0.75s**（冻结）
- `front_end_shaping_buffer_bytes_min` ≥ **75MB/阵面**（冻结）

*验收*：系统能报告上述参数；可用 data_ts/ingest_ts 计算实际分布。

**[E2-6.3] MUST**：录制文件/索引必须同时保留 `data_ts` 与 `ingest_ts`，用于 E2E 分解。
*验收*：录制元数据含双时间戳。

#### 5.2.3 录制旁路隔离

**[E2-6.4] MUST**：Recorder 必须旁路 best-effort，独立队列/独立线程；任何拥塞不得反向阻塞 Receiver。
*验收*：Recorder 拥塞时 Receiver 吞吐不下降。

**[E2-6.5] MUST**：录制拥塞时采用 `drop-oldest` 或磁盘环形覆写；丢弃必须计数并可观测，但不视为故障。
*验收*：drop_count 可观测，系统状态不因丢弃变为 Fault。

**[E2-6.6] MUST**：丢弃/水位上报必须异步汇总降频，禁止 per-drop 同步上报。
*验收*：无 per-drop 同步上报代码路径。

#### 5.2.4 写盘落点与两级存储

**[E2-6.7] MUST**：长期留存落盘目录为 `/data/rec`（RAID10，36.1TB）。
*验收*：录制文件落在 /data/rec。

**[E2-6.8] SHOULD**：保留 NVMe 作为可选一级缓冲（吸收 RAID 抖动）：
- `/nvme/rec_staging`：短时 staging
- 后台搬移到 `/data/rec`

*验收*：配置 staging 时，两级存储正常工作。

**[E2-6.9] MUST**：Recorder 写盘线程不得阻塞热路径；RAID 抖动时优先丢弃录制数据而非反压主链路。
*验收*：RAID 抖动注入时热路径不阻塞。

#### 5.2.5 录制配额与水位线

**[E2-6.10] MUST**：对 `/data/rec` 启用固定配额 `Q_rec`（默认 **30TB**），禁止录制占满整个 /data。
*验收*：配额配置项存在且生效。

**[E2-6.11] MUST**：采用磁盘环形覆写 + 分段文件：
- `T_seg` 默认 **60s**（≈18GB/段，按 300MB/s）
- 按 `run_id` 建目录、按段序号递增命名

*验收*：录制目录结构符合规范。

**[E2-6.12] MUST**：定义两级水位线：

| 水位线 | 阈值 | 动作 |
|--------|------|------|
| `HWM_disk` | 85% of Q_rec | 进入清理预警区，启动异步清理 |
| `CWM_disk` | 95% of Q_rec | 强制覆写区，删除/覆写最旧段 |

*验收*：逼近 95% 配额时强制覆写生效，不突破 Q_rec。

**[E2-6.13] FORBIDDEN**：出现 ENOSPC 使系统级关键路径（日志/控制面/TrackData）受影响。
*验收*：CWM_disk 触发时系统其他模块不受影响。

#### 5.2.6 空间耗尽确定性动作

**[E2-6.14] MUST**：清理失败或配额异常时，Recorder 进入 " 强制覆写模式 " 并记录 `recorder_space_pressure=true`；持续异常可触发 `Degraded`（reason=REC_SPACE_PRESSURE），但不得导致 TrackData 中断。
*验收*：空间压力时进入 Degraded，TrackData 不中断。

**[E2-6.15] SHOULD**：对 " 录制缺口 " 生成结构化元信息（段缺失索引/时间范围）。
*验收*：缺口元信息可查。

#### 5.2.7 压缩策略

**[E2-6.16] FORBIDDEN（V1）**：Recorder 临界写入路径同步压缩。
*验收*：写入路径无同步压缩代码。

**[E2-6.17] SHOULD（预留）**：仅允许后台异步压缩 " 冷段 "，受 CPU 配额与 IO 预算约束。
*验收*：压缩仅发生在冷段且不影响热路径。

#### 5.2.8 容量模型汇总

| 指标 | 数值 |
|------|------|
| 后端录制输入（整形成立） | 100 MB/s/阵面，300 MB/s/三阵面 |

### 5.3 E3：E2E 时延预算与拆分模型

#### 5.3.1 总口径（冻结）

| 口径 | 公式 | 说明 |
|------|------|------|
| E2E latency | `L_e2e = output_ts - data_ts` | 覆盖前端整形/网络/后端处理/网关发送 |
| backend_latency | `L_backend = output_ts - ingest_ts` | 仅后端可控/可优化链路 |
| pre_ingest_latency | `L_pre = ingest_ts - data_ts` | 前端整形 + 链路贡献 |

#### 5.3.2 计入规则

**[E3-1] MUST**：E2E latency 口径固定为 `output_ts - data_ts`，不得使用模块内部自定义起点/终点替代。
*验收*：指标计算代码使用三时间戳。

**[E3-2] MUST**：backend_latency 口径固定为 `output_ts - ingest_ts`，用于后端性能目标、优化与验收归因。
*验收*：指标计算代码正确。

**[E3-3] MUST**：`ingest_ts` 必须由接收/重组线程在 " 形成完整处理单元 " 后打点。
*验收*：打点位置代码审查。

**[E3-4] MUST**：`output_ts` 必须由网关线程在 " 组包完成或发送时刻 " 打点。
*验收*：打点位置代码审查。

#### 5.3.3 一级分段预算

| 一级分段 | 计时边界 | 指标名 | P99 预算占比 |
|----------|----------|--------|-------------|
| Pre-ingest | `data_ts → ingest_ts` | `L_pre` | 不纳入后端 SLO |
| Ingress | `ingress_begin → ingest_ts` | `L_ingress` | 10% |
| Processing | `ingest_ts → publish_commit` | `L_processing` | 75% |
| Egress | `gateway_dequeue → output_ts` | `L_egress` | 15% |

**[E3-B1.1] MUST**：后端性能验收以 `L_backend` 为主；`L_pre` 仅用于解释 E2E，不得稀释后端责任。
*验收*：验收报告分别列出 L_backend 与 L_pre。

#### 5.3.4 二级分段预算

| 二级分段 | 指标名 | P99 预算占比（对 L_backend） |
|----------|--------|---------------------------|
| Receiver 拼包 | `L_rx_reassembly` | 10%（**注意**：V3.1 冻结 T_reasm=100ms，最坏情况下单帧重组可达 100ms；此预算仅覆盖正常情况下的拼包延迟。当触发 T_reasm 超时补零输出时，该帧延迟将显著超出此预算，但属于协议允许的降级行为，须通过 `reasm.incomplete_total` 指标可观测，不计入正常 P99 SLO） |
| Raw 队列等待 | `L_raw_qwait` | 10% |
| CPU 预处理 | `L_sp_cpu` | 5% |
| GPU 调度等待 | `L_gpu_qwait` | 20%（重点） |
| H2D 拷贝 | `L_h2d` | 10% |
| GPU kernel | `L_kernel` | 20% |
| D2H 拷贝 | `L_d2h` | 10% |
| Plots 队列等待 | `L_plots_qwait` | 5% |
| DataProc | `L_dp` | 10% |
| Gateway | `L_gateway` | 10–15% |

#### 5.3.5 必测点

**[E3-M0.1] MUST**：每帧/每处理单元必须具备唯一 `frame_seq`（单调递增）。
*验收*：frame_seq 单调性检查。

**[E3-M0.2] MUST**：热路径打点不得引入阻塞。
*验收*：打点代码无同步等待。

**[E3-M0.3] MUST**：所有打点采用同一时间基准（北斗/PTP 同源）。
*验收*：时间源配置检查。

**必测 Span 列表**（每个 span 产出 P50/P95/P99、最大值、计数）：

1. `L_pre = ingest_ts - data_ts`
2. `ingress_begin` → `ingest_ts`
3. `raw_enq/raw_deq`（队列等待）
4. `plots_enq/plots_deq`（队列等待）
5. `gpu_submit_req` → `h2d_begin` → `h2d_end` → `kern_begin` → `kern_end` → `d2h_begin` → `d2h_end`
6. `dp_begin` → `publish_commit`
7. `gateway_deq` → `output_ts`

#### 5.3.6 与 Degraded/Fault 的绑定

**关键阈值（冻结）**：定义与口径统一见 [分块设计文档/顶层设计_3.md](分块设计文档/顶层设计_3.md#ssot-trackdata-gap-thresholds)。

**[E3-5] MUST**：时延超限必须与 Degraded 判据同框（至少以处理延迟 P99 作为 SEV-2 输入项）。
*验收*：P99 超限时系统进入 Degraded。

**[E3-6] MUST**：当 `TrackData gap ≥ T1_trackdata_gap_ms(500ms)`，进入 Degraded，并在 TrackData 置位 `OUTPUT_THROTTLED + DEGRADED_MODE`。
*验收*：trackdata_gap_ms ≥ 500ms 时状态正确切换。

**[E3-7] MUST**：当 `TrackData gap ≥ T2_trackdata_gap_ms(2s)`，进入 Fault 并停止对外输出（不得用继续输出旧快照掩盖）。
*验收*：trackdata_gap_ms ≥ 2s 时进入 Fault 并停止输出。

#### 5.3.7 禁止项

**[E3-8] FORBIDDEN**：热路径（Receiver/SignalProc/DataProc）禁止 sleep/condvar/sem_wait 等被动等待。
*验收*：热路径代码审查无被动等待。

**[E3-9] FORBIDDEN**：任何以 " 保证录制完整性 " 为由阻塞或降低实时链路速率的行为。
*验收*：录制不影响实时链路。

| 枚举值 | V3.1 原始语义 | 后端处理 |
|--------|--------------|----------|
| `APPLIED` | 控制已按时生效 | 正常 ACK |
| `APPLIED_LATE` | 控制已生效但超窗口 | ACK + 告警 + 上报 `snapshot.mismatch_total` |
| `NOT_APPLIED` | 控制被拒绝（时统未锁定/参数越界等） | NACK + reason_code=DACS_NOT_APPLIED |
| `OVERRUN` | 新控制覆盖前一控制（前一控制未生效即被覆盖） | ACK + 告警 |
| `TIMEOUT` | 超 RTO_MS×MAX_RETRY 无响应 | TIMEOUT |
