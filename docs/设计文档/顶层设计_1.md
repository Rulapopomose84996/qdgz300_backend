## 第一部分：系统定位与业务需求

### 1. 系统的基本职责（Q1/A1）

#### 输出对象

系统按以下优先级输出信息：
1. **一级受众**：显控操作员（人类值守，通过显控终端）
2. **二级受众**：显控平台系统（上级系统自动联动）
3. **三级辅助**：日志与监控系统（事件记录与系统诊断）
4. **调试支持**：本地存档分析（工程验证阶段）

---

### 2. 业务必达内容定义（Q1/A2/D2）

<a id="ssot-trackdata-business"></a>

#### 核心交付物：TrackData（航迹）✅

**必须保证输出**的内容与质量标准：

| 维度 | 要求 | 备注 |
|---|---|---|
| **内容** | 每个稳定跟踪目标的 ID、时间戳、位置、速度、置信度 | 完整的目标状态 |
| **更新率** | ≥ 1 Hz（正常负载下） | 实时感知的保证 |
| **质量** | 可信、实时、可操作 | 不能是虚假/延迟的航迹 |
| **业务定义** | 输出状态持续可得（允许降低频率但不能中断） | 状态 > 每次更新 |

**补充裁决（P0-1，冻结）**：
- **MUST**：对外 TrackData 中的轨迹集合必须为**全局航迹集合**（跨三阵面统一口径）；全局航迹身份（track_id/track_uid）由 **M03（DataProc，全局 TrackManager 子职责）**唯一裁决。
- **FORBIDDEN**：任何“阵面独立航迹”（阵面内临时 Track/Local Track）**不得**直接写入对外 TrackData，不得绕过 M03 直接对外暴露。
- **MUST**：目标跨越阵面边界时，必须以“时间连续性与显示侧逻辑一致性”为优先级保持**轨迹身份连续**（详见第三篇 4.4 节跨阵面交接冻结规则）。

**业务失败场景**（仅有以下情况视为业务完全失败）：
- 系统无法输出任何可信 TrackData
- TrackData 严重延迟（超过可接受时间窗口）
- 航迹虚假、不连贯或不可操作（包括：跨阵面边界处出现无法解释的身份断裂/重复并存/频繁跳变）

#### 允许丢弃的中间产物：Plots（点迹）❌

| 特性 | 说明 |
|---|---|
| **属性** | 中间处理产物，非最终业务成果 |
| **可丢弃性** | 允许在资源紧张或故障时丢弃 |
| **限制** | 需保证已有 Plots 能持续生成有效 Track |
| **用途** | 调试、离线分析，不影响在线业务 |

### 3. 过载时的丢数据优先级（Q2/A2）

在系统过载时，按以下**从最可丢到最不可丢**的顺序决定数据处理策略：

```txt
优先级排序（可丢弃程度从高到低）
═════════════════════════════════

1️⃣  Raw IQ（原始采样数据）
   ✅ 可完全丢弃
   • 默认不持久化；仅调试模式下可选保存
   • 原因：收集后即失效，不参与在线业务

2️⃣  Logs / Telemetry（日志与遥测）
   ✅ 可降频、采样或丢弃
   • 不影响核心功能，仅用于监控与分析
   • 需保持最小覆盖率用于告警

3️⃣  Detection / Plots（点迹）
   ⚠️ 可部分丢弃
   • 允许丢低 SNR、低置信度点迹
   • 允许跳帧处理（减少计算量）
   • 限制：需保证 Track 有足够输入继续生成

4️⃣  TrackData（航迹）
   ❌ 不可丢弃（业务必达）
   • 允许降低更新率（但不能中断输出）
   • 允许合并/简化更新（但保持状态连续性）

5️⃣  Control Events（控制指令）
   ❌ 绝对不可丢
   • 通过独立高可靠通道传输
   • 优先级高于所有数据面流量
   • 采用"至少一次 + 幂等性"投递语义（等效实现恰好一次语义）
   • 与 DACS 的 UDP 控制面通过 CRC32C + ACK 重传闭环保障可靠性
```

### 4. 系统的失败定义（E1）

**系统 " 失效 " 的准确定义**（冻结）：

- **不失效条件**：只要系统仍能提供**可信的、部分空域的实时态势感知**，就不视为完全失效
- **定义分类**：此类故障定义为**可恢复故障（Recoverable Failure）**
- **含义**：单阵面或部分功能可能故障，但整体系统仍能输出有用信息

**最低可用标准**（Q5）：
- 当前阶段无硬性 SLA 量化，需边调试边积累
- 建议建立 " 最低 Track 输出率 " 与 " 最低覆盖空域 " 作为监测指标

## 第二部分：系统架构与分层模型

### 8. 总体分层（PART 3-A）

系统采用**两层平面隔离**的架构：

#### 8.1 架构概览

```txt
┌──────────────────────────────────────────────┐
│         显控终端 / HMI / 下级系统             │
└──────────────┬───────────┬─────────────┘
               TCP控制面     UDP数据面
               (可靠有序)   (TrackData快照)
                    ┌────▼────────▼───────┐
                    │   M04: 数据网关/发送    │ ◄─ 控制面（Node0）
                    │   TCP控制 + UDP推送     │
                    │   监控/运维/日志/配置    │
                    └────▲─────────┬──────────┘
      Command Queue      │         │ Status Events
       (MPSC, sync)      │         │ (Async)
                    ┌────┴─────────▼──────────┐
                    │  System Orchestrator    │
                    │   (状态机 FSM)          │ ◄─ 实时数据面（Node1）
                    └────┬──────────▲─────────┘
                    M01/M02/M03      │ Event Bus
                    数据处理核心    │ (Async)
                    (SPSC Queues)   │
                    └────────────────┘
                    • Receiver (三线程)
                    • SignalProc (GPU+CPU)
                    • DataProc (融合)
```

#### 8.2 处理平面分离（冻结）

<a id="ssot-plane-split"></a>

##### 实时数据平面（Data Plane）

- **所属模块**：M01（接收）、M02（信号处理）、M03（数据处理）
- **进程模型**：单进程多线程（一个高性能进程）
- **CPU 绑定**：**MUST** 静态绑定至 **NUMA Node 1 (CPU 16-31)**
- **内存分配**：**MUST** 仅使用 Node 1 本地内存池，严禁跨节点分配
- **实时保证**：禁止动态分配、禁止阻塞 IO、禁止同步日志

##### 非实时控制/管理平面（Control/Management Plane）

- **所属模块**：M04（数据网关）、监控、配置、远程管理、日志落盘
- **进程模型**：多进程/多容器（可以是独立进程或容器化）
- **CPU 绑定**：**MUST** 运行于 **NUMA Node 0 (CPU 0-15)**
- **禁止**：**FORBIDDEN** 抢占 Node 1 的核心资源
- **特性**：可以有 IO 阻塞、动态分配，相对非实时

#### 8.3 进程间通信方式

| 方向 | 通道 | 特性 |
|---|---|---|
| 数据面 → 控制面 | 事件总线（异步） | 无锁、降频、异步上报 |
| 控制面 → 数据面 | MPSC 队列（同步） | 有背压、保证投递、禁止持锁 |

### 9. 三阵面并行组织（Q3）

#### 9.1 并行模型

**模式**：流式并行（Streaming Parallelism）

**核心原则**：
- 逻辑流水线独立
- 物理资源共享调度

#### 9.2 各模块的并行方式

| 模块 | 组织方式 | 说明 |
|---|---|---|
| **M01 接收层** | 三线程分立 | 每个雷达阵面一条接收线程，独立入队 |
| **M02 信号处理** | 共享 GPU 调度 | 建立统一的 GPU 调度模块，内部使用 3 个独立 CUDA Streams |
| **M03 数据处理** | 逻辑分立后融合 | 先进行阵面内的点迹 - 航迹关联（分立执行），再进入全局融合 |

#### 9.3 融合策略

- **点迹融合**：不存在（三个阵面严格无重叠，120° × 3 无缝拼接）
- **航迹融合**：在 TrackData 层进行（时间连续性与显示侧逻辑一致性）
- **全局身份裁决（P0-1，冻结）**：
   - **MUST**：M03（DataProc）承担**全局航迹身份裁决者**职责（可称 Global TrackManager/Fusion），对外只输出其裁决后的全局 Track 集合。
   - **FORBIDDEN**：阵面内的临时航迹/局部关联结果不得直接对外输出；不得由 M01/M02 或阵面线程自行分配“对外 track_id”。
   - **MUST**：跨阵面边界交接（handover）必须通过 M03 的 COASTING/重现规则保持身份连续，不得在可关联窗口内新建对外 track_id（详见第三篇 4.4）。
- **融合器设计**：轮询三条独立 SPSC Track 队列，而非多生产者写同一队列

### 10. 时间管理（C1/C2）

#### 10.1 时间源层级（冻结）

```txt
权威来源：外部高精度时间服务器（PTP 北斗/GPS 授时设备）
    ▼
    ├─ 系统权威时间源（Source of Truth for Time）
    ├─ 故障转移：守时模式 + 算法对齐（待商榷）

当前原型阶段（临时）：NTP 自同步
    • 架构上"不承认"NTP 为有效时间源
    • 仅因尚无北斗模块，作为验证阶段的过渡方案
```

#### 10.2 时间质量等级与 DACS 时统映射（冻结，与 FPGA 协议 V3.1 对齐）

FPGA 协议 V3.1 定义了 GNSS/PTP/本地晶振分级时间质量，后端必须通过 DACS 心跳赅测获取时间质量状态并映射到显控 `clock_domain`：

| 时间质量等级 | 含义 | 定时生效策略 | 后端 FSM 影响 |
|---|---|---|---|
| **Q0 (Locked)** | GNSS/PTP 锁定，时间可信 | 允许秒边界定时生效（T_APPLY_WINDOW_MS=5ms） | Running 正常运行 |
| **Q1 (Holdover)** | 守时模式，精度逐渐降低 | 默认禁止定时生效，需降级或改用“立即生效”策略 | 应反映为 Degraded 或 Running-with-limitations |
| **Q2 (Freerun)** | 本地晶振自由运行，时间不可信 | 必须禁止定时生效 | 应反映为 Degraded，并同步到显控提示 |

**映射要求（MUST）**：
- 后端必须将 DACS 心跳中的 `TimeCoordSource` / `TimeQuality` 映射为显控协议 V1 的 `clock_domain` 字段
- `StateSnapshot` 中必须暴露“定时生效可用性”与“当前时间来源”
- 统一 Unix ms→ns 转换规则并记录分辨率（避免显控将 ms 级时间戳误判为 ns 级精度）

#### 10.3 双时间体系

系统内区分两类时间，各司其职：

| 时间类别 | 精度 | 用途 | 来源 |
|---|---|---|---|
| **近似时间 / 挂钟时间** | ms 级 | 日志、监控、一般事件 | 系统时钟/NTP |
| **高精度强一致时间** | 亚微秒级 | 雷达信号处理、数据融合 | 北斗授时 |

**时间戳规范（与 FPGA 协议 V3.1 对齐，冻结）**：
- DACS 协议时间戳统一为 Unix Epoch 毫秒（ms）
- 显控协议 V1 要求 TrackData 携带 `data_timestamp`（ns）作为新鲜度主判据
- 后端网关必须执行 ms→ns 统一转换，并在字段中明确实际分辨率边界

#### 10.4 全局排序与时间一致性（C2）

**问题**：虽然三个阵面覆盖空域严格无重叠，不涉及多源融合，但仍需全局时间一致性

**原因**：
- 目标跨越阵面边界时，相邻观测的时间戳必须连续
- 若各阵面本地时钟漂移，导致：
  - 显示轨迹速度失真
  - 方向跳变
  - 因果倒置
  - 操作员认知混乱

**解决**：
- 所有阵面时钟必须对齐到北斗（同步精度 ≤ 1 μs）
- 故障时进入守时模式，定期检查漂移

## 第三部分：控制流设计

### 11. FSM 与系统状态机（B1/Q4/Q5）

<a id="ssot-fsm-5state"></a>

#### 11.1 系统运行模式集合（B1）

系统明确定义以下 **5 个运行模式**（冻结）：

> **重要说明**：`Shutdown` 不属于 `system_state` 值域，而是 **lifecycle 事件**（进程生命周期控制命令）。
> 关闭命令触发优雅退出流程（资源清理、状态持久化），但在此过程中 `system_state` 仍为触发前的状态（如 Running/Standby），
> 直到进程退出。对外上报的 system_state 不包含 Shutdown。

```txt
┌─────────────────────────────────────────────────────────┐
│  系统运行模式状态机（5 态）                              │
└─────────────────────────────────────────────────────────┘

     ┌──────────┐
     │   Init   │  ← 系统启动/复位
     │ 初始化   │
     └─────┬────┘
           │ 初始化成功
           ▼
     ┌──────────┐
     │ Standby  │ ← 等待启动命令
     │ 待命     │
     └─────┬────┘
           │ 收到启动命令(Start Command)
           ▼
     ┌──────────┐
     │ Running  │ ← 正常业务态
     │ 运行     │ ──┐ 局部故障
     └────┬─────┘   │
           │        │
           │   ┌────▼────────┐
           │   │  Degraded   │ ← 降级但仍有TrackData输出
           │   │  降级模式   │
           │   └──────┬──────┘
           │          │ 故障恢复
           │          ▼
           ▼     ┌──────────┐
     ┌──────────┐│  Running │ ← 恢复正常
     │  Fault   │└──────────┘
     │  故障    │ ← 业务失败，无可信输出
     └────┬─────┘
          │ 手动重启/故障复位
          ▼
       [Init]
```

**Shutdown（关闭）作为 lifecycle 事件的生命周期**：
1. 任意状态下收到关闭命令 → 执行资源清理、状态持久化
2. 清理完成后进程退出
3. 不作为 system_state 上报，仅作为审计日志事件记录

#### 11.2 状态转移规则

<a id="ssot-fsm-transitions"></a>

| 当前状态 | 目标状态 | 触发条件 | 原子性 | 说明 |
|---|---|---|---|---|
| Init | Standby | 初始化成功 | 强原子性 | 所有核心模块初始化完成 |
| Standby | Running | 启动命令（Start Command） | 强原子性 | 采集、处理、网关全部就绪 |
| Running | Running | （自转移） | — | 正常工作 |
| Running | Degraded | 局部故障，仍有 TrackData | 非原子性 | 允许降级继续工作 |
| Degraded | Running | 故障恢复（连续帧间隔 < 500ms + 无告警 ≥ 5s） | 强原子性 | 需重新验证功能完整性 |
| Degraded | Fault | TrackData 中断 ≥ 2s | 强原子性 | 无可信输出 |
| Running | Fault | TrackData 中断 ≥ 2s | 强原子性 | 业务失败 |
| Fault | Init | 手动重启/故障复位 | 强原子性 | 必须经过完整 Init 复位 |

**禁止转移（FORBIDDEN）**：
- `Degraded → Standby`：禁止直接从降级恢复到待命，必须先恢复到 Running

**说明**：`system_state` 值域冻结为 5 态：Init/Standby/Running/Degraded/Fault。`Shutdown` 为 lifecycle 事件，不属于 system_state 值域。关闭命令触发优雅退出流程（资源清理、状态持久化），执行完成后进程退出。

#### 11.3 FSM 的 " 强一致性 " 定义（Q4）

**一致性范围**：**进程内强一致**（不跨进程分布式一致）

**含义**：
- FSM 的状态变更权力**必须（MUST）** 归属于运行在 **NUMA Node 0 的控制/管理平面主进程**中的**单一调度线程**
- 该线程是 **System Orchestrator 的状态管理核心**
- 所有状态变更必须**串行化**，无并发修改
- 其他模块通过**事件异步上报**，不直接修改状态

### 12. System Orchestrator 角色（Q5）

#### 12.1 Orchestrator 的职责

**System Orchestrator 为全局状态的唯一写入点（State-of-Truth）**

| 职责 | 详述 |
|---|---|
| **状态管理** | 维护 FSM 的唯一真实副本，串行化所有状态转移 |
| **事件处理** | 接收来自各模块的事件，决策是否触发状态转移 |
| **命令下发** | 向各模块下发执行命令，等待完成反馈 |
| **故障检测** | 监听来自 M01/M02/M03 的故障告警，触发降级或进入 Fault |
| **心跳维护** | 与显控保持双向心跳，断开时自动进入保护状态 |

#### 12.2 通信模式

```txt
数据面模块 → Orchestrator
  • 事件总线（异步）
  • 只读上报，无背压
  • 内容：故障告警、性能指标、状态反馈

Orchestrator → 数据面模块
  • MPSC 队列（同步阻塞）
  • 必须收到确认才能发出下一条
  • 内容：启动、停止、参数更新、复位命令

Orchestrator → 显控平台
  • TCP 控制面 + 应用层 ACK + 心跳（可靠有序）
  • 内容：状态同步（Hello/Welcome/StateSnapshot）、告警、遥测、ControlRequest/Response
  • 握手完成后必须发送 StateSnapshot；未同步前 HMI 禁止发送业务控制

M04 Gateway → 显控平台（UDP 数据面）
  • 单向推送 TrackData 快照
  • 允许丢包、乱序（UDP 特性），全量快照语义降低丢包飘移
  • MaxUdpPayloadBytes = 1200B（冻结），单包不分片，超限必须截断并标记
  • 携带 backend_instance_id / frame_seq / clock_domain / data_timestamp(ns)

Orchestrator ↔ DACS 射频前端（控制桥接）
  • UDP/8888 控制面：CRC32C + ACK 重传闭环（RTO=2500ms, MAX_RETRY=3）
  • 幂等键 (SourceID, ControlEpoch, SequenceNumber) ↔ HMI command_id 桥接映射
  • ControlEpoch 变化需触发 DACS 去重缓存切换
```

### 13. 状态转移的原子性语义（Q6）

#### 13.1 分情况讨论（冻结）

状态转移的原子性要求**不是一刀切**的，而是**分阶段的**：

| 转移阶段 | 原子性要求 | 说明 |
|---|---|---|
| **进入 Running** | **强原子性** | 任何核心模块失败 → 全系统禁止进入 Running |
| **Running 内故障** | **非原子性** | 允许局部失效 → 进入 Degraded；仍有 TrackData 时维持 Degraded |
| **Degraded → Fault** | **强原子性** | TrackData 中断达到阈值 → 立即转移到 Fault |
| **Fault → 任何状态** | **强原子性** | 必须经过完整 Init 复位，不允许跳跃 |
| **状态退出** | **强原子性** | 保证资源清理与状态重置完整 |

**关闭命令处理**：任意状态下收到关闭命令后，执行优雅清理资源、停止处理、持久化状态，作为 lifecycle 事件处理（非 system_state 转移）。

#### 13.2 设计原则

> **核心进出必须强一致，中间故障允许降级。**

这样既保证了系统的可靠性边界，又允许在 " 已进入 Running" 后的故障不会直接导致系统停止。

### 14. 控制事件的投递语义（Q8）

#### 14.1 分层投递机制

| 通道 | 投递语义 | 实现方式 |
|---|---|---|
| **进程内（M01/02/03 → Orchestrator）** | 至少一次 + 幂等性 | MPSC 无锁队列 + 同步阻塞 |
| **后端 → 显控（TCP 控制面）** | 至少一次 + 幂等性 | TCP + 应用层 ACK + 重试；显控幂等键为 `command_id`（TTL≥10min） |
| **后端 → DACS（UDP 控制面）** | 至少一次 + 幂等性 | UDP + CRC32C + ACK 重传闭环；DACS 幂等键为 `(SourceID, ControlEpoch, SequenceNumber)` |

#### 14.2 实现细节

**进程内队列（MUST）**：
- 禁止使用无界队列
- 必须使用**静态预分配的长队列**
- 队列满时：**同步阻塞发送线程**（直到空间释放）
- 目的：保证控制事件 " 绝对不丢 "

**网络传输（TCP）**：
- 编排器发出指令后，**必须收到显控的 ACK 确认**
- 若规定时间内未收到 ACK，触发**重试或报警**
- 显控与后端间维持**双向心跳**
- 心跳断开 → 进入**断链保护**（`conn_state=Down`，详见《第二篇》9.5），执行缓存策略与重连恢复流程
- 显控重连后必须先同步 `StateSnapshot`，未完成同步前 **禁止发送业务控制（FORBIDDEN）**
- `backend_instance_id` 变化时显控必须清除缓存，`manual_only` 禁止自动重放

**DACS 控制闭环（UDP，与 FPGA 协议 V3.1 对齐，冻结）**：
- 控制运行在 UDP 之上，必须由应用层显式提供可靠性
- RTO_MS=2500ms、MAX_RETRY=3、指数退避
- 重传包必须保持 SequenceNumber 与 Payload 完全一致（冻结）
- DACS 必须对重复控制做幂等去重（同键不重复执行，仅复用结果）
- Orchestrator 必须实现可审计的控制状态机：发送→等待 Ack→超时重传→失败归因

**幂等桥接层（MUST，冻结）**：
- 后端必须构建“桥接映射表”：`command_id` →（一个或多个）DACS Control 事务
- 每个 DACS 事务必须维护 Ack 结果聚合，否则会出现：显控已收到 ACK 但 FPGA 未按时生效，或 FPGA 已生效但显控误判失败
- `ControlEpoch` 变化时必须触发 DACS 去重缓存切换并拒绝旧纪元控制
- 控制结果必须以 `ControlResponse` 唯一判定，禁止 UI 侧“猜测成功”

### 15. 快环控制与慢环控制（补丁点 1）

在系统架构中，应明确区分两类控制机制，虽然都属于 " 控制 " 但本质不同：

#### 15.1 慢环控制（Slow-loop Control）

| 特征 | 内容 |
|---|---|
| **作用域** | 系统级状态迁移、全局参数配置、服务重启 |
| **频率** | 低频（Hz 量级） |
| **实时性要求** | 无严格要求 |
| **部署位置** | 控制平面（NUMA Node 0） |
| **例子** | 启动/停止命令、参数配置更新、告警响应 |

#### 15.2 快环控制（Fast-loop Control）

| 特征 | 内容 |
|---|---|
| **作用域** | 高频率、低延迟的精密跟踪任务 |
| **频率** | 高频（kHz 量级） |
| **实时性要求** | 严格确定性、微秒级响应 |
| **部署位置** | 数据平面（NUMA Node 1） |
| **例子** | 波束指向动态调整、实时增益更新、跟踪滤波闭环 |

#### 15.3 架构意义

显式划分这两类控制路径的目的：
- 厘清系统职责边界
- 为后续模块设计提供明确指导
- 避免慢环阻塞快环
- 为性能优化预留空间

### 16. 控制面的阻塞约束（Q9）

**控制面事件处理 MUST 满足**（冻结）：

| 约束 | 说明 |
|---|---|
| **非阻塞** | 禁止因 IO 或同步等待而阻塞 |
| **短处理** | 每个事件处理时间 < 10 ms（推荐） |
| **不持锁进入数据面** | 禁止拿着锁去调用数据面接口 |

**推荐方式**：
- 异步通知机制（事件驱动）
- 回调 + 队列
- 异步任务调度

### 18. 队列模型与背压策略（C2）

<a id="ssot-queue-drop"></a>

#### 18.1 队列的生产者/消费者关系（冻结）

| 连接 | 模型 | 说明 |
|---|---|---|
| Receiver → SignalProc | SPSC ringbuffer | 一条接收线程 → GPU 处理 |
| SignalProc → DataProc | SPSC ringbuffer | GPU 输出 → 航迹生成 |
| 三阵面汇聚 | 轮询多 SPSC | FusionProc 轮询三条独立队列 |
| 数据面 → 控制面 | 事件总线（异步） | 单向报告，无背压 |
| 控制面 → 数据面 | MPSC 队列 | 同步阻塞投递 |

**DACS 心跳/遥测优先队列（与 FPGA 协议 V3.1 对齐，冻结）**：
- V3.1 允许 Data(0x03) 与 Heartbeat(0x04) 复用 UDP/9999
- 接收端 **MUST** 按 `PacketType` 分类进入不同优先级队列，确保数据洪峰下心跳仍可处理
- 建议通过 DSCP 区分优先级；若 DSCP 不可靠，必须做软件侧分类保底
- 否则将导致：数据洪峰→心跳饿死→误判前端失联→错误降级/复位的级联故障

**设计理由**：
- SPSC 获得最低抖动与最高吞吐
- 多消费者场景使用 fan-out（每个消费者独立 SPSC）
- 多阵面融合采用 " 轮询 " 而非 " 多生产者写同一队列 "

#### 18.2 队列满时的处理策略（冻结）

系统采用**分数据级别的差异化策略**，不是 " 一刀切 "：

```txt
┌─────────────────────────────────────────────────────────────┐
│         队列满时的处理策略（按数据类型）                    │
└─────────────────────────────────────────────────────────────┘

1. Receiver → SignalProc（Raw IQ）
   • 数据属性：原始采样
   • 业务级别：可丢弃
   • 队列满策略：drop-oldest（覆盖最老）
   • 约束：禁止阻塞 Receiver
   • 监控：记录丢弃计数

2. SignalProc → DataProc（Plots 点迹）
   • 数据属性：中间产物
   • 业务级别：可部分丢弃
   • 队列满策略：drop-oldest 或按价值丢弃（低 SNR 优先）
   • 约束：禁止阻塞 SignalProc
   • 注意：需保证 Track 有足够输入

3. DataProc → Gateway（TrackData）★ 关键
   • 数据属性：核心输出
   • 业务级别：业务必达
    • 队列满策略：latest-state-wins（合并更新/仅保留最新快照）
       - **语义裁决（P0-2，冻结）**：该通道是“快照邮箱/最新态缓存”，Gateway 只需拿到**最新一帧全量快照**即可对外发送；允许跳过中间帧。
       - 对同一 TargetID，仅保留最新状态；对整帧快照，允许“新快照覆盖旧快照（未发送则丢弃旧快照）”。
       - 允许降低更新频率（但保持状态可读），并必须在 TrackData 质量字段中反映（第三篇 `OUTPUT_THROTTLED`）。
    • 是否阻塞：**禁止**（数据面热路径不得因发送侧变慢而阻塞）
    • Degraded 触发（与第二篇对齐）：队列持续满/覆盖达到阈值时必须进入 Degraded，并启动降频动作集合。
    • 原因：业务必达 ≠ 每次更新必达；承诺“状态连续可读”，不承诺“每次更新必达”。
    • **显控输出约束（与显控协议 V1 对齐，冻结）**：
       - Gateway 对外输出的每个 UDP TrackData 单包 **MUST ≤ 1200B**（MaxUdpPayloadBytes=1200B，冻结）
       - 应用层不分片（单 datagram），禁止 IP 分片（否则引入额外丢包与乱序放大）
       - 超限时必须按威胁/置信度/距离选择性输出并截断，显式标记 `is_truncated` / `drop_reason` / `dropped_count`
       - 每帧语义为全量快照（丢包不漂移），携带 `frame_seq`（单调递增）与 `backend_instance_id`

4. Control Events（控制指令）★ 绝对可靠
   • 数据属性：控制命令
   • 业务级别：绝对必达
   • 队列满策略：禁止丢弃，同步阻塞
   • 约束：必须与数据面队列隔离
   • 优先级：高于所有数据流量

5. Receiver → Recorder（Raw IQ 录制旁路）
   • 数据属性：debug 数据
   • 业务级别：必须有但旁路
   • 队列满策略：drop-oldest 或 drop-newest
   • 约束：不得反向阻塞主链路
```

#### 18.3 队列满策略汇总表

| 通道 | 数据类型 | 业务级别 | 队列满策略 | 是否阻塞 |
|---|---|---|---|---|
| Receiver → SignalProc | Raw IQ | 可丢 | drop-oldest | 禁止 |
| SignalProc → DataProc | Plots | 可部分丢 | drop-oldest | 禁止 |
| DataProc → Gateway | **TrackData** | **必达** | **latest-state-wins** | **禁止** |
| Control Channel | 控制事件 | **绝对必达** | **阻塞/重试** | **允许** |
| Receiver → Recorder | Raw 录制 | 必须有 | drop-oldest | 禁止 |

### 19. 丢弃水位与控制面通知机制（C2 补充）

<a id="ssot-watermarks"></a>

#### 19.1 丢弃触发水位策略（冻结）

**总体结论**：采用**固定水位阈值**，不采用动态自适应阈值

**原因**：
- v1.0 是原型验证阶段，需要可预测、可复现的行为
- 动态水位会引入隐含反馈回路，增加调试复杂度与失稳风险
- 当前无稳定 SLA 与负载模型，不具备动态调参的基础
- 动态水位作为 v2+ 优化方向，不进入 v1.0 架构约束

#### 19.2 固定双水位模型

所有有界队列统一采用**双水位模型**：

| 水位 | 定义 | 推荐值 | 行为 |
|---|---|---|---|
| **HWM（High Watermark）** | 进入 " 拥塞预警区 " | 70-80% | 启动拥塞处理 |
| **FWM（Full Watermark）** | 进入 " 强制丢弃区 " | 100% | 触发丢弃策略 |

#### 19.3 各数据类型在不同水位的行为

| 数据级别 | HWM 行为 | FWM 行为 |
|---|---|---|
| Raw IQ | 记录预警计数 | drop-oldest 执行 |
| Plots | 可启用价值丢弃 | drop-oldest 执行 |
| TrackData | 允许合并更新 | latest-state-wins 执行 |
| Control Events | 不适用 | 禁止丢弃 |
| Raw Recording | 记录预警 | drop-oldest 执行 |

#### 19.4 丢弃事件的控制面上报（冻结）

**总体原则**：
- 丢弃行为**必须被观测**，但**不得同步上报**
- **禁止**在数据面丢弃路径中直接触发控制面逻辑
- 所有上报采用**异步、汇总、降频**方式

**上报粒度与周期**：

| 指标 | 上报方式 | 周期 |
|---|---|---|
| `drop_count_total` | 累计值 | 100 ms - 1 s |
| `drop_rate (drops/sec)` | 速率 | 100 ms - 1 s |
| `queue_occupancy_percent` | 百分比 | 100 ms - 1 s |

**禁止**：per-drop event（单次丢弃事件报告）

**控制面响应级别**：

| 触发条件 | 响应 |
|---|---|
| HWM 触发 | 仅记录指标与告警提示，不改变系统运行模式 |
| FWM 持续触发（超过阈值时长） | 上报控制面，允许控制面决策进入 `Degraded` |

**架构不变量**：
- 数据面丢弃是 " 行为 "，状态切换是 " 控制面裁决 "
- 数据面不得因 " 上报 " 增加阻塞或抖动
- 所有水位与策略在 v1.0 中为**静态配置**，运行期不可自修改

---

### 20. 内存管理策略

#### 20.1 预分配内存池（冻结）

热路径内**禁止**动态分配，所有对象必须来自**预分配内存池**。

**内存池组织**：
- 按 NUMA 节点分池
- Node 1 内存池：用于数据处理
- Node 0 内存池：用于控制管理
- 池内部分对象类别（Raw Buffer、Plot、Track、Event 等）

**关键规则**：
- MUST 静态预分配（应用启动时）
- MUST 显式回收（引用计数或对象池管理器）
- MUST 监控内存使用率（告警高水位）
- SHOULD 支持热重载池参数（仅限 v1.1+）

#### 20.2 数据结构设计建议

| 数据类型 | 推荐容器 | 说明 |
|---|---|---|
| 环形缓冲区 | ringbuffer | 固定大小，零拷贝 |
| 对象池 | object pool | 预分配对象集合，轮转复用 |
| 无锁队列 | lock-free queue | 支持 SPSC/MPMC，自旋 CAS |
| 内存池 | memory pool | 预分配内存块，碎片管理 |

## 第五部分：录制与回放（Q6）

### 21. Raw Payload 录制

#### 21.1 录制的必要性（Q6）

**必须实现**（冻结）。原因：
- 算法问题诊断：无原始数据无法复现问题
- 工程验证：需要真实数据集用于算法优化
- 后续测试集：为 v1.1+ 构建标准测试数据

#### 21.2 录制策略

| 维度 | 规格 |
|---|---|
| **录制内容** | 原始 IQ 数据（UDP Payload） |
| **触发条件** | 系统启动时自动启用（可配置） |
| **存储介质** | SSD（读写速率足够） |
| **缓冲策略** | 内存缓冲 + 异步后写 |
| **队列** | 独立旁路 SPSC 队列（不影响热路径） |
| **丢弃策略** | drop-oldest（仅保留最新一段） |
| **容量管理** | 循环覆写（ring buffer on disk） |

#### 21.3 回放支持（v1.1+）

- 支持从文件加载 IQ 数据进行算法验证
- 支持时间同步重放（模拟实时处理）
- 与 Recorder 相对称的 Playback 模块

## 附录 A：架构约束清单（冻结条款总览）

### 冻结条款 Checklist

- [x] **平台隔离**：数据面 MUST 绑定 Node1，控制面 MUST 绑定 Node0
- [x] **进程模型**：数据面单进程，控制面多进程
- [x] **FSM 一致性**：进程内强一致（单一调度线程）
- [x] **热路径红线**：禁止动态分配、阻塞 IO、同步日志、跨 NUMA
- [x] **队列模型**：SPSC 为主，融合采用轮询，不用 MPSC
- [x] **队列满策略**：分数据级别差异化处理（不是一刀切）
- [x] **丢弃水位**：固定阈值（HWM/FWM），不动态自适应
- [x] **上报机制**：异步、汇总、降频，不同步反馈
- [x] **控制事件**：至少一次投递 + 幂等性，禁止跨 NUMA 取内存
- [x] **业务必达**：TrackData 状态持续可得（允许降频）
- [x] **时间源**：北斗权威，NTP 仅作临时验证
- [x] **录制必须**：Raw Payload 录制为基本要求
- [x] **FSM 5 态**：Init/Standby/Running/Degraded/Fault（Shutdown 为 lifecycle 事件，不属于 system_state）
- [x] **M01 重组对齐 V3.1**：T_reasm=100ms、缺片补零、IncompleteFrame 标记、尾包 Snapshot 提取
- [x] **显控通信双面**：TCP 控制面（可靠有序）+ UDP 数据面（TrackData 快照）
- [x] **TrackData 输出约束**：MaxUdpPayloadBytes=1200B、单包不分片、超限截断标记
- [x] **幂等桥接**：HMI command_id ↔ DACS (SourceID, ControlEpoch, SequenceNumber) 映射
- [x] **时间质量映射**：DACS Q0/Q1/Q2 → 显控 clock_domain；时间戳 ms→ns 转换规范
- [x] **心跳优先队列**：DACS 数据洪峰下心跳 MUST 优先处理，禁止饿死
